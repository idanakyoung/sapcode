# 1. GateWay EntitySet Create - CARRIERSETS

## 1.1 목적

OData에서 `POST` 요청이 들어왔을 때  
`CarrierSet` 엔터티 데이터를 DB 테이블 `ZTSCARR_G01`에 INSERT 한다.

즉,

```text
OData POST 요청
   ↓
Gateway CREATE_ENTITY 호출
   ↓
Request Body 읽기
   ↓
ABAP 구조로 변환
   ↓
DB INSERT
   ↓
성공 시 201 Created
```

---

## 1.2 실습 흐름

```text
CREATE 메서드 REDEFINE
   ↓
Signature 확인
   ↓
IO_DATA_PROVIDER->READ_ENTRY_DATA 사용
   ↓
DB 테이블 구조로 변환
   ↓
INSERT
   ↓
Gateway Client에서 POST 테스트
   ↓
201 Created 확인
```

---

## 1.3 코드

```abap
METHOD CARRIERSET_CREATE_ENTITY.
  DATA : LS_ZTSCARR TYPE ZTSCARR_G01.

  IO_DATA_PROVIDER->READ_ENTRY_DATA(
    IMPORTING
      ES_DATA = ER_ENTITY ).

  MOVE-CORRESPONDING ER_ENTITY TO LS_ZTSCARR.
  INSERT ZTSCARR_G01 FROM LS_ZTSCARR.

  IF SY-SUBRC <> 0.
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION.
  ENDIF.
ENDMETHOD.
```

---

## 1.4 코드 해석

### 핵심
- `READ_ENTRY_DATA`
  - HTTP Body(JSON/XML)를 읽어서 `ER_ENTITY`에 넣음
- `MOVE-CORRESPONDING`
  - OData 엔터티 구조 → DB 테이블 구조로 매핑
- `INSERT`
  - 실제 DB 저장
- `SY-SUBRC <> 0`
  - 실패 시 OData 비즈니스 예외 발생

---

## 1.5 요청 예시

### URI
```text
/sap/opu/odata/SAP/ZGW_NAVI_G01_SRV/CarrierSet
```

### HTTP Method
```text
POST
```

### Request Body

```json
{
  "Carrid" : "KO",
  "Carrname" : "Korean Airlines",
  "Currcode" : "KRW"
}
```

---

## 1.6 주의사항

### 1) 마지막 콤마 제거
잘못된 JSON 예:
```json
{
  "Carrid" : "KO",
  "Carrname" : "Korean Airlines",
  "Currcode" : "KRW",
}
```

→ 마지막 콤마 때문에 오류

### 2) URI는 엔터티셋까지만
- `CarrierSet`
- 개별 키 URL 아님

### 3) 성공 기준
- HTTP Status `201 Created`

---

## 1.7 결과 확인

```text
POST 성공
   ↓
HTTP 201
   ↓
Data Preview / GET 조회
   ↓
새 Carrier 데이터 확인
```

# 3. GateWay EntitySet Update - CARRIERSETS

## 3.1 목적

`PUT` 요청으로 기존 Carrier 데이터를 수정한다.

즉,

```text
URL의 Key 읽기
   ↓
Request Body 읽기
   ↓
DB 테이블 구조로 매핑
   ↓
UPDATE
```

---

## 3.2 코드

```abap
METHOD CARRIERSET_UPDATE_ENTITY.

  DATA : LS_ZTSCARR TYPE ZTSCARR_G01.

  IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
    IMPORTING
      ES_KEY_VALUES = ER_ENTITY ).
  LS_ZTSCARR-CARRID = ER_ENTITY-CARRID.

  IO_DATA_PROVIDER->READ_ENTRY_DATA(
    IMPORTING
      ES_DATA = ER_ENTITY ).

  MOVE-CORRESPONDING ER_ENTITY TO LS_ZTSCARR.
  UPDATE ZTSCARR_G01 FROM LS_ZTSCARR.

  IF SY-SUBRC <> 0.
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION.
  ENDIF.
ENDMETHOD.
```

---

## 3.3 코드 해석

### 핵심 단계

#### 1) URL Key 읽기
```abap
GET_CONVERTED_KEYS
```

- URL의 `Carrid` 값을 읽음

#### 2) Key 보존
```abap
LS_ZTSCARR-CARRID = ER_ENTITY-CARRID.
```

- Key 값을 DB 구조에 먼저 세팅

#### 3) 요청 Body 읽기
```abap
READ_ENTRY_DATA
```

- 수정할 값 읽기

#### 4) UPDATE 실행
```abap
UPDATE ZTSCARR_G01 FROM LS_ZTSCARR.
```

---

## 3.4 요청 예시

### URI
```text
/sap/opu/odata/SAP/ZGW_NAVI_G01_SRV/CarrierSet('OZ')
```

### HTTP Method
```text
PUT
```

### Body

```json
{
  "Carrid" : "OZ",
  "Carrname" : "Asiana",
  "Currcode" : "USD"
}
```

### 주의
- URI는 엔터티셋 + Key
- 마지막 콤마 제거

# 5. GateWay EntitySet Delete - CARRIERSETS

## 5.1 목적

Carrier 1건을 Key 기준으로 삭제한다.

---

## 5.2 코드

```abap
METHOD CARRIERSET_DELETE_ENTITY.
  DATA : LS_ENTITY TYPE ZCL_ZGW_NAVI_G01_MPC=>TS_CARRIER.

  IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
    IMPORTING
      ES_KEY_VALUES = LS_ENTITY ).

  DELETE FROM ZTSCARR_G01
    WHERE CARRID = LS_ENTITY-CARRID.

  IF SY-SUBRC <> 0.
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION.
  ENDIF.
ENDMETHOD.
```

---

## 5.3 처리 흐름

```text
DELETE 요청
   ↓
URL Key 읽기
   ↓
DB DELETE
   ↓
성공 시 삭제 완료
```

---

## 5.4 요청 방식

### URI
```text
/sap/opu/odata/SAP/ZGW_NAVI_G01_SRV/CarrierSet('OZ')
```

### HTTP Method
```text
DELETE
```

### 결과
- `OZ` 데이터 삭제됨

# 6. GateWay EntitySet Delete - CONNECTSETS (자습)

## 6.1 코드

```abap
METHOD CONNECTSET_DELETE_ENTITY.
  DATA : LS_ENTITY TYPE ZCL_ZGW_NAVI_G01_MPC=>TS_CARRIER.

  IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
    IMPORTING
      ES_KEY_VALUES = LS_ENTITY ).

  DELETE FROM ZTSCARR_G01
       WHERE CARRID = LS_ENTITY-CARRID.

  IF SY-SUBRC <> 0.
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION.
  ENDIF.
ENDMETHOD.
```

---

## 6.2 주의

이 코드는 실습 원문 그대로 적으면 **논리적으로 이상한 부분이 있다.**

### 이유
`CONNECTSET_DELETE_ENTITY`인데
- 타입이 `TS_CARRIER`
- 삭제 대상도 `ZTSCARR_G01`

즉, Carrier 삭제 코드처럼 보인다.

### ConnectSet 삭제라면 보통 이렇게 가야 자연스럽다
- 타입: `TS_CONNECT`
- 테이블: `ZTSPFLI_G01`
- Key: `CARRID + CONNID`

즉, 원문 코드는 실습 중 오타/복붙 실수 가능성이 높다.

---

## 6.3 테스트 URL

```text
/sap/opu/odata/SAP/ZGW_NAVI_G01_SRV/ConnectSet(Carrid='AA',Connid='0017')
```

# 7. GateWay EntitySet Create & Update & Delete - E14

## 7.1 개요

이번 파트는 직접 DB 테이블을 `INSERT/UPDATE/DELETE` 하는 대신  
**BAPI를 호출해서 BusinessPartner CRUD를 처리**한다.

즉,

```text
OData 요청
   ↓
Gateway CRUD 메서드
   ↓
BAPI 호출
   ↓
BAPI Return 확인
   ↓
오류면 OData 예외
   ↓
정상이면 응답 반환
```

---

## 7.2 Create - BUSINESSPARTNERS_CREATE_ENTITY

### 코드

```abap
METHOD BUSINESSPARTNERS_CREATE_ENTITY.
  DATA: LS_HEADERDATA TYPE BAPI_EPM_BP_HEADER,
        LS_BP_ID      TYPE BAPI_EPM_BP_ID,
        LT_RETURN     TYPE TABLE OF BAPIRET2.

* Get request data
  IO_DATA_PROVIDER->READ_ENTRY_DATA( IMPORTING ES_DATA = ER_ENTITY ).

* Map request fields to function module parameters
  LS_HEADERDATA = VALUE #( BP_ROLE       = ER_ENTITY-BUSINESSPARTNERROLE
                           EMAIL_ADDRESS = ER_ENTITY-EMAILADDRESS
                           COMPANY_NAME  = ER_ENTITY-COMPANYNAME
                           CURRENCY_CODE = ER_ENTITY-CURRENCYCODE
                           CITY          = ER_ENTITY-CITY
                           STREET        = ER_ENTITY-STREET
                           COUNTRY       = ER_ENTITY-COUNTRY
                           ADDRESS_TYPE  = ER_ENTITY-ADDRESSTYPE ).

* Create data
  CALL FUNCTION 'BAPI_EPM_BP_CREATE'
    EXPORTING
      HEADERDATA        = LS_HEADERDATA
*     PERSIST_TO_DB     = ABAP_TRUE
    IMPORTING
      BUSINESSPARTNERID = LS_BP_ID
    TABLES
      RETURN            = LT_RETURN.

  IF LT_RETURN IS NOT INITIAL.
    "Message Container
    MO_CONTEXT->GET_MESSAGE_CONTAINER( )->ADD_MESSAGES_FROM_BAPI( LT_RETURN ).
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION
      EXPORTING
        TEXTID            = /IWBEP/CX_MGW_BUSI_EXCEPTION=>BUSINESS_ERROR
        MESSAGE_CONTAINER = MO_CONTEXT->GET_MESSAGE_CONTAINER( ).
  ENDIF.

* Fill response data
  ER_ENTITY = VALUE #( BASE ER_ENTITY
                       BUSINESSPARTNERID = LS_BP_ID-BP_ID ).
ENDMETHOD.
```

---

## 7.3 Create 코드 해석

### 흐름
1. 요청 Body 읽기
2. BAPI 입력 구조(`LS_HEADERDATA`)로 매핑
3. `BAPI_EPM_BP_CREATE` 호출
4. 에러면 OData 예외
5. 성공하면 생성된 `BusinessPartnerID`를 응답에 채움

### 핵심 포인트
- `READ_ENTRY_DATA`
- `VALUE #( ... )` 로 명시적 필드 매핑
- 생성 후 ID 반환

### URI
```text
/sap/opu/odata/SAP/ZGW100_G01_STUDENT_SRV/BusinessPartnerSet
```

    "Message Container

## 7.5 Update 코드 해석

### 핵심 포인트

#### 1) Key 읽기
- URL에서 BusinessPartnerID 읽기

#### 2) 요청 Body 읽기
- 변경할 값 읽기

#### 3) `HEADERDATA`
- 실제 변경 데이터

#### 4) `HEADERDATAX`
- 어떤 필드를 변경할지 표시하는 플래그 구조
- 각 필드에 `'X'`를 넣어야 BAPI가 변경 대상으로 인식

### 아주 중요
BAPI Update는 **값만 넘긴다고 수정되는 게 아니라**,  
`X 구조`에 변경 대상 필드를 표시해야 반영된다.

### URI
```text
/sap/opu/odata/SAP/ZGW100_G01_STUDENT_SRV/BusinessPartnerSet('0100000000')
```

# 8. GateWay RFC - GENERATE : DATA MAPPING

## 8.1 개념

Gateway Service Builder에서 RFC/BAPI를 가져올 때  
데이터 매핑을 자동 생성하거나 수동 드래그 앤 드롭으로 연결할 수 있다.

즉,

```text
RFC/BAPI 구조
   ↓
Gateway Entity 구조
   ↓
매핑 정의
   ↓
Generate
   ↓
Runtime 코드 자동 생성
```

---

## 8.2 실습 흐름 요약

```text
RFC 모델 생성
   ↓
Import 대상 선택
   ↓
Key 값 설정
   ↓
자동 매핑 시도
   ↓
부족하면 수동 드래그 앤 드롭
   ↓
Append로 필드 추가
   ↓
Generate
   ↓
저장 / 활성화
```

---

## 8.3 핵심 메모

### 자동 매핑
- 버튼을 누르면 이름 기준으로 자동 매핑됨

### 오류 나는 경우
- 이름/타입이 완전히 안 맞으면 오류
- 이럴 때는 직접 드래그 앤 드롭으로 매핑

### Append 사용
- 필드가 부족하면 Append 아이콘으로 추가
- 필요한 셀을 더 만들어서 수동 매핑 가능

---

## 8.4 개별 조회 URL 예시

```text
(Airlineid='LH',Connectid='0400',Flightdate=datetime'2025-08-03T00%3A00%3A00')
```

### 의미
복합 키를 가진 Entity의 개별 조회 가능

---

## 8.5 추가 메모

- 클릭 한 번 더 해서 활성화 / 비활성화 선택 가능
- FlightDate 같은 필드는 새 셀 Append 후 직접 드래그 매핑 가능
- 결국 대부분은
  - 클릭
  - Append
  - Drag & Drop
  - Generate  
  흐름으로 진행됨

# 9. GateWay RFC - GENERATE : MAP CREATE OPERATION

## 9.1 개념

RFC/BAPI 기반 Create/Change 매핑은  
단순 조회 매핑보다 더 주의할 점이 있다.

특히 Update 류 BAPI는  
**변경할 필드에 대응하는 X 구조(Customer_Data_X 등)** 를 같이 매핑해야 한다.

---

## 9.2 실습 흐름

```text
새 RFC 모델 생성
   ↓
키 설정
   ↓
자동 매핑 시도
   ↓
오류 많으면 수동 드래그 앤 드롭
   ↓
Create/Change 관련 입력 구조 매핑
   ↓
X 구조용 필드 추가
   ↓
필요 필드 드래그 앤 드롭
   ↓
Generate
   ↓
저장 / 활성화
```

---

## 9.3 핵심 포인트

### 1) 자동 매핑만으로 끝나지 않는 경우 많음
- 이름이 안 맞거나
- BAPI 구조가 복잡하면
- 직접 매핑 필요

### 2) X 구조 필수
예:
- `CUSTOMER_DATA`
- `CUSTOMER_DATA_X`

BAPI Change 류에서는  
값 구조와 함께 **변경 표시 구조(X 구조)** 를 같이 넘겨야 실제 업데이트가 된다.

### 3) 실무 감각
- “여기까지 한다고 바피는 데이터가 업데이트 되는 것이 아니라서 추가 과정 필수”
- 즉, 매핑만 했다고 Update 동작이 완성되는 게 아니다
- 어떤 필드를 바꿀지 X 구조를 반드시 함께 세팅해야 한다

---

## 9.4 X 구조 매핑 요령

```text
변경할 필드
   ↓
같은 이름의 CUSTOMER_DATA_X 필드 찾기
   ↓
Append로 셀 추가
   ↓
드래그 앤 드롭 매핑
   ↓
Generate
```

### 핵심
- 이름이 안 와도 됨
- 중요한 건 “변경 표시” 구조가 연결되는 것

# 10. CRUD 전체 구조 한눈에 보기

## 10.1 DB 직접 처리형

```text
POST   → CREATE_ENTITY → INSERT
PUT    → UPDATE_ENTITY → UPDATE
DELETE → DELETE_ENTITY → DELETE
```

### 특징
- 간단함
- 직접 테이블 제어
- 실습용/커스텀 테이블에 적합

---

## 10.2 BAPI 처리형

```text
POST   → CREATE_ENTITY → BAPI_CREATE
PUT    → UPDATE_ENTITY → BAPI_CHANGE + X구조
DELETE → DELETE_ENTITY → BAPI_DELETE
```

### 특징
- SAP 표준 비즈니스 로직 활용
- 오류 처리 체계적
- 실무형

---

# 11. 자주 하는 실수 정리

## 11.1 Create
- JSON 마지막 콤마 남김
- URI를 EntitySet이 아니라 Key URL로 보냄
- POST인데 GET/PUT으로 보냄

## 11.2 Update
- Key URL 안 쓰고 필터 URL 사용
- `READ_ENTRY_DATA`만 하고 Key 처리 안 함
- BAPI Update에서 X 구조 누락

## 11.3 Delete
- Key 안 읽고 삭제 실행
- 잘못된 테이블/잘못된 타입 사용
- Connect 삭제인데 Carrier 타입/테이블 쓰는 실수

## 11.4 RFC Mapping
- 자동 매핑만 믿고 검증 안 함
- 키 설정 누락
- 날짜/복합키 필드 수동 추가 안 함
- X 구조 매핑 누락

---

# 12. 최종 핵심 요약

## 12.1 Create
- `READ_ENTRY_DATA`로 요청 데이터를 읽고
- DB INSERT 또는 BAPI CREATE 수행

## 12.2 Update
- URL Key + Request Body 둘 다 중요
- DB UPDATE 또는 BAPI CHANGE 수행
- BAPI는 `X 구조`가 핵심

## 12.3 Delete
- Key URL 기준으로 대상 1건을 삭제
- 직접 DELETE 또는 BAPI DELETE 사용

## 12.4 RFC Generate
- 조회용은 자동 매핑이 꽤 잘 되지만
- Create/Change는 수동 매핑과 X 구조 연결이 중요

## 12.5 가장 중요한 한 줄
> **Gateway CRUD의 핵심은 “요청 데이터와 키를 정확히 읽어 백엔드(DB/BAPI)에 올바르게 전달하는 것”이고, RFC/BAPI Update에서는 X 구조 매핑이 성패를 가른다.**
