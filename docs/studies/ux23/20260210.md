# 1. SAP Gateway Service 사원 테이블 예시 문제 (하드 코딩 X)

## 1.1 목적

기존에는 성별 코드값을 직접 `CASE` 문으로 하드코딩해서 텍스트로 바꿨다.

예:
- `1 -> 남성`
- `2 -> 여성`

하지만 이 방식은
- 코드 변경 시 재배포 필요
- 도메인 값 변경에 취약
- 재사용성이 낮음

그래서 이번 방식은  
**도메인 고정값을 읽어서 텍스트를 동적으로 변환**하는 방식이다.

---

## 1.2 핵심 아이디어

```text
도메인 값 테이블 로드
   ↓
내부 테이블 GT_GESCH 저장
   ↓
조회 결과의 성별 코드(GESCH)와 매칭
   ↓
설명 텍스트(DDTEXT) 채움
```

즉,
- 성별 텍스트를 하드코딩하지 않고
- Domain에 등록된 값을 런타임에 읽어서 사용한다.

---

## 1.3 클래스 속성 정의

### 해야 할 일
- 클래스에 속성 하나 정의
- 타입을 **도메인 값 테이블 타입**으로 설정

즉, `GET_DOMAIN_VALUES` 결과를 담을 내부 테이블을 클래스 전역 속성으로 선언해 둔다.

예상 개념:

```abap
CLASS-DATA GT_GESCH TYPE TABLE OF DD07V.
```

실제 타입명은 시스템/실습 환경에 따라 다를 수 있지만,  
핵심은 **도메인 값 테이블을 담을 속성**을 만들어 두는 것이다.

---

## 1.4 CLASS_CONSTRUCTOR 구현

### 코드

```abap
METHOD CLASS_CONSTRUCTOR.
  CALL FUNCTION 'GET_DOMAIN_VALUES'
    EXPORTING
      DOMNAME         = 'YDGESCH_01'
    TABLES
      VALUES_TAB      = GT_GESCH
    EXCEPTIONS
      NO_VALUES_FOUND = 1
      OTHERS          = 2.
  IF SY-SUBRC <> 0.
* Implement suitable error handling here
  ENDIF.
ENDMETHOD.
```

---

## 1.5 코드 해석

### 무엇을 하는가?
- 클래스가 처음 로드될 때 한 번 실행됨
- 도메인 `YDGESCH_01`의 고정값 목록을 읽어옴
- 결과를 `GT_GESCH`에 저장

### 왜 좋은가?
- 매 요청마다 도메인 값을 다시 안 읽어도 됨
- 하드코딩 제거
- 유지보수 쉬움

### 흐름

```text
클래스 최초 로드
   ↓
CLASS_CONSTRUCTOR 자동 실행
   ↓
GET_DOMAIN_VALUES 호출
   ↓
YDGESCH_01 도메인 값 읽기
   ↓
GT_GESCH에 저장
```

---

## 1.6 ESEMPSET_GET_ENTITYSET

### 코드

```abap
LOOP AT ET_ENTITYSET INTO LS_ENTITY.
  LS_ENTITY-GESTX = GT_GESCH[ DOMVALUE_L = LS_ENTITY-GESCH ]-DDTEXT.

  MODIFY ET_ENTITYSET FROM LS_ENTITY.
  CLEAR LS_ENTITY.
ENDLOOP.
```

### 의미
- `ET_ENTITYSET`은 여러 건 조회 결과
- 각 행마다 `GESCH` 코드값을 읽어서
- `GT_GESCH`에서 같은 도메인 값(`DOMVALUE_L`)을 찾아
- 설명 텍스트(`DDTEXT`)를 `GESTX`에 채운다

### 왜 LOOP가 필요한가?
여러 개의 데이터를 동시에 조회하므로  
각 행마다 변환해야 해서 LOOP가 필요하다.

### 흐름

```text
사원 목록 조회
   ↓
ET_ENTITYSET 채움
   ↓
LOOP
   ↓
GESCH 코드값으로 GT_GESCH 검색
   ↓
DDTEXT를 GESTX에 대입
```

---

## 1.7 ESEMPSET_GET_ENTITY

### 코드

```abap
ER_ENTITY-GESTX = GT_GESCH[ DOMVALUE_L = ER_ENTITY-GESCH ]-DDTEXT.
```

### 의미
- `ER_ENTITY`는 1건 조회 결과
- 한 건만 처리하면 되므로 LOOP 필요 없음
- 해당 코드에 맞는 도메인 텍스트를 바로 넣으면 됨

---

## 1.8 중간 Internal Table 사용 VS ET_ENTITYSET 직접 SELECT

### 1) 중간 Internal Table 사용

#### 구조
```text
DB → GT_DATA → (복잡 가공 / 재사용) → ET_ENTITYSET → OData 응답
```

#### 예시

```abap
DATA: gt_data TYPE TABLE OF yt1001_01.

SELECT *
  INTO TABLE gt_data
  FROM yt1001_01.

" 복잡한 로직 처리

MOVE-CORRESPONDING gt_data TO et_entityset.
```

#### 장점
- 가공 로직이 복잡할 때 유리
- 재사용 가능
- 중간 단계 디버깅 편함

#### 단점
- 메모리 한 번 더 사용
- 단순 조회에는 다소 번거로울 수 있음

---

### 2) ET_ENTITYSET 직접 SELECT

#### 구조
```text
DB → ET_ENTITYSET → (가공) → OData 응답
```

#### 예시

```abap
SELECT *
  INTO CORRESPONDING FIELDS OF TABLE et_entityset
  FROM yt1001_01.

LOOP AT et_entityset ASSIGNING <ls>.
  " 값 가공
ENDLOOP.
```

#### 장점
- 단순하고 빠름
- 코드가 짧음

#### 단점
- 복잡한 가공 로직이 많아지면 불편
- 중간 데이터 재사용이 어려움

# 2. SAP Gateway Service 데이터 연결 3 - E10

## 2.1 목적

새로운 Entity Type을 만들고,  
Business Partner 데이터를 BAPI로 조회해서 OData 서비스에 연결한다.

---

## 2.2 실습 흐름 요약

```text
새 Entity Type 생성
   ↓
구조체 작성
   ↓
Addressable 체크
   ↓
Generate Runtime
   ↓
GET_ENTITYSET 메서드 구현
   ↓
BAPI 호출
   ↓
오류 처리
   ↓
ET_ENTITYSET에 매핑
   ↓
Gateway Client에서 조회
```

---

## 2.3 BUSINESSPARTNERS_GET_ENTITYSET 코드

```abap
DATA: LT_HEADERDATA TYPE TABLE OF BAPI_EPM_BP_HEADER,
      LT_RETURN     TYPE TABLE OF BAPIRET2.

* Get data
CALL FUNCTION 'BAPI_EPM_BP_GET_LIST'
  TABLES
    BPHEADERDATA = LT_HEADERDATA
    RETURN       = LT_RETURN.

IF LT_RETURN IS NOT INITIAL.
  "Message Container
  MO_CONTEXT->GET_MESSAGE_CONTAINER( )->ADD_MESSAGES_FROM_BAPI( LT_RETURN ).
  RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION
    EXPORTING
      TEXTID            = /IWBEP/CX_MGW_BUSI_EXCEPTION=>BUSINESS_ERROR
      MESSAGE_CONTAINER = MO_CONTEXT->GET_MESSAGE_CONTAINER( ).
ENDIF.

* Map properties from the back-end to output response structure
ET_ENTITYSET = VALUE #( FOR HEADER IN LT_HEADERDATA
                        ( BUSINESSPARTNERID   = HEADER-BP_ID
                          BUSINESSPARTNERROLE = HEADER-BP_ROLE
                          EMAILADDRESS        = HEADER-EMAIL_ADDRESS
                          COMPANYNAME         = HEADER-COMPANY_NAME
                          CURRENCYCODE        = HEADER-CURRENCY_CODE
                          CITY                = HEADER-CITY
                          STREET              = HEADER-STREET
                          COUNTRY             = HEADER-COUNTRY
                          ADDRESSTYPE         = HEADER-ADDRESS_TYPE ) ).
```

---

## 2.4 코드 해석

### 한 줄 요약
> `BAPI_EPM_BP_GET_LIST`로 Business Partner 목록을 조회하고, 에러가 있으면 OData 에러로 던지고, 정상 데이터는 `ET_ENTITYSET` 구조에 맞춰 매핑해서 응답한다.

### 단계별 해석

#### 1) BAPI 호출
- `BAPI_EPM_BP_GET_LIST`
- Business Partner 목록 조회

#### 2) 오류 처리
- `LT_RETURN`이 비어있지 않으면
- Message Container에 BAPI 메시지 추가
- `/IWBEP/CX_MGW_BUSI_EXCEPTION` 발생

#### 3) 응답 매핑
- `VALUE #( FOR ... IN ... )`
- 내부 테이블 `LT_HEADERDATA`를 순회하면서
- OData 응답 구조 `ET_ENTITYSET`에 맞춰 필드를 매핑

### 장점
- `CORRESPONDING`보다 더 명시적
- 필드명이 다를 때도 깔끔하게 매핑 가능

# 3. SAP Gateway Navigation 생성 실습

## 3.1 목적

Carrier와 Connection 두 엔티티를 만들고  
각각 조회가 가능하도록 서비스 로직을 구현한다.

사용 테이블:
- Carrier → `ZTSCARR_G01`
- Connect → `ZTSPFLI_G01`

---

## 3.2 실습 흐름 요약

```text
Carrier / Connect 엔티티 생성
   ↓
각 엔티티의 키 설정
   ↓
Generate Runtime
   ↓
서비스 등록
   ↓
DPC_EXT에서 GET_ENTITY / GET_ENTITYSET 구현
   ↓
Gateway Client에서 테스트
```

---

## 3.3 CARRIERSET_GET_ENTITY

```abap
METHOD CARRIERSET_GET_ENTITY.
  IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
    IMPORTING
      ES_KEY_VALUES = ER_ENTITY ).

  SELECT SINGLE *
    INTO CORRESPONDING FIELDS OF ER_ENTITY
    FROM ZTSCARR_G01
    WHERE CARRID = ER_ENTITY-CARRID.

  IF SY-SUBRC <> 0.
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION.
  ENDIF.
ENDMETHOD.
```

### 의미
- URL에서 Carrier 키를 읽음
- `CARRID` 기준 1건 조회
- 없으면 OData 예외 발생

---

## 3.4 CARRIERSET_GET_ENTITYSET

```abap
SELECT *
  INTO CORRESPONDING FIELDS OF TABLE ET_ENTITYSET
  FROM ZTSCARR_G01.
```

### 의미
- Carrier 전체 목록 조회

---

## 3.5 CONNECTSET_GET_ENTITY

```abap
METHOD CONNECTSET_GET_ENTITY.
  IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
    IMPORTING
      ES_KEY_VALUES = ER_ENTITY ).

  SELECT SINGLE *
    INTO CORRESPONDING FIELDS OF ER_ENTITY
    FROM ZTSPFLI_G01
    WHERE CARRID = ER_ENTITY-CARRID
      AND CONNID = ER_ENTITY-CONNID.

  IF SY-SUBRC <> 0.
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION.
  ENDIF.
ENDMETHOD.
```

### 의미
- Connect는 Key가 2개
  - `CARRID`
  - `CONNID`
- 두 키를 함께 사용해 1건 조회

---

## 3.6 CONNECTSET_GET_ENTITYSET

```abap

  INTO CORRESPONDING FIELDS OF TABLE ET_ENTITYSET
```

# 4. SAP Gateway Navigation Association 실습 1

## 4.1 목적

Association과 Navigation Property를 생성해서  
**Carrier 1건에서 관련 Connect 목록으로 이동**할 수 있게 만든다.

즉:

```text
Carrier → to_Connect → Connect 목록
```

---

## 4.2 핵심 메모

- **Association** 생성
- Navigation Property 설정
- 데이터 모델이 변경되면 **Generate Runtime 다시 필수**

---

## 4.3 CONNECTSET_GET_ENTITYSET 코드 변경

```abap
METHOD CONNECTSET_GET_ENTITYSET.

  DATA : LT_CARRID TYPE /IWBEP/T_COD_SELECT_OPTIONS,
         LS_CARRID TYPE /IWBEP/S_COD_SELECT_OPTION.

  DATA : LS_CARRIER TYPE ZCL_ZGW_NAVI_G01_MPC=>TS_CARRIER.

  CASE IO_TECH_REQUEST_CONTEXT->GET_SOURCE_ENTITY_TYPE_NAME( ).
    WHEN ZCL_ZGW_NAVI_G01_MPC=>GC_CARRIER.
      IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_SOURCE_KEYS(
        IMPORTING
          ES_KEY_VALUES = LS_CARRIER ).

      LS_CARRID-SIGN = 'I'.
      LS_CARRID-OPTION = 'EQ'.
      LS_CARRID-LOW = LS_CARRIER-CARRID.
      APPEND LS_CARRID TO LT_CARRID.

  ENDCASE.

  SELECT *
    INTO CORRESPONDING FIELDS OF TABLE ET_ENTITYSET
    FROM ZTSPFLI_G01
    WHERE CARRID IN LT_CARRID.

ENDMETHOD.
```

---

## 4.4 코드 해석

### 핵심 아이디어
요청이 일반 `ConnectSet` 조회인지,  
아니면 `CarrierSet(...)/to_Connect` 같은 **네비게이션 조회인지**를 구분한다.

### 동작 순서

#### 1) Source Entity Type 확인
```abap
IO_TECH_REQUEST_CONTEXT->GET_SOURCE_ENTITY_TYPE_NAME( )
```

- 요청이 어디서 출발했는지 확인
- 여기서는 `Carrier`에서 온 경우 처리

#### 2) 상위 엔티티 키 읽기
```abap
IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_SOURCE_KEYS(
  IMPORTING
    ES_KEY_VALUES = LS_CARRIER ).
```

- 상위 Carrier의 키값(CARRID)을 읽음

#### 3) Range 테이블 생성
- `LT_CARRID`에 `EQ` 조건으로 상위 Carrier의 `CARRID` 넣음

#### 4) 하위 데이터 조회
```abap
SELECT ...
  FROM ZTSPFLI_G01
  WHERE CARRID IN LT_CARRID.
```

- 해당 Carrier에 연결된 Connect만 조회

---

## 4.5 테스트 URL

```text
/sap/opu/odata/SAP/ZGW_NAVI_G01_SRV/CarrierSet('AZ')/to_Connect?$format=json
```

### 의미
- Carrier `AZ` 1건을 기준으로
- 그 항공사와 연관된 `to_Connect` 데이터 조회

# 5. SAP Gateway Navigation Association 실습 2 - E11

## 5.1 목적

BusinessPartner와 Product 간 Association을 만들고  
상위 BusinessPartner 기준으로 관련 Product를 필터링해서 조회한다.

---

## 5.2 테스트 URL

```text
/sap/opu/odata/SAP/ZGW100_G01_STUDENT_SRV/BusinessPartnerSet('0100000000')/ProductSet?$format=json
```

### 의미
- BusinessPartner ID가 `0100000000`인 1건을 기준으로
- 관련 ProductSet 조회

---

## 5.3 PRODUCTSET_GET_ENTITYSET

```abap
METHOD PRODUCTSET_GET_ENTITYSET.
  DATA: LT_HEADERDATA TYPE TABLE OF BAPI_EPM_PRODUCT_HEADER,
        LT_RETURN     TYPE TABLE OF BAPIRET2.

  DATA: LS_BP_ID           TYPE BAPI_EPM_BP_ID,
        LS_BP_HEADERDATA   TYPE BAPI_EPM_BP_HEADER,
        LT_SO_SUPPLIER     TYPE TABLE OF BAPI_EPM_SUPPLIER_NAME_RANGE,
        LS_BUSINESSPARTNER TYPE ZCL_ZGW100_G01_STUDENT_MPC=>TS_BUSINESSPARTNER.

* Get key for navigation source
  CASE IO_TECH_REQUEST_CONTEXT->GET_SOURCE_ENTITY_TYPE_NAME( ).
    WHEN ZCL_ZGW100_G01_STUDENT_MPC=>GC_BUSINESSPARTNER.
      IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_SOURCE_KEYS(
        IMPORTING
          ES_KEY_VALUES = LS_BUSINESSPARTNER
      ).
      LS_BP_ID-BP_ID = LS_BUSINESSPARTNER-BUSINESSPARTNERID.

      CALL FUNCTION 'BAPI_EPM_BP_GET_DETAIL'
        EXPORTING
          BP_ID      = LS_BP_ID
        IMPORTING
          HEADERDATA = LS_BP_HEADERDATA
        TABLES
          RETURN     = LT_RETURN.

      IF LT_RETURN IS NOT INITIAL.
        " Message Container
        MO_CONTEXT->GET_MESSAGE_CONTAINER( )->ADD_MESSAGES_FROM_BAPI( LT_RETURN ).
        RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION
          EXPORTING
            TEXTID            = /IWBEP/CX_MGW_BUSI_EXCEPTION=>BUSINESS_ERROR
            MESSAGE_CONTAINER = MO_CONTEXT->GET_MESSAGE_CONTAINER( ).
      ENDIF.

      LT_SO_SUPPLIER = VALUE #( ( SIGN   = 'I'
                                  OPTION = 'EQ'
                                  LOW    = LS_BP_HEADERDATA-COMPANY_NAME ) ).
  ENDCASE.

* Get data
  CALL FUNCTION 'BAPI_EPM_PRODUCT_GET_LIST'
    TABLES
      HEADERDATA            = LT_HEADERDATA
      SELPARAMSUPPLIERNAMES = LT_SO_SUPPLIER
      RETURN                = LT_RETURN.

  IF LT_RETURN IS NOT INITIAL.
    " Message Container
    MO_CONTEXT->GET_MESSAGE_CONTAINER( )->ADD_MESSAGES_FROM_BAPI( LT_RETURN ).
    RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION
      EXPORTING
        TEXTID            = /IWBEP/CX_MGW_BUSI_EXCEPTION=>BUSINESS_ERROR
        MESSAGE_CONTAINER = MO_CONTEXT->GET_MESSAGE_CONTAINER( ).
  ENDIF.

* Fill response data
  ET_ENTITYSET = CORRESPONDING #( LT_HEADERDATA ).
ENDMETHOD.
```

---

## 5.4 코드 핵심 해석

### 한 줄 요약
> 상위 BusinessPartner 키를 읽고, 그 BP의 상세정보에서 회사명을 얻은 뒤, 그 회사명을 Supplier 필터로 써서 Product 목록을 조회한다.

---

## 5.5 달라진 점 정리

### 1) 추가된 DATA 선언
네비게이션 상위 엔티티 키를 읽고,  
필터를 만들기 위한 변수들이 추가되었다.

```abap
DATA: ls_bp_id           TYPE bapi_epm_bp_id,
      ls_bp_headerdata   TYPE bapi_epm_bp_header,
      lt_so_supplier     TYPE TABLE OF bapi_epm_supplier_name_range,
      ls_businesspartner TYPE ...
```

---

### 2) 요청이 BusinessPartner에서 온 네비게이션인지 확인

```abap
CASE io_tech_request_context->get_source_entity_type_name( ).
  WHEN ...=>gc_businesspartner.
```

- 일반 ProductSet 조회와
- `BusinessPartnerSet(...)/ProductSet` 조회를 구분

---

### 3) Source Key 읽기

```abap
io_tech_request_context->get_converted_source_keys(
  IMPORTING
    es_key_values = ls_businesspartner
).
```

- 상위 BusinessPartner의 키를 가져옴

---

### 4) 상위 BP 상세 조회

```abap
CALL FUNCTION 'BAPI_EPM_BP_GET_DETAIL'
```

- BusinessPartner 상세 조회
- 여기서 회사명(`COMPANY_NAME`)을 얻음

---

### 5) Supplier 필터 생성

```abap
lt_so_supplier = VALUE #(
  ( sign   = 'I'
    option = 'EQ'
    low    = ls_bp_headerdata-company_name )
).
```

- Product 조회용 선택조건 생성
- 회사명 = Supplier Name 으로 연결

---

### 6) Product 조회 시 필터 전달

```abap
CALL FUNCTION 'BAPI_EPM_PRODUCT_GET_LIST'
  TABLES
    headerdata            = lt_headerdata
    selparamsuppliernames = lt_so_supplier
```

- 원래는 전체 Product 목록 조회
- 이제는 Supplier 조건이 걸린 Product 목록 조회

---

## 5.6 BUSINESSPARTNERS_GET_ENTITY

```abap
DATA: ls_bp_id      TYPE bapi_epm_bp_id,
      ls_headerdata TYPE bapi_epm_bp_header,
      lt_return     TYPE TABLE OF bapiret2.

* Get key fields from request
io_tech_request_context->get_converted_keys(
```

# 7. CONNECTSET_GET_ENTITYSET - 다중 필터 확장

## 7.1 목적

기존에는 `CARRID`만 조건으로 조회했는데,  
이제는 여기에 더해

- 출발 도시(`CITYFROM`)
- 도착 도시(`CITYTO`)

까지 필터링할 수 있도록 확장한다.

---

## 7.2 코드

```abap
METHOD CONNECTSET_GET_ENTITYSET.

  DATA : LT_CARRID TYPE /IWBEP/T_COD_SELECT_OPTIONS,
         LS_CARRID TYPE /IWBEP/S_COD_SELECT_OPTION.

  DATA : LS_CARRIER TYPE ZCL_ZGW_NAVI_G01_MPC=>TS_CARRIER.

  DATA: LT_CITYFROM TYPE RANGE OF ZTSPFLI_G00-CITYFROM,
        LT_CITYTO   TYPE RANGE OF ZTSPFLI_G00-CITYTO.

  DATA: LS_FILTER LIKE LINE OF IT_FILTER_SELECT_OPTIONS.

  CASE IO_TECH_REQUEST_CONTEXT->GET_SOURCE_ENTITY_TYPE_NAME( ).
    WHEN ZCL_ZGW_NAVI_G01_MPC=>GC_CARRIER.
      IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_SOURCE_KEYS(
        IMPORTING
          ES_KEY_VALUES = LS_CARRIER ).

      LS_CARRID-SIGN = 'I'.
      LS_CARRID-OPTION = 'EQ'.
      LS_CARRID-LOW = LS_CARRIER-CARRID.
      APPEND LS_CARRID TO LT_CARRID.
  ENDCASE.

  IF IT_FILTER_SELECT_OPTIONS IS NOT INITIAL.
    READ TABLE IT_FILTER_SELECT_OPTIONS INTO LS_FILTER
      WITH TABLE KEY PROPERTY = 'Cityfrom'.
    IF SY-SUBRC = 0.
      LT_CITYFROM = CORRESPONDING #( LS_FILTER-SELECT_OPTIONS ).
    ENDIF.

    READ TABLE IT_FILTER_SELECT_OPTIONS INTO LS_FILTER
      WITH TABLE KEY PROPERTY = 'Cityto'.
    IF SY-SUBRC = 0.
      LT_CITYTO = CORRESPONDING #( LS_FILTER-SELECT_OPTIONS ).
    ENDIF.
  ENDIF.

  SELECT *
    INTO CORRESPONDING FIELDS OF TABLE ET_ENTITYSET
    FROM ZTSPFLI_G01
    WHERE CARRID IN LT_CARRID
      AND CITYFROM IN LT_CITYFROM
      AND CITYTO IN LT_CITYTO.

ENDMETHOD.
```

---

## 7.3 업그레이드된 점 정리

### 1) `LT_CITYFROM`, `LT_CITYTO` Range 변수가 추가됨

기존:
- 항공사 기준 조회만 가능

변경 후:
- 항공사 + 출발도시 + 도착도시 조건 가능

즉,

```text
기존: CARRID
변경: CARRID + CITYFROM + CITYTO
```

---

### 2) `IT_FILTER_SELECT_OPTIONS` 처리 로직 추가

기존:
- OData `$filter`를 전혀 반영하지 않음

변경 후:
- 요청에 필터가 들어오면
- 이를 읽어서
- ABAP Range로 변환
- DB WHERE 조건에 반영

---

### 3) `PROPERTY = 'Cityfrom'`, `PROPERTY = 'Cityto'` 필터 추출 기능 추가

즉, 아래 같은 요청이 가능해진다.

```text
$filter=Cityfrom eq 'SEOUL' and Cityto eq 'TOKYO'
```

이 조건이 그대로 ABAP 조회 조건에 반영된다.

---

### 4) `CORRESPONDING #( )`을 활용한 Range 자동 매핑

```abap
LT_CITYFROM = CORRESPONDING #( LS_FILTER-SELECT_OPTIONS ).
```

- OData 필터 구조 → ABAP Range 구조 자동 변환
- 직접 LOOP 돌며 만드는 것보다 훨씬 깔끔함

---

## 7.4 최종 조회 흐름

```text
네비게이션이면 상위 Carrier 키 읽기
   ↓
LT_CARRID 생성
   ↓
OData 필터에서 Cityfrom / Cityto 읽기
   ↓
LT_CITYFROM / LT_CITYTO 생성
   ↓
SELECT ...
  WHERE CARRID   IN LT_CARRID
    AND CITYFROM IN LT_CITYFROM
    AND CITYTO   IN LT_CITYTO
```

- 모델에서 해당 Property를 **Filterable**로 설정

### 주의
- 체크 안 했다고 해서 무조건 필터가 안 되는 건 아님
- 하지만 **메타데이터에 필터 가능 정보 저장**을 위해 설정하는 것이 좋다

# 8. 최종 핵심 요약

## 8.1 하드코딩 제거
- 도메인 고정값은 `GET_DOMAIN_VALUES`로 읽어와 사용
- `CLASS_CONSTRUCTOR`에 올려두면 재사용성과 성능이 좋음

## 8.2 Navigation / Association
- Association은 엔티티 간 관계 정의
- Navigation Property는 그 관계를 따라가는 URL 경로
- 상위 엔티티 키는 `GET_CONVERTED_SOURCE_KEYS`로 읽는다

## 8.3 GET_ENTITYSET 확장
- 일반 조회뿐 아니라
- 네비게이션 조회
- 필터 조회
- 복합 조건 조회
까지 처리 가능

## 8.4 Filter 처리
- 방법 1: `IO_TECH_REQUEST_CONTEXT` 직접 사용
- 방법 2: `IT_FILTER_SELECT_OPTIONS` 사용
- 일반적으로 `IT_FILTER_SELECT_OPTIONS`가 더 단순하고 실용적

## 8.5 실무 포인트
- 모델 변경 시 **Generate Runtime 다시 필수**
- BAPI 오류는 `Message Container + /IWBEP/CX_MGW_BUSI_EXCEPTION`
- OData 필터를 ABAP Range로 바꾸면 SQL WHERE와 자연스럽게 연결 가능

## 8.6 가장 중요한 한 줄
> **SAP Gateway Association/Navigation의 핵심은 “상위 엔티티의 키를 읽어서 하위 데이터를 필터링해 반환하는 것”이며, OData Filter는 이를 더 세밀하게 제어하는 장치다.**
