# 1. OData Overview 이론

## 1.1 SAP Gateway에서 왜 OData를 쓰는가?

SAP 시스템은 원래 **ABAP 내부에서만 데이터 사용**하도록 만들어졌지만,  
요즘에는 Fiori 앱, 외부 웹/모바일 앱 등 다른 시스템도 SAP 데이터에 접근해야 한다.

그래서 등장한 것이:

- **SAP Gateway**  
  → SAP 안에 있는 데이터를 밖에서 인터넷처럼 쓰게 해주는 문
- **OData**  
  → 그 문을 통해 데이터를 꺼내는 규칙

## 전체 구조

```text
[ UI (Fiori / Web / App) ]
            |
            |  HTTP 요청
            v
      [ SAP Gateway ]
            |
            |  OData 처리
            v
     [ SAP ABAP System ]
            |
            v
        [ DB Table ]
```

## 핵심 의미
- Gateway는 **외부 세계와 SAP 내부를 연결하는 출입문**
- SAP 내부 데이터를 웹처럼 접근 가능하게 해줌
- OData는 그 출입문을 통해 데이터를 주고받는 표준 규칙

---

## 1.2 REST란 무엇인가? (통신 방식의 기본)

### 개념
- URL + HTTP Method로 서버의 자원을 다루는 방식
- SAP Gateway도 이 REST 방식을 그대로 사용

### REST 요청 예시

```text
GET /customers/1001
```

→ 고객 번호 1001번 데이터 조회

### REST의 핵심 개념 4가지

| 개념 | 설명 |
|---|---|
| Resource | 서버가 제공하는 데이터 (예: 고객, 주문) |
| URL | Resource의 주소 |
| HTTP Method | 무엇을 할지 (조회, 생성, 수정, 삭제) |
| Stateless | 요청 하나하나가 독립적 |

### HTTP Method와 의미

| Method | 의미 | SAP에서 예 |
|---|---|---|
| GET | 조회 | 고객 목록 조회 |
| POST | 생성 | 신규 고객 생성 |
| PUT | 전체 수정 | 고객 정보 전체 수정 |
| PATCH | 부분 수정 | 고객 전화번호만 수정 |
| DELETE | 삭제 | 고객 삭제 |

## 1.3 OData란 무엇인가? (REST + 규칙)

### 개념
- REST를 SAP에서 쓰기 좋게 **표준화한 규칙**
- SAP Gateway는 사실상 **OData 전용 서버**라고 볼 수 있음

### 왜 필요한가?
REST는 자유도가 높아서 시스템마다 방식이 달라질 수 있다.  
이걸 **정해진 규칙으로 통일한 것**이 OData다.

### OData의 가장 중요한 특징

| 특징 | 설명 |
|---|---|
| Entity | 데이터 한 건 (Row) |
| Entity Set | 데이터 목록 (Table) |
| Metadata | 데이터 구조 설명서 |
| Query Option | 필터, 정렬, 페이징 |

### REST vs OData 비교

| 구분 | REST | OData |
|---|---|---|
| 규칙 | 느슨함 | 엄격함 |
| 표준 쿼리 | 없음 | 있음 |
| SAP 친화성 | 낮음 | 매우 높음 |
| 메타데이터 | 없음 | `$metadata` 제공 |

---

## 1.4 OData의 데이터 구조 (Entity 중심 사고)

### SAP 테이블과 OData의 매핑

| SAP 개념 | OData 개념 |
|---|---|
| Table | Entity Set |
| Row | Entity |
| Field | Property |
| Key Field | Key Property |

### 예시: SAP 테이블 `ZCUSTOMER`

| CUST_ID | NAME | CITY |
|---|---|---|
| 1001 | KIM | SEOUL |
| 1002 | LEE | BUSAN |

→ OData 구조

```text
EntitySet: Customers
 └─ Entity: Customer
     ├─ CUST_ID (Key)
     ├─ NAME
     └─ CITY
```

---

## 1.5 OData Operation (CRUD 개념)

### 개념
- OData는 데이터를 **Operation**으로 처리한다.
- CRUD와 OData Operation은 아래처럼 매핑된다.

| CRUD | 의미 | HTTP Method | SAP Gateway 메서드 |
|---|---|---|---|
| Create | 생성 | POST | CREATE_ENTITY |
| Read | 조회 | GET | GET_ENTITY / GET_ENTITYSET |
| Update | 수정 | PUT / PATCH | UPDATE_ENTITY |
| Delete | 삭제 | DELETE | DELETE_ENTITY |

### 조회 예시 (Read)

```text
GET /sap/opu/odata/sap/ZGW_CUSTOMER_SRV/Customers
```

---

## 1.6 Query Service (OData Query Option)

### 개념
Query Service는 **데이터를 조건에 맞게 조회하는 기능**이다.  
즉, SQL의 `WHERE / ORDER BY / LIMIT / OFFSET` 같은 역할을 한다.

### Query Option 정리

| 옵션 | 역할 | SQL 비유 |
|---|---|---|
| `$filter` | 조건 조회 | WHERE |
| `$select` | 필드 선택 | SELECT 컬럼 |
| `$orderby` | 정렬 | ORDER BY |
| `$top` | 개수 제한 | LIMIT |
| `$skip` | 건너뛰기 | OFFSET |
| `$expand` | 연관 데이터 | JOIN |

### Query Option 예시

```text
GET /Customers?$filter=City eq 'SEOUL'
```

→ 서울 고객만 조회

```text
GET /Customers?$select=CUST_ID,NAME
```

→ 고객 번호와 이름만 조회

```text
GET /Customers?$top=10&$skip=20
```

→ 21번째부터 10건 조회 (페이징)

# 2. SAP Gateway Service Implement 이론

## 2.1 Managing an SAP Gateway Service

### 개념
Gateway 서비스는 그냥 하나의 프로그램이 아니라  
**등록하고 활성화해야 쓸 수 있는 서비스**다.

즉,
- 서비스 생성
- 런타임 오브젝트 생성
- 서비스 등록
- 서비스 활성화  
이 과정을 거쳐야 실제 호출 가능하다.

## 전체 구조

```text
[ 화면 / 앱 / Fiori ]
        |
        |  GET / POST / PUT / DELETE
        |  + $filter, $top, $skip ...
        v
[ SAP Gateway Service ]
        |
        |  MPC  → 데이터 모양 정의
        |  DPC  → 데이터 처리 로직
        v
[ ABAP SELECT / UPDATE ]
        |
        v
[ DB Table ]
```

### 핵심 의미
- **MPC** = 데이터 모델 정의
- **DPC** = 실제 데이터 처리 로직

---

## 2.2 Defining a Data Model

### Service Builder에서 보는 구조 예시

```text
[ BusinessPartner ]
   |
   |-- n Contact
   |
   |-- n Product
   |
   |-- n SalesOrder
            |
            |-- n SalesOrderItem
```

### 개념 및 용어 정리

| 용어 | 진짜 의미 |
|---|---|
| Entity Type | 데이터 한 줄의 구조 |
| Entity Set | 데이터 여러 줄 |
| Property | 컬럼 |
| Key | PK |

---

## 2.3 Implementing Reading Operations

### 전체 흐름

```text
화면에서 /Customers 요청
        ↓
Gateway
        ↓
GET_ENTITYSET
        ↓
SELECT 문 실행
        ↓
응답 반환
```

### Gateway에서 쓰는 요청 종류 + ABAP 메서드

#### 1) 조회(Read) 관련 – GET 계열

| 요청 종류 | 의미 | 언제 쓰나 | 호출되는 ABAP 메서드 |
|---|---|---|---|
| GET_ENTITY | 한 건 조회 | 키값으로 1건만 필요할 때 | GET_ENTITY |
| GET_ENTITYSET | 여러 건 조회 | 목록, 리스트 화면 | GET_ENTITYSET |

#### 2) 변경(Change) 관련 – CUD 계열

| 동작 | 의미 | 화면 동작 예 | ABAP 메서드 |
|---|---|---|---|
| Create | 생성 | 신규 데이터 저장 | CREATE_ENTITY |
| Update | 수정 | 기존 데이터 변경 | UPDATE_ENTITY |
| Delete | 삭제 | 데이터 삭제 | DELETE_ENTITY |

#### 3) CRUD 정리

| CRUD | HTTP 요청 | 의미 | Gateway ABAP 메서드 |
|---|---|---|---|
| Create | POST | 새로 만든다 | CREATE_ENTITY |
| Read (1건) | GET | 하나 조회 | GET_ENTITY |
| Read (여러 건) | GET | 목록 조회 | GET_ENTITYSET |
| Update | PUT / PATCH | 수정 | UPDATE_ENTITY |
| Delete | DELETE | 삭제 | DELETE_ENTITY |

4. 전체 선택 후 Next

- 전체(최대) 자리 수
- 숫자 전체 자릿수 제한

예:
- 전체 자리 수: **7자리**
- 소수점 앞: 5자리
- 소수점 뒤: 2자리

# 6. SAP Gateway Service 데이터 연결 2 - E8

## 6.1 목적
직접 `SELECT`하는 대신 BAPI를 호출해서 Product 데이터를 OData 서비스로 제공한다.

---

## 6.2 구현 흐름

```text
GET_ENTITYSET 재정의
   ↓
BAPI_EPM_PRODUCT_GET_LIST 호출
   ↓
리턴 메시지 확인
   ↓
오류 시 OData 예외 발생
   ↓
정상 시 ET_ENTITYSET에 매핑
   ↓
Gateway 실행 후 URL 테스트
```

---

## 6.3 코드

```abap
DATA: LT_HEADERDATA TYPE TABLE OF BAPI_EPM_PRODUCT_HEADER,
      LT_RETURN     TYPE TABLE OF BAPIRET2.

* Get data
CALL FUNCTION 'BAPI_EPM_PRODUCT_GET_LIST'
  TABLES
    HEADERDATA = LT_HEADERDATA
    RETURN     = LT_RETURN.

IF LT_RETURN IS NOT INITIAL.
  " Message Container
  MO_CONTEXT->GET_MESSAGE_CONTAINER( )->ADD_MESSAGES_FROM_BAPI( LT_RETURN ).
  RAISE EXCEPTION TYPE /IWBEP/CX_MGW_BUSI_EXCEPTION
    EXPORTING
      TEXTID            = /IWBEP/CX_MGW_BUSI_EXCEPTION=>BUSINESS_ERROR
      MESSAGE_CONTAINER = MO_CONTEXT->GET_MESSAGE_CONTAINER( ).
ENDIF.

* Fill response data
ET_ENTITYSET = CORRESPONDING #( LT_HEADERDATA ).
```

---

## 6.4 코드 해석

### 1) BAPI 호출
- `BAPI_EPM_PRODUCT_GET_LIST`
- 제품 목록 조회

### 2) 오류 처리
- `LT_RETURN`에 메시지가 있으면
- Message Container에 담고
- `/IWBEP/CX_MGW_BUSI_EXCEPTION` 발생
- 즉, OData 오류 응답으로 전달

### 3) 응답 채우기
- `LT_HEADERDATA` → `ET_ENTITYSET`
- `CORRESPONDING #(...)` 로 구조 맞춰서 자동 매핑

# 7. SAP Gateway Service 데이터 연결 개별 표시 1

## 7.1 목적
EntitySet 전체 조회가 아니라  
**Key를 받아 특정 데이터 1건만 조회**하는 `GET_ENTITY` 구현

---

## 7.2 핵심 코드 1: URL에서 키 꺼내기

```abap
IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
 IMPORTING
   ES_KEY_VALUES = ER_ENTITY
).
```

### 의미
- URL로 넘어온 Key 값을 읽어서
- `ER_ENTITY`에 변환해서 넣는다.

예:
```text
/InfoSet('LH')
```

이면
- `ER_ENTITY-CARRID = 'LH'`

---

## 7.3 핵심 코드 2: 실제 1건 조회

```abap
SELECT SINGLE *
  INTO CORRESPONDING FIELDS OF ER_ENTITY
  FROM ZTGW100_G01
  WHERE CARRID = ER_ENTITY-CARRID.
```

### 의미
- URL에서 받은 `CARRID`로
- DB에서 1건 조회
- 결과를 `ER_ENTITY`에 채움

---

## 7.4 디버깅 포인트

- **External Breakpoint** 사용
- 이유: SAP GUI와 Gateway 요청 세션이 다를 수 있기 때문

# 8. SAP Gateway Service 데이터 연결 개별 표시 2 - E9

## 8.1 목적
ProductSet에서 특정 ProductID 1건을  
BAPI로 조회해서 반환하는 `GET_ENTITY` 구현

---

## 8.2 코드

```abap
DATA: ls_product_id TYPE bapi_epm_product_id,
      ls_headerdata TYPE bapi_epm_product_header,
      lt_return     TYPE TABLE OF bapiret2.

* Get key fields from request
io_tech_request_context->get_converted_keys(
  IMPORTING
    es_key_values = er_entity
).

* Map key fields to function module parameters
ls_product_id-product_id = er_entity-product_id.

* Get data
CALL FUNCTION 'BAPI_EPM_PRODUCT_GET_DETAIL'
  EXPORTING
    product_id = ls_product_id
  IMPORTING
    headerdata = ls_headerdata
  TABLES
    return     = lt_return.

IF lt_return IS NOT INITIAL.
  " Message Container
  mo_context->get_message_container( )->add_messages_from_bapi( lt_return ).
  RAISE EXCEPTION TYPE /iwbep/cx_mgw_busi_exception
    EXPORTING
      textid            = /iwbep/cx_mgw_busi_exception=>business_error
      message_container = mo_context->get_message_container( ).
ENDIF.

* Fill response data
er_entity = CORRESPONDING #( ls_headerdata ).
```

---

## 8.3 코드 해석

### 한 줄 요약
> URL로 넘어온 키(ProductID)를 꺼내서 BAPI로 상품 1건을 조회하고, 에러면 OData 에러로 던지고, 정상이면 응답으로 돌려준다.

### 단계별 해석

#### 1) URL Key 추출
- `get_converted_keys`
- ProductID를 `ER_ENTITY`에 가져옴

#### 2) BAPI 입력 파라미터 매핑
- `ls_product_id-product_id = er_entity-product_id`

#### 3) 상세 조회
- `BAPI_EPM_PRODUCT_GET_DETAIL` 호출

#### 4) 오류 처리
- `lt_return`에 오류가 있으면
- OData 비즈니스 예외 발생

#### 5) 응답 변환
- `ls_headerdata` → `er_entity`

# 9. SAP Gateway Service 사원 테이블 예시 문제

## 9.1 EntitySet 전체 조회 예시

```abap
DATA: LS_ENTITY LIKE LINE OF ET_ENTITYSET.

SELECT *
  INTO CORRESPONDING FIELDS OF TABLE ET_ENTITYSET
  FROM YT1001_01.

LOOP AT ET_ENTITYSET ASSIGNING FIELD-SYMBOL(<LS>).
  CASE <LS>-GESCH.
    WHEN '1'. <LS>-GESTX = '남성'.
    WHEN '2'. <LS>-GESTX = '여성'.
    WHEN OTHERS. <LS>-GESTX = ''.
  ENDCASE.
ENDLOOP.
```

### 해석
- `YT1001_01` 사원 테이블 전체 조회
- `GESCH` 값을 사람이 읽을 수 있는 텍스트 `GESTX`로 변환
  - `1` → 남성
  - `2` → 여성
  - 그 외 → 공백

### 처리 흐름

```text
사원 테이블 전체 조회
   ↓
ET_ENTITYSET 채움
   ↓
LOOP
   ↓
GESCH 코드값 해석
   ↓
GESTX 텍스트 생성
```

## 9.2 Entity 1건 조회 예시

```abap
IO_TECH_REQUEST_CONTEXT->GET_CONVERTED_KEYS(
 IMPORTING
   ES_KEY_VALUES = ER_ENTITY
).

SELECT SINGLE *
  INTO CORRESPONDING FIELDS OF ER_ENTITY
  FROM YT1001_01
  WHERE PERNR = ER_ENTITY-PERNR.

CASE ER_ENTITY-GESCH.
  WHEN '1'. ER_ENTITY-GESTX = '남성'.
  WHEN '2'. ER_ENTITY-GESTX = '여성'.
  WHEN OTHERS. CLEAR ER_ENTITY-GESTX.
ENDCASE.
```

### 해석
- URL에서 `PERNR` 키를 꺼냄
- 해당 사원 1건 조회
- 성별 코드 `GESCH`를 텍스트 `GESTX`로 변환

### 흐름

```text
URL에서 PERNR 추출
   ↓
SELECT SINGLE
   ↓
ER_ENTITY 채움
   ↓
GESCH 코드 해석
   ↓
GESTX 텍스트 설정
```

# 10. 전체 흐름 한눈에 보기

## 10.1 OData 요청 처리 전체 구조

```text
브라우저 / Fiori / 외부 앱
   ↓
HTTP 요청 (GET / POST / PUT / DELETE)
   ↓
SAP Gateway
   ├─ MPC : Entity / Property / Key 정의
   └─ DPC : GET_ENTITYSET / GET_ENTITY / CREATE / UPDATE / DELETE 구현
   ↓
ABAP 로직
   ├─ SELECT
   ├─ BAPI 호출
   └─ 데이터 가공
   ↓
DB / BAPI / 응답
```

---

## 10.2 GET_ENTITYSET vs GET_ENTITY

| 구분 | GET_ENTITYSET | GET_ENTITY |
|---|---|---|
| 목적 | 여러 건 조회 | 한 건 조회 |
| 화면 예 | 리스트 / 테이블 | 상세 페이지 |
| 입력 | 보통 없음 또는 Query Option | Key 필요 |
| 반환 | ET_ENTITYSET | ER_ENTITY |

---

## 10.3 SAP Gateway 서비스 생성 흐름 요약

```text
1. 구조체 / 테이블 준비
2. SEGW 프로젝트 생성
3. Entity / EntitySet 정의
4. Key 설정
5. Runtime Object 생성
6. 서비스 등록
7. DPC_EXT 메서드 재정의
8. GET_ENTITYSET / GET_ENTITY 구현
9. Gateway Client에서 테스트
```

# 11. 최종 핵심 요약

## 11.1 OData
- REST를 SAP에 맞게 표준화한 데이터 서비스 규칙
- Entity / EntitySet / Metadata / Query Option이 핵심

## 11.2 SAP Gateway
- SAP 내부 데이터와 외부 앱을 연결하는 출입문
- OData 서비스를 등록하고 활성화해서 사용

## 11.3 서비스 구현
- **MPC**: 데이터 구조 정의
- **DPC**: 실제 데이터 처리 로직 구현

## 11.4 조회 구현
- 여러 건 조회 → `GET_ENTITYSET`
- 한 건 조회 → `GET_ENTITY`
- DB 직접 조회 또는 BAPI 호출로 구현 가능

## 11.5 Query Option
- `$filter`, `$select`, `$top`, `$skip`, `$expand`
- SQL의 WHERE / SELECT / LIMIT / OFFSET / JOIN 역할과 유사

## 11.6 실무 감각 포인트
- 단순 테이블 조회는 `SELECT ... INTO CORRESPONDING FIELDS`
- SAP 표준 비즈니스 데이터는 BAPI를 통해 가져오는 경우 많음
- 오류는 Message Container + `/IWBEP/CX_MGW_BUSI_EXCEPTION`으로 OData 예외 처리

## 11.7 가장 중요한 한 줄
> **SAP Gateway는 SAP 데이터를 웹 서비스처럼 노출하는 문이고, OData는 그 문을 통해 데이터를 표준 방식으로 주고받는 규칙이다.**
