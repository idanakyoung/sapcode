# 1. GateWay RFC - MAP CREATE OPERATION - E15

## 1.1 개념

이 파트는 Gateway Service Builder에서  
**RFC / BAPI 기반 Create Operation을 매핑**하는 실습이다.

즉, 직접 `CREATE_ENTITY` ABAP 코드를 길게 쓰는 대신  
SEGW 안에서 RFC/BAPI 입력 구조와 OData Entity를 연결해서  
**Create 요청이 들어왔을 때 BAPI가 호출되도록 매핑**하는 흐름이다.

---

## 1.2 전체 흐름

```text
SEGW 프로젝트 열기
   ↓
RFC/BAPI 기반 Create Operation 추가
   ↓
RFC Import 구조 확인
   ↓
Entity / Property 생성 또는 연결
   ↓
키 설정
   ↓
Propose Mapping 또는 수동 매핑
   ↓
필요 시 새 열(Append) 추가
   ↓
입력 구조 / X 구조 매핑
   ↓
Generate
   ↓
서비스 활성화 후 테스트
```

---

## 1.3 #1 실습 핵심 요약

이미지 흐름을 기준으로 보면 첫 번째 실습은 대략 아래 과정이다.

### 실습 순서

```text
RFC/BAPI Create 관련 모델 선택
   ↓
입력 구조 확인
   ↓
Entity/Property 생성
   ↓
키값 설정
   ↓
필드 자동/수동 매핑
   ↓
창이 떠도 확인
   ↓
Generate
```

### 핵심 포인트
- Create Operation은 **입력 구조 매핑**이 가장 중요
- 단순 조회처럼 출력만 연결하는 게 아니라
- **요청 Body → RFC/BAPI 입력 파라미터** 로 연결되어야 함
- 자동 매핑이 안 되면 드래그 앤 드롭으로 직접 연결

---

## 1.4 #2 실습 핵심 요약

두 번째 실습은 `Propose Mapping`을 사용한 뒤  
부족한 필드를 새 열로 추가하는 흐름이다.

### 실습 순서

```text
Propose Mapping 실행
   ↓
자동 매핑 결과 확인
   ↓
누락 필드 발견
   ↓
새 열 추가
   ↓
수동 매핑 보완
   ↓
Generate
```

### 핵심 포인트
- `Propose Mapping`은 시작점일 뿐
- 누락되거나 잘못 연결된 필드는 사람이 다시 확인해야 함
- 필드명이 달라서 자동 매핑이 안 되는 경우가 많음
- 결국 **Generate 전에 매핑 검증**이 중요

---

## 1.5 Create Mapping에서 꼭 보는 것

### 1) 입력 구조
- Request Body가 들어갈 RFC/BAPI 입력 구조

### 2) Key
- 생성 후 응답에서 식별할 수 있는 키
- 또는 후속 Read/Navigation에 필요한 Key

### 3) 자동 매핑 여부
- 이름이 맞으면 자동 매핑 가능
- 안 맞으면 수동 연결 필요

### 4) Generate
- 데이터 모델 / 매핑 변경 후 반드시 Generate 필요

---

## 1.6 실무 감각 포인트

```text
조회 매핑
  → 출력 구조 매핑 중심

Create 매핑
  → 입력 구조 매핑 중심

Update 매핑
  → 입력 구조 + X 구조 매핑 중심
```

즉,
- Read는 응답이 중요하고
- Create는 입력이 중요하고
- Update는 **입력 + 변경표시(X)** 가 핵심이다.

# 2. GateWay - Search Help Generator - E16

## 2.1 개념

이 파트는 Gateway에서  
**Search Help / Value Help용 OData 구조를 생성**하는 흐름이다.

즉, Fiori/UI5에서 입력 도움말(Search Help)을 붙일 때  
백엔드에서 검색 가능한 엔터티셋과 연관 속성을 제공하도록 만드는 과정이다.

---

## 2.2 전체 흐름

```text
Search Help Generator 진입
   ↓
대상 테이블/구조/도메인 확인
   ↓
Value Help용 Entity 생성
   ↓
필드 선택
   ↓
Generate
   ↓
Gateway 서비스 테스트
   ↓
UI5/Fiori Value Help에서 사용
```

---

## 2.3 조회 예시 해석

### CountrySet에서 데이터를 조회하는데 앞에서 2건만 가져오기
이건 사실상 OData Paging 예시다.

예시 개념:

```text
/CountrySet?$top=2
```

의미:
- CountrySet 전체가 아니라
- 앞에서 2건만 응답

---

## 2.4 `$expand=CountryName`

```text
$expand=CountryName
```

### 의미
- 거래처 정보만 조회하는 것이 아니라
- 거래처와 연관된 국가 정보까지 같이 조회

즉,

```text
BusinessPartner
   ↓
연관 엔티티 CountryName
   ↓
한 번에 함께 응답
```

### 비유
- 기본 조회 = 메인 테이블만 조회
- `$expand` = 관련 테이블까지 같이 펼쳐서 조회

### SQL 비유
- JOIN 비슷한 개념

---

## 2.5 Search Help Generator 핵심 요약

- 검색 도움말용 OData 구조를 빨리 만들 수 있음
- 입력 필드의 Value Help와 연결하기 좋음
- `$top`, `$expand` 같은 OData 옵션과 조합 가능
- UI5/Fiori의 검색/선택 도움말과 자연스럽게 이어짐

# 3. ZDEMO_01 - 실습 : 기본적인 데이터 바인딩 만들기

## 3.1 목적

UI5 화면에서
- 조건 입력
- OData Read 호출
- 조회 결과를 JSONModel에 담고
- 화면 컨트롤에 바인딩하는

가장 기본적인 데이터 바인딩 예제다.

---

## 3.2 화면 구조 개요

화면은 크게 두 영역으로 구성된다.

```text
Page
 ├─ Condition Panel
 │   ├─ Airline Code 입력
 │   └─ Search 버튼
 │
 └─ Information Panel
     ├─ Carrid
     ├─ Begda
     ├─ Endda
     ├─ Price
     ├─ Waers
     ├─ Insert 버튼
     └─ Update 버튼
```

---

## 3.3 View 코드

```xml
<mvc:View controllerName="code.zdemo01.controller.Main"
    xmlns:mvc="sap.ui.core.mvc"
    xmlns="sap.m">s
    <Page id="page" title="{i18n>title}">
        <Panel id="PanelCond" headerText="Condition">
            <content>
                <Label id="LblCarrid" text="Airline Code"></Label>
                <Input id="InpCarrid" value=""></Input>
                <Button id="BtnSearch" text="Search" press="onSearch"></Button>
            </content>
        </Panel>

        <Panel id="PanelInfo" headerText="Information">
            <content>
                <Input id="InpInfoCarrid" value="{Info>Carrid}" maxLength="3" width="50px"></Input>
                 <!-- strictParsing: true 설정하지 않으면, 9999-12-31dms 10000.01.01로 표시 -->
                <!-- UTC : true = Local time zone 말고 GW(Gateway Server)에서 주는 날짜 그대로  --> 
                <DatePicker id="DPBegda" value="{path: 'Info>Begda',
                                         type: 'sap.ui.model.odata.type.Date',
                                         formatOptions: {pattern: 'yyyy.MM.dd', strictParsing: true, UTC:true}}"
                                         displayFormat = "yyyy.MM.dd"
                                         valueFormat = "yyyy.MM.dd"></DatePicker>

                <DatePicker id="DPEndda" value="{path: 'Info>Endda',
                                         type: 'sap.ui.model.odata.type.Date',
                                         formatOptions: {pattern: 'yyyy.MM.dd', strictParsing: true, UTC:true}}"
                                         displayFormat = "yyyy.MM.dd"
                                         valueFormat = "yyyy.MM.dd"></DatePicker>

                <Input id="InpInfoPrice"
                     value= "{parts:[{path: 'Info>Price'}, {path:'Info>Wears'}],
                     type: 'sap.ui.model.type.Currency',
                     formatOptions : {showMeasure: false}}"></Input>

                <Text id="TxtInfoWaers" text="{Info>Waers}"></Text>
                <Button id="BtnInsert" text="Insert" press="onInsert"> </Button>
                <Button id="BtnUpdate" text="Update" press="onUpdate"> </Button>
            </content>
        </Panel>
    </Page>
</mvc:View>
```

---

## 3.4 View 구조 해석

## 3.4.1 Condition Panel
- `InpCarrid`
  - 조회 조건 입력
- `BtnSearch`
  - `onSearch` 이벤트 호출

즉,

```text
사용자 입력
   ↓
Search 버튼 클릭
   ↓
Controller의 onSearch 실행
```

---

## 3.4.2 Information Panel

조회 결과를 표시하는 영역이다.

### 바인딩 대상
- `Carrid`
- `Begda`
- `Endda`
- `Price`
- `Waers`

여기서 모두 `Info`라는 이름의 모델에 바인딩되어 있다.

예:
```xml
value="{Info>Carrid}"
```

의미:
- 모델 이름: `Info`
- 경로: `/.../Carrid` 아래의 Carrid 속성

---

## 3.5 날짜 바인딩 해석

### DatePicker 예시

```xml
<DatePicker id="DPBegda" value="{path: 'Info>Begda',
                                 type: 'sap.ui.model.odata.type.Date',
                                 formatOptions: {pattern: 'yyyy.MM.dd', strictParsing: true, UTC:true}}"
                                 displayFormat = "yyyy.MM.dd"
                                 valueFormat = "yyyy.MM.dd"></DatePicker>
```

### 핵심 옵션

| 옵션 | 의미 |
|---|---|
| `type: sap.ui.model.odata.type.Date` | OData 날짜 타입으로 해석 |
| `pattern: 'yyyy.MM.dd'` | 표시 형식 |
| `strictParsing: true` | 잘못된 날짜 자동 보정 방지 |
| `UTC: true` | 로컬 타임존이 아니라 Gateway가 준 날짜 그대로 사용 |

### 주석 의미
- `strictParsing: true`가 없으면 날짜가 이상하게 보정될 수 있음
- `UTC: true`를 쓰면 Gateway 날짜를 로컬 타임존 변환 없이 그대로 보여줌

---

## 3.6 통화 바인딩 해석

### 코드

```xml
<Input id="InpInfoPrice"
     value= "{parts:[{path: 'Info>Price'}, {path:'Info>Wears'}],
     type: 'sap.ui.model.type.Currency',
     formatOptions : {showMeasure: false}}"></Input>
```

### 의미
- `Price`와 `Waers` 2개를 같이 사용해서 Currency 타입으로 표시
- `showMeasure: false`
  - 통화코드까지 같이 안 보이고 값만 표시

### 구조
```text
Price + Waers
   ↓
Currency 타입으로 포맷
   ↓
Input에 표시
```

## 3.7 Controller 코드

```javascript
sap.ui.define([
    "sap/ui/core/mvc/Controller"
], (Controller) => {
    "use strict";

    return Controller.extend("code.zdemo01.controller.Main", {
        onInit() {
        },
        onSearch() {
            let condCarrid = this.getView().byId("InpCarrid").getValue();
            let oModel = this.getView().getModel();
            let sPath = oModel.createKey("/InfoSet", {
                Carrid : condCarrid
            });

            // alert( sPath );

            let oView = this.getView();
            oModel.read( sPath,{
                success : function( oData, oResponse ) {
                    let jsonModel = new sap.ui.model.json.JSONModel({});
                    oView.setModel(jsonModel, "Info");
                    oView.getModel("Info").setProperty("/demo", oData);
                    console.log( oData );
                    oView.byId("PanelInfo").bindElement("Info>/demo");
                },
                error: function( oError ) {
                    let oMesg = JSON.parse( oError.responseText );
                    sap.m.MessageToast.show( oMesg.error.message.value);
                    let oPanelInfo = oView.byId("PanelInfo");
                    oPanelInfo.unbindElement();
                }
            } )
        }
    });
});
```

---

## 3.8 Controller 흐름 해석

## 3.8.1 onSearch 전체 흐름

```text

   ↓
OData 모델 가져오기
   ↓

   ↓
oModel.read 호출
   ├─ success → JSONModel 생성 / 데이터 바인딩
   └─ error   → 오류 메시지 표시 / 바인딩 해제
```

---

## 3.8.2 입력값 읽기

```javascript
let condCarrid = this.getView().byId("InpCarrid").getValue();
```

### 의미
- Airline Code 입력값 가져오기

## 3.8.3 OData Key URL 만들기

```javascript
let sPath = oModel.createKey("/InfoSet", {
    Carrid : condCarrid
});
```

### 의미
- `InfoSet` 엔터티셋에서
- Key가 `Carrid = 입력값` 인 개별 엔터티 경로 생성

예를 들면:

```text
/InfoSet(Carrid='AA')
```

같은 경로를 자동으로 만들어준다.

### 장점
- 문자열 직접 이어붙이기보다 안전함
- OData Key 포맷을 자동으로 맞춰줌

---

## 3.8.4 OData Read 호출

```javascript
oModel.read( sPath, { ... } )
```

### 의미
- 만들어진 Key 경로로 1건 조회

## 3.9 success 처리

```javascript
success : function( oData, oResponse ) {
    let jsonModel = new sap.ui.model.json.JSONModel({});
    oView.setModel(jsonModel, "Info");
    oView.getModel("Info").setProperty("/demo", oData);
    console.log( oData );
    oView.byId("PanelInfo").bindElement("Info>/demo");
},
```

### 단계별 해석

#### 1) JSONModel 생성
```javascript
let jsonModel = new sap.ui.model.json.JSONModel({});
```

- 조회 결과를 담을 클라이언트 모델 생성

#### 2) 뷰에 모델 설정
```javascript
oView.setModel(jsonModel, "Info");
```

- 모델 이름을 `"Info"`로 등록

#### 3) 조회 결과 저장
```javascript
oView.getModel("Info").setProperty("/demo", oData);
```

- `/demo` 경로 아래에 조회 결과 저장

#### 4) Panel 바인딩
```javascript
oView.byId("PanelInfo").bindElement("Info>/demo");
```

- `PanelInfo` 전체를 `Info>/demo`에 바인딩
- 내부 컨트롤들은 상대 경로로 값 사용 가능

즉,

```text
oData
   ↓
JSONModel /demo에 저장
   ↓
PanelInfo를 /demo에 bindElement
   ↓
하위 Input / Text / DatePicker 자동 표시
```

---

## 3.10 error 처리

```javascript
error: function( oError ) {
    let oMesg = JSON.parse( oError.responseText );
    sap.m.MessageToast.show( oMesg.error.message.value);
    let oPanelInfo = oView.byId("PanelInfo");
    oPanelInfo.unbindElement();
}
```

### 의미
- OData 조회 실패 시
- Gateway가 돌려준 에러 메시지를 파싱해서 Toast로 보여줌
- 기존 바인딩을 해제해서 잘못된 정보가 남아있지 않게 처리

### 흐름

```text
Read 실패
   ↓
responseText 파싱
   ↓
MessageToast 출력
   ↓
PanelInfo 바인딩 해제
```

## 3.11 바인딩 구조 한눈에 보기

```text
[Input: InpCarrid]
   ↓
onSearch()
   ↓
OData Model.read( /InfoSet(Key) )
   ↓
success(oData)
   ↓
JSONModel("Info") 생성
   ↓
/demo 에 데이터 저장
   ↓
PanelInfo bindElement("Info>/demo")
   ↓
하위 컨트롤 자동 표시
```

## 3.12 이 예제의 핵심 포인트

### 1) ODataModel 직접 바인딩이 아니라 JSONModel 중간 사용
즉,

```text
OData Read
   ↓
JSONModel 저장
   ↓
View 바인딩
```

이 구조다.

### 장점
- 화면에서 다루기 쉽다
- Insert/Update 전 가공하기 좋다
- 디버깅이 편하다

---

### 2) bindElement 사용
`PanelInfo` 전체를 한 번에 바인딩하므로  
하위 필드가 상대적으로 깔끔하게 연결된다.

---

### 3) 날짜 / 통화 타입 포맷팅
- Date는 OData Date Type
- Price는 Currency Type

즉, 단순 문자열 바인딩이 아니라  
**타입 기반 바인딩** 예제라는 점이 중요하다.

---

# 4. 최종 핵심 요약

## 4.1 RFC Create Mapping
- SEGW에서 RFC/BAPI 기반 Create Operation을 매핑할 수 있다.
- 자동 매핑 후에도 누락 필드는 반드시 확인해야 한다.
- Generate 전 최종 매핑 검증이 중요하다.

## 4.2 Search Help Generator
- Value Help / Search Help용 OData 구조를 빠르게 생성하는 기능
- `$top`, `$expand`와 함께 활용 가능
- UI5/Fiori의 도움말과 자연스럽게 연결된다.

## 4.3 UI5 기본 데이터 바인딩
- 입력값으로 Key를 만들고
- OData Read 수행 후
- JSONModel에 담아
- `bindElement`로 화면에 연결한다.

## 4.4 View 포인트
- DatePicker는 날짜 타입 포맷이 중요
- Currency 바인딩은 `parts + Currency type` 구조 이해 필요

## 4.5 Controller 포인트
- `createKey()`로 OData Key 경로 생성
- `read()` 성공 시 JSONModel에 담고
- `bindElement()`로 패널 전체 바인딩
- 오류 시 MessageToast + unbind 처리

## 4.6 가장 중요한 한 줄
> **UI5 기본 바인딩의 핵심은 “OData에서 읽은 데이터를 화면에 바로 꽂는 것”이 아니라, “필요한 형태의 모델(JSONModel)로 옮긴 뒤 View에 안정적으로 바인딩하는 것”이다.**
