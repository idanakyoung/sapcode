# cf. ABAP on HANA 등장 배경

## 1. 기존 방식 (Classical ABAP)

```text
[과거] DB → 모든 데이터 → ABAP → 계산

┌───────────────────────────┐
│     Application Layer     │
│          (ABAP)           │
│                           │
│  - LOOP                   │
│  - READ TABLE             │
│  - 계산 로직               │
│  - 집계 / 필터링           │
│                           │
└───────────▲───────────────┘
            │ 모든 데이터
            │ (대량 전송)
┌───────────┴───────────────┐
│      Database Layer       │
│           (DB)            │
│                           │
│  - 데이터 저장             │
│  - 단순 SELECT            │
│                           │
└───────────────────────────┘
```

## 2. 현재 방식 (ABAP on HANA / New Programming Approach)

```text
[현재] DB(계산 완료) → 결과만 → ABAP

┌───────────────────────────┐
│     Application Layer     │
│          (ABAP)           │
│                           │
│  - 결과 사용               │
│  - UI / 흐름 제어          │
│                           │
└───────────▲───────────────┘
            │ 결과만 전송
            │ (최소 데이터)
┌───────────┴───────────────┐
│      Database Layer       │
│          (HANA)           │
│                           │
│  - 계산 로직               │
│  - 필터링                  │
│  - 집계                    │
│  - JOIN                    │
│  - CDS / SQL / AMDP       │
│                           │
└───────────────────────────┘
```

## 3. 핵심 비교

### 기존 방식
- DB는 주로 저장과 단순 조회 역할
- 대량 데이터를 ABAP 서버로 가져온 뒤 계산
- LOOP, READ TABLE, 후처리 로직이 많음
- 데이터 전송량이 큼

### 현재 방식
- HANA가 계산, 집계, 필터링, JOIN 수행
- ABAP는 결과 소비와 UI/흐름 제어 중심
- 필요한 결과만 전송
- Code Push-down 철학 반영

# 0. Bottom-Up Approach 개요

## 정의

- **Database(HANA)에서 먼저 모델링**
- 이후 ABAP에서 이를 **소비(consume)** 하는 방식
- Code Push-down 전략의 실현 방법 중 하나

## 흐름

```text
HANA (View / Procedure 생성)
        ↓
ABAP에서 호출
```

## 핵심 의미
- 데이터 모델이나 프로시저를 먼저 DB에 만든다.
- 이후 ABAP 프로그램이 그 결과를 사용한다.
- 즉, 상위 애플리케이션보다 **DB 모델링이 먼저**라는 점에서 Bottom-Up이다.

| 재사용성 | 낮음 |

- Native SQL 필수

- ABAP Transport와 HANA Transport 이원화

## 사용 시점

- ABAP 7.4 SP02 이전
- SAP HANA가 Secondary DB인 경우
- 레거시 시스템

# 2. Bottom-Up Approach – 개선된 방식 (AS ABAP ≥ 7.4 SP02)

## Figure 33 기준: Proxy Objects 사용

## 핵심 개념

> HANA 객체를 ABAP Repository 안의 Proxy 객체로 감싸서 사용

## 전체 구조

```text
AS ABAP
 ├─ External View
 ├─ Database Procedure Proxy
 └─ SAP HANA Transport Container

        ↓
SAP HANA
 ├─ SAP HANA Views
 ├─ Stored Procedures
```

## 구조 시각화

```text
SAP HANA Object
   ↓
ABAP Proxy Object로 노출
   ↓
ABAP Repository 안에서 사용
   ↓
CTS와 연계된 Transport 관리
```

| ABAP 객체 | 역할 |
|---|---|

| Database Procedure Proxy | HANA Stored Procedure의 ABAP Proxy |
| SAP HANA Transport Container | ABAP–HANA Transport 동기화 |

## 장점

| 항목 | 개선 내용 |
|---|---|
| SQL 사용 | Native SQL 불필요 |
| DB 독립성 | 향상 |
| Transport | CTS로 통합 관리 |
| 유지보수 | 쉬움 |
| 재사용성 | 높음 |

# 3. Bottom-Up Approach 접근 방식 비교 요약

| 구분 | Direct Access | Proxy Objects |
|---|---|---|
| ABAP 버전 | < 7.4 SP02 | ≥ 7.4 SP02 |
| SQL 방식 | Native SQL | Open SQL |
| ABAP 객체 | 없음 | External View / Proxy |
| Transport | 분리 | 통합 |
| 권장 여부 | 비권장 | 권장 |

## 요약 해석

### Direct Access
- 빠르게 붙일 수는 있지만
- Native SQL 기반이라 DB 종속성이 매우 높음
- 유지보수와 이관이 어렵다

### Proxy Objects
- ABAP Repository 안에서 HANA 객체를 다룰 수 있음
- CTS와 통합되어 관리가 쉬움
- 실무 권장 방식

# 4. Transparent Optimizations (기본 최적화 계층)

원래 강의 그림은 이미지였지만, 개념을 텍스트로 정리하면 다음과 같다.

## 개념

Transparent Optimizations는  
개발자가 코드를 크게 수정하지 않아도  
DB Interface나 실행 구조 차원에서 어느 정도 최적화 효과를 얻는 계층이다.

## 시각화

```text
기존 ABAP 코드
   ↓
Open SQL 실행
   ↓
DB Interface / Runtime 차원의 최적화
   ↓
DB 처리 성능 개선
```

## 특징
- 개발 복잡도가 낮다
- 기존 코드를 크게 바꾸지 않는다
- 성능 향상 폭은 제한적이다
- 마이그레이션 초기 단계에서 유용하다

---

# 5. Code-to-Data 접근 방법 3가지 비교

## 시각화

```text
[낮은 복잡도 / 낮은 성능]
Transparent Optimizations
        ↓
Advanced SQL in ABAP
        ↓
SAP HANA Native Features
[높은 복잡도 / 높은 성능]
```

## 비교 표

| 구분 | Transparent Optimizations | Advanced SQL in ABAP | SAP HANA Native Features |
|---|---|---|---|
| 성능 향상 | 낮음 | 중간~높음 | 매우 높음 |
| 코드 복잡도 | 낮음 | 중간 | 높음 |
| DB 종속성 | 없음 | 낮음 | 높음 |
| 유지보수 | 매우 쉬움 | 쉬움 | 어려움 |
| 권장도 | 마이그레이션 초기 | 표준 권장 | 제한적 사용 |

## 해석

### 1) Transparent Optimizations
- 진입이 가장 쉽다
- 기존 시스템을 크게 건드리지 않아도 된다
- 성능 향상은 제한적

### 2) Advanced SQL in ABAP
- 실무에서 가장 균형이 좋다
- Open SQL 강화 문법을 활용해 DB 계산을 적극 활용
- 표준 권장 접근

### 3) SAP HANA Native Features
- 가장 강력한 성능 최적화 가능
- 대신 HANA 종속성이 높음
- 유지보수가 어렵고 제한적으로 사용

# 8. Open SQL, Native SQL, Database Interface, Open SQL의 강화 정리

## 전체 구조

```text
ABAP Program
   │
   ├─ Open SQL        (DB 독립) : ABAP 전용의 DB 독립적인 SQL 인터페이스
   └─ Native SQL      (DB 종속) : 특정 데이터베이스의 고유 SQL 문법을 직접 사용
        ↓
Database Interface (DBI) : ABAP과 실제 데이터베이스 사이의 중간 계층
        ↓
Database-Specific Library (DBSL)
        ↓
Database (HANA, Oracle, DB2 ...)
```

## 시각화

```text
ABAP Program
   ├─ Open SQL
   └─ Native SQL
         ↓
       DBI
         ↓
       DBSL
         ↓
      Database
```

## Open SQL vs Standard SQL vs Native SQL 핵심 비교 표

| 구분 | Open SQL | Standard SQL | Native SQL |
|---|---|---|---|
| 사용 위치 | ABAP | DB 일반 | ABAP |
| DB 독립성 | 높음 | 중간 | 없음 |
| SQL 범위 | 제한적 | 광범위 | DB 전체 |
| DDL/DCL | 불가 | 가능 | 가능 |
| 이식성 | 매우 좋음 | 중간 | 매우 나쁨 |

## Open SQL 제한 사항

따라서 Open SQL이 점점 강화되기 시작했다.

| 제한 항목 | 설명 |
|---|---|
| No expressions | CASE, 계산식, 함수 사용 불가 |
| Limited JOIN types | INNER JOIN 중심, 제약 많음 |
| No UNION / UNION ALL | 결과 집합 결합 불가 |
| Additional restrictions | 서브쿼리, 조건식 제한 |

## Open SQL 진화 단계

```text
SQL-92 Standard
      ▲
Standardized Open SQL   (≥ ABAP 7.5)
      ▲
Enhanced Open SQL       (ABAP 7.4)
      ▲
Open SQL                (Before ABAP 7.4)
```

## 핵심 해석

- 초기 Open SQL은 단순 조회 중심
- 이후 HANA 시대에 맞춰 점차 강화됨
- 강화된 Open SQL은 DB 계산, 함수, 표현식, 집합 연산을 지원하게 됨

# 9. New Open SQL Syntax

## 9.1 변경 1) Host Variable 표기: `@` 필수

```abap
" [OLD] (대략 ABAP 7.40 이전 스타일)
SELECT * FROM sbook
  INTO TABLE it_bookings
  WHERE customid = lv_customer.     " 변수 표시 없음

" [NEW] (New Open SQL)
SELECT FROM sbook
  FIELDS *
  WHERE customid = @lv_customer     " ABAP 변수는 @로 명확히 표시
  INTO TABLE @it_bookings.          " 타깃도 @ 가능(권장)
```

### 핵심
- ABAP 변수는 `@`로 명확히 표시
- SQL 영역과 ABAP 변수를 구분하기 쉬워짐

---

## 9.2 변경 2) INTO 절 위치: SELECT 끝으로 이동 (ABAP 7.50)

```abap
" [OLD] INTO가 FROM 전에 위치
SELECT carrid connid fldate
  INTO TABLE it_bookings
  FROM sbook
  WHERE customid = lv_customer
  UP TO 10 ROWS.

" [NEW] INTO는 SELECT 문 끝
SELECT carrid, connid, fldate
  FROM sbook
  WHERE customid = @lv_customer
  INTO TABLE @it_bookings
  UP TO 10 ROWS.
```

### 핵심
- `INTO` 절이 뒤로 이동하면서 SQL 문장 구조가 더 자연스러워짐

---

## 9.3 변경 3) `FIELDS` 키워드 도입 (ABAP 7.50)

```abap
" [OLD] SELECT 다음에 바로 필드 나열
SELECT carrid connid fldate
  FROM sbook
  WHERE customid = lv_customer
  INTO TABLE it_bookings.

" [NEW] FROM 먼저 → FIELDS로 필드 목록 명시 (ADT 자동완성 유리)
SELECT FROM sbook
  FIELDS carrid, connid, fldate
  WHERE customid = @lv_customer
  INTO TABLE @it_bookings.
```

### 핵심
- `FROM` 먼저, `FIELDS`로 필드 명시
- ADT 자동완성과 가독성 측면에서 유리

---

## 9.4 변경 4) `UNION / UNION ALL` 지원 + INTO는 마지막에 1번만

```abap
" [OLD] UNION 미지원 → ABAP에서 합치기
SELECT id name FROM table_a INTO TABLE @DATA(it_a).
SELECT id name FROM table_b INTO TABLE @DATA(it_b).
APPEND LINES OF it_b TO it_a.

" [NEW] SQL에서 UNION 가능, INTO는 맨 끝에 1번
SELECT FROM table_a
  FIELDS id, name
UNION ALL
SELECT FROM table_b
  FIELDS id, name
INTO TABLE @DATA(it_union).
```

### 주의
- `SELECT 절에 *` 를 그대로 쓰지 않는 구조를 자주 사용
- `FIELDS` 명은 두 SELECT 간 구조가 맞아야 함
- `INTO`는 전체 UNION 문 마지막에 한 번만 사용

---

## 9.5 변경 5) Expressions(계산식) 지원 강화

```abap
" [OLD] DB에서 계산 어려움 → ABAP에서 후처리
SELECT carrid loccuram FROM sbook INTO TABLE @DATA(it_old).
LOOP AT it_old ASSIGNING FIELD-SYMBOL(<r>).
  <r>-loccuram = <r>-loccuram * 1.1.   " ABAP 계산
ENDLOOP.

" [NEW] DB에서 계산식 수행 (Code Push-down)
SELECT FROM sbook
  FIELDS carrid,
         loccuram,
         loccuram * 1.1 AS loccuram_taxed
  INTO TABLE @DATA(it_expr).
```

### 핵심
- 계산을 ABAP가 아니라 DB에서 수행
- 대표적인 Code Push-down 예시

---

## 9.6 변경 6) `CASE` 지원 (DB에서 조건 분기)

```abap
" [OLD] CASE를 ABAP에서 처리
SELECT carrid smoker FROM sbook INTO TABLE @DATA(it_old_case).
LOOP AT it_old_case ASSIGNING FIELD-SYMBOL(<c>).
  IF <c>-smoker = 'X'.
    " smoker_text = 'SMOKER'
  ELSE.
    " smoker_text = 'NON-SMOKER'
  ENDIF.
ENDLOOP.

" [NEW] CASE를 Open SQL에서 바로 처리
SELECT FROM sbook
  FIELDS carrid,
         smoker,
         CASE
           WHEN smoker = 'X' THEN 'SMOKER'
           ELSE 'NON-SMOKER'
         END AS smoker_text
  INTO TABLE @DATA(it_case).
```

### 핵심
- 조건 분기를 DB 레벨에서 처리 가능
- LOOP와 IF/ELSE 후처리를 줄일 수 있음

---

## 9.7 변경 7) `COALESCE` 지원 (NULL 대체)

```abap
" [OLD] NULL 처리 위해 ABAP에서 IF/IS INITIAL
SELECT carrid order_date fldate FROM sbook INTO TABLE @DATA(it_old_null).
" LOOP 돌며 order_date 비면 fldate로 대체하는 로직 필요

" [NEW] COALESCE로 DB에서 처리
SELECT FROM sbook
  FIELDS carrid,
         COALESCE( order_date, fldate ) AS effective_date
  INTO TABLE @DATA(it_coalesce).
```

### 핵심
- NULL 처리 로직을 ABAP 후처리 없이 DB에서 수행

---

## 9.8 변경 8) `CAST` 지원 (타입 변환)

```abap
" [OLD] ABAP에서 타입 변환
SELECT carrid loccuram FROM sbook INTO TABLE @DATA(it_old_cast).
" LOOP + CONV fltp( ... ) 등의 ABAP 변환 필요

" [NEW] DB에서 CAST로 변환
SELECT FROM sbook
  FIELDS carrid,
         CAST( loccuram AS FLTP ) AS amount_flt
  INTO TABLE @DATA(it_cast).
```

### 핵심
- 타입 변환을 DB에서 직접 수행 가능

---

## 9.9 변경 9) 수학 함수 지원 (DIV, MOD, ABS, FLOOR, CEIL 등)

```abap
" [OLD] ABAP에서 계산
SELECT carrid loccuram FROM sbook INTO TABLE @DATA(it_old_math).
" LOOP에서 abs, floor/ceil 등을 ABAP로 구현/호출

" [NEW] DB에서 함수 수행
SELECT FROM sbook
  FIELDS carrid,
         ABS( loccuram )    AS abs_amount,
         FLOOR( loccuram )  AS floor_amt,
         CEIL( loccuram )   AS ceil_amt
  INTO TABLE @DATA(it_math).
```

### 핵심
- 수학 함수도 DB에서 수행 가능
- ABAP LOOP 내 계산 로직 감소

---

## 9.10 변경 10) 문자열 결합 `&&` 및 문자열 함수 지원

```abap
" [OLD] ABAP에서 CONCATENATE
SELECT id name_first name_last FROM scustom INTO TABLE @DATA(it_old_str).
" LOOP에서 CONCATENATE name_first name_last INTO full_name SEPARATED BY space.

" [NEW] DB에서 문자열 결합 (&&)
SELECT FROM scustom
  FIELDS id,
         name_first && ' ' && name_last AS full_name
  INTO TABLE @DATA(it_concat).
```

```abap
" [NEW] 문자열 함수 예시
SELECT FROM scustom
  FIELDS id,
         UPPER( name_last )  AS last_upper,
         LOWER( name_first ) AS first_lower,
         LENGTH( name_last ) AS last_len
  INTO TABLE @DATA(it_str_func).
```

### 핵심
- 문자열 결합과 문자열 함수도 DB 처리 가능
- 후처리 `CONCATENATE`, `TRANSLATE`, `STRLEN` 등을 줄일 수 있음

---

## 9.11 변경 11) `ORDER BY / UP TO / OFFSET` 조합 (페이징)

```abap
" [OLD] 페이징/정렬 조합이 제한적이거나 후처리 필요
SELECT carrid connid fldate FROM sbook INTO TABLE @DATA(it_old_order).

" [NEW] DB에서 정렬 + 제한 + 오프셋 처리
SELECT FROM sbook
  FIELDS carrid, connid, fldate
  ORDER BY fldate DESC
  INTO TABLE @DATA(it_page)
  UP TO 20 ROWS
  OFFSET 10.
```

### 핵심
- 정렬, 제한, 오프셋을 DB에서 처리
- 페이지네이션 구현에 유리

# 10. New Open SQL 전체 핵심 요약

## 문법 변화 요약표

| 항목 | 변화 |
|---|---|
| Host Variable | `@` 사용 |
| INTO 위치 | 문장 뒤쪽 |
| 필드 명시 | `FIELDS` 사용 |
| 집합 연산 | `UNION / UNION ALL` 가능 |
| 계산식 | Expression 지원 |
| 조건 분기 | `CASE` 지원 |
| NULL 처리 | `COALESCE` 지원 |
| 타입 변환 | `CAST` 지원 |
| 함수 | 수학/문자열 함수 지원 |
| 페이징 | `ORDER BY + UP TO + OFFSET` 가능 |

## 최종 한 줄 정리

> **ABAP on HANA의 핵심은 "데이터를 ABAP로 가져와 계산하는 것"이 아니라, "DB에서 최대한 계산을 끝내고 결과만 가져오는 것"이다.**  
> 그리고 이를 실무에서 가장 대표적으로 구현하는 방식이 **New Open SQL**과 **Code Push-down**이다.
