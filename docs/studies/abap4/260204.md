## CALCULATION 1

![image.png](attachment:1df45647-8465-4f0a-97a4-d5ecb0f00db6:image.png)

→ 클래스 생성 안 인터페이스 추가 

![image.png](attachment:e2736e1a-996f-4ac1-874f-f3911bbcf5db:image.png)

→ 메소드 안 작성할 코드 작성 

```csharp
CLASS ZCL_G01_COMPUTE_01 DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC .

  PUBLIC SECTION.
    INTERFACES IF_OO_ADT_CLASSRUN .
  PROTECTED SECTION.
  PRIVATE SECTION.
ENDCLASS.

CLASS ZCL_G01_COMPUTE_01 IMPLEMENTATION.

  METHOD IF_OO_ADT_CLASSRUN~MAIN.

* Declarations

    DATA number1 TYPE i.
    DATA number2 TYPE i.

    DATA result TYPE p LENGTH 8 DECIMALS 2.

* Input Values

    number1 = -8.
    number2 = 3.

* Calculation

*    DATA(result) = number1 / number2.

    result = number1 / number2.

    DATA(output) = |{ number1 } / { number2 } = { result }|.

* Output

    out->write( output ).
  ENDMETHOD.
ENDCLASS.
```

1. 이 코드는 ADT에서 실행하는 클래스이며 실행 시작점은 `IF_OO_ADT_CLASSRUN~MAIN` 메소드이다. 실행하면 MAIN이 호출되고 `out->write( )`로 결과를 출력한다.
2. 클래스 정의부 특징은 `PUBLIC FINAL CREATE PUBLIC` 이다.
    1. `PUBLIC` : 외부에서 접근 가능
    2. `FINAL` : 상속 불가
    3. `CREATE PUBLIC` : 외부에서 인스턴스 생성 가능
        
        `CREATE PRIVATE`와 달리 재사용/생성에 열려있는 형태다.
        
3. 선언된 변수는 3개다.
    1. `number1 TYPE i` : 첫 번째 정수 입력값
    2. `number2 TYPE i` : 두 번째 정수 입력값
    3. `result TYPE p LENGTH 8 DECIMALS 2` : 나눗셈 결과를 소수 2자리로 저장하기 위한 Packed Number
4. 입력값은 하드코딩 되어 있다.
    - `number1 = -8`
    - `number2 = 3`
        
        여기서 포인트는 “음수 / 양수 나눗셈” 테스트 케이스라는 점이다.
        
5. 계산 로직은 나눗셈 하나만 수행한다.
    - `result = number1 / number2.`
        
        따라서 이 예제는 CASE 분기 없이 “연산 1개”를 단순 실행하는 형태다.
        
6. 주석 처리된 라인 `DATA(result) = number1 / number2.` 는 같은 의미의 “인라인 선언 + 계산” 예시다.
    - 하지만 이미 위에서 `result`를 선언했기 때문에 이 코드는 그대로 쓰면 중복 선언이 된다.
    - 즉 학습 포인트는 “인라인 선언 방식도 가능하다” 정도이며, 이 코드에서는 사용하지 않았다.
7. `result` 타입이 `p DECIMALS 2` 이므로 나눗셈 결과는 소수점 2자리로 저장/표현된다.
    - `8 / 3`은 정수로 떨어지지 않으므로 소수 결과가 발생한다.
    - 결과는 시스템의 계산/반올림 규칙에 따라 소수 2자리로 저장된다.
8. 출력 문자열은 문자열 템플릿으로 생성한다.
    - `DATA(output) = |{ number1 } / { number2 } = { result }|.`
        
        여기서 `DATA(output)`는 인라인 선언이며 `output`은 `string`으로 추론되어 생성된다.
        
9. 최종 출력은 `out->write( output ).` 한 줄로 처리한다.
    
    실행 시 콘솔에는 `-8 / 3 = (계산 결과)` 형태로 출력된다.
    
10. 이 코드에서 예외 처리는 없다.
    - `number2`가 0이면 `cx_sy_zerodivide` 런타임 예외(덤프) 가능성이 있다.
    - 따라서 실무/재사용 목적이라면 `TRY...CATCH cx_sy_zerodivide` 또는 사전 체크(`IF number2 = 0`)가 필요하다.
11. 재사용 관점 개선 포인트는 다음과 같다.
    1. 하드코딩 입력을 파라미터로 분리
    2. 0 나누기 방어 로직 추가
    3. 계산/출력을 분리하여 계산 메소드화

## CALCULATION 2 - ABAP Branch (CASE) & 예외 처리

```csharp
class ZCL_G01_BRANCH definition
  public
  FINAL
  create PUBLIC .

public section.
    INTERFACES if_oo_adt_classrun .
protected section.
private section.
ENDCLASS.

CLASS ZCL_G01_BRANCH IMPLEMENTATION.

  METHOD if_oo_adt_classrun~main.

* Declarations
**************************

    DATA number1 TYPE i.
    DATA number2 TYPE i.

    DATA result  TYPE p LENGTH 8 DECIMALS 2.

    DATA op      TYPE c LENGTH 1.

    DATA output TYPE string.

* Input Values
**************************

    number1 = 123.
    number2 = 0.
    op      = '/'.

* Calculation
**************************

* a) calculation based on op
*    CASE op.
*      WHEN '+'.
*        result = number1 + number2.
*      WHEN '-'.
*        result = number1 - number2.
*      WHEN '*'.
*        result = number1 * number2.
*      WHEN '/'.
*        result = number1 / number2.
*    ENDCASE.
*
*    DATA(output) = |{ number1 } { op } { number2 } = { result }|.

* b) handle invalid operator
*    CASE op.
*      WHEN '+'.
*        result = number1 + number2.
*      WHEN '-'.
*        result = number1 - number2.
*      WHEN '*'.
*        result = number1 * number2.
*      WHEN '/'.
*        result = number1 / number2.
*      WHEN OTHERS.
*        output = |'{ op }' is not a valid operator!|.
*    ENDCASE.
*
*    IF output IS INITIAL.  "no error so far
*      output = |{ number1 } { op } { number2 } = { result }|.
*    ENDIF.

* c) handle division by zero

    CASE op.
      WHEN '+'.
        result = number1 + number2.
      WHEN '-'.
        result = number1 - number2.
      WHEN '*'.
        result = number1 * number2.
      WHEN '/'.

        TRY.
            result = number1 / number2.
          CATCH cx_sy_zerodivide.
            output = |Division by zero is not defined|.
        ENDTRY.

      WHEN OTHERS.

        output = |'{ op }' is not a valid operator!|.

    ENDCASE.

    IF output IS INITIAL.  "no error so far

      output = |{ number1 } { op } { number2 } = { result }|.

    ENDIF.

* Output
**************************

    out->write( output ).

  ENDMETHOD.
ENDCLASS.
```

1. 이 코드는 ADT에서 실행하는 클래스이며 실행 시작점은 `IF_OO_ADT_CLASSRUN~MAIN` 메소드이다. 프로그램처럼 실행하면 MAIN이 호출되고, 마지막에 `out->write( )`로 출력한다.
2. 클래스 정의는 `PUBLIC CREATE PRIVATE` 이다. 즉 외부에서 `CREATE OBJECT`로 인스턴스를 만들어 쓰는 용도라기보다, ADT 실행용 샘플 클래스 성격이 강하다.
3. 변수 선언은 5개다.
    1. `number1 TYPE i` : 첫 번째 정수 입력값
    2. `number2 TYPE i` : 두 번째 정수 입력값
    3. `result TYPE p LENGTH 8 DECIMALS 2` : 연산 결과(소수 2자리까지 표현 가능)
    4. `op TYPE c LENGTH 1` : 연산자 1글자(+ - * /)
    5. `output TYPE string` : 최종 출력 문자열(정상 결과 또는 오류 메시지)
4. 입력값은 코드에 하드코딩 되어 있다.
    - `number1 = 123`
    - `number2 = 0`
    - `op = '/'`
5. `a)`, `b)` 블록은 실행되지 않는다(주석 처리). 학습 단계별 버전이다.
    - a) 단순 CASE 분기만 있고 오류 처리 없음
    - b) `WHEN OTHERS`로 잘못된 연산자 처리 추가
    - 하지만 a)와 b)는 `/`에서 0으로 나누면 덤프 위험이 있다
6. 실제 실행되는 로직은 `c) handle division by zero` 아래의 `CASE op.` 블록이다. 연산자에 따라 분기해서 계산하거나 오류 메시지를 만든다.
7. `CASE op`의 `WHEN '+'`, `WHEN '-'`, `WHEN '*'`는 예외 없이 단순 계산을 수행한다.
    - `result = number1 + number2`
    - `result = number1 - number2`
    - `result = number1 * number2`
8. `WHEN '/'`는 나눗셈이라서 예외 가능성이 있어 `TRY...CATCH`로 감싼다.
    - `TRY` 안에서 `result = number1 / number2` 수행
    - `number2`가 0이면 ABAP 런타임 예외 `cx_sy_zerodivide` 발생
    - `CATCH cx_sy_zerodivide`에서 `output = |Division by zero is not defined|`로 오류 메시지 세팅
        
        이 처리가 없으면 프로그램이 덤프(런타임 오류) 날 수 있다.
        
9. `WHEN OTHERS`는 연산자가 `+ - * /` 중 하나가 아닐 때 실행된다.
    - `output = |'{ op }' is not a valid operator!|`
        
        즉 입력 검증(방어 로직) 역할이다.
        
10. `CASE`가 끝난 후 `IF output IS INITIAL.`로 “오류가 있었는지”를 판단한다.
    - `output`이 비어 있으면 정상 계산으로 간주하고 결과 문자열을 만든다.
    
    `output = |{ number1 } { op } { number2 } = { result }|`
    
    - `output`이 비어 있지 않으면 이미 오류 메시지가 들어 있으므로 그대로 유지한다.
11. 최종적으로 `out->write( output ).`로 화면(콘솔)에 출력한다.
    - 정상: `123 + 10 = 133.00` 같은 형태
    - 현재 입력값(123 / 0): `Division by zero is not defined`가 출력된다.
12. 이 코드의 핵심 포인트는 “분기 + 오류 처리”다.
    - `CASE`로 연산자별 분기 처리
    - `WHEN OTHERS`로 잘못된 연산자 처리
    - `/`에서 `TRY...CATCH cx_sy_zerodivide`로 0 나누기 덤프 방지
    - 오류가 없을 때만 결과 문자열을 생성(`output IS INITIAL` 체크)
13. 재사용 관점에서의 한계는 계산 로직이 `MAIN` 안에 박혀 있다는 점이다. 다른 클래스/프로그램에서 재사용하려면 계산 부분을 별도 메소드로 분리해야 한다(입력 파라미터 받고 결과 문자열 또는 예외로 반환).

## INTERNAL TABLES - 피보나치 수열 실습

![image.png](attachment:bf193640-9e52-4d38-b424-d4087f0bae1e:image.png)

→ 클래스 생성 

![image.png](attachment:57bfdd4c-c78c-4674-9db5-8098140c6a3d:image.png)

→ 결과 화면 

```csharp
CLASS ZCL_G01_ITERATE DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC .

  PUBLIC SECTION.

    INTERFACES IF_OO_ADT_CLASSRUN .
  PROTECTED SECTION.
  PRIVATE SECTION.
ENDCLASS.

CLASS ZCL_G01_ITERATE IMPLEMENTATION.

  METHOD IF_OO_ADT_CLASSRUN~MAIN.
  
    CONSTANTS max_count TYPE i VALUE 20.

    DATA numbers TYPE TABLE OF i.
    DATA output TYPE TABLE OF string.

* Iteration to calculate the Fibonacci numbers

    DO max_count TIMES.

      CASE sy-index.
        WHEN 1.
          APPEND 0 TO numbers.
        WHEN 2.
          APPEND 1 TO numbers.
        WHEN OTHERS.

          APPEND numbers[  sy-index - 2 ]
               + numbers[  sy-index - 1 ]
              TO numbers.

      ENDCASE.

    ENDDO.

* Loop over numbers to  prepare the formatted output

    DATA(counter) = 0.
    LOOP AT numbers INTO DATA(number).

      counter = counter + 1.

      APPEND |{ counter WIDTH = 4 }: { number WIDTH = 10 ALIGN = RIGHT }|
          TO output.

      out->write(  data   = output
                   name   = |The first { max_count } Fibonacci Numbers| ) .

    ENDLOOP.
    
  ENDMETHOD.
ENDCLASS.
```

- 이 코드는 ADT에서 실행하는 클래스이며 실행 시작점은 `IF_OO_ADT_CLASSRUN~MAIN` 메소드이다. 실행하면 MAIN이 호출되고 결과를 `out->write`로 출력한다.
- 구조
    
    ```
    DO TIMES
     └─CASE sy-index
         ├─1 →0
         ├─2 →1
         └─ n → n-1 + n-2
            ↓
     numbers 테이블 완성
            ↓
    LOOP AT numbers
            ↓
     output 테이블
            ↓
    out->write
    ```
    
- 클래스 정의는 `PUBLIC FINAL CREATE PUBLIC` 이다.
    1. 외부 접근 가능(PUBLIC)
    2. 상속 불가(FINAL)
    3. 외부 인스턴스 생성 가능(CREATE PUBLIC)
- 상수 `max_count TYPE i VALUE 20`은 “피보나치 수를 몇 개 만들 것인지”를 결정한다. 현재는 처음 20개를 생성한다.
- 내부 테이블이 2개 선언된다.
    1. `numbers TYPE TABLE OF i` : 피보나치 수를 순서대로 저장하는 정수 테이블
    2. `output TYPE TABLE OF string` : 화면 출력용으로 포맷된 문자열들을 저장하는 테이블
- 피보나치 계산은 `DO max_count TIMES.` 반복문에서 수행된다.
    - `DO ... TIMES`는 지정 횟수만큼 반복하며, 반복 횟수는 `sy-index`(1부터 시작)로 확인한다.
- 반복문 안에서 `CASE sy-index`로 초기값과 일반 케이스를 분기한다.
    1. `WHEN 1.` : 첫 번째 값으로 0을 `numbers`에 추가
    2. `WHEN 2.` : 두 번째 값으로 1을 `numbers`에 추가
    3. `WHEN OTHERS.` : 그 이후는 직전 2개를 더해 다음 값을 만든다
- `WHEN OTHERS`에서 사용하는 핵심이 “테이블 표현식(Table Expression)”이다.
    - `numbers[ sy-index - 2 ]` : (이론상) n-2번째 요소
    - `numbers[ sy-index - 1 ]` : (이론상) n-1번째 요소
    
    > **n번째 값 = (n-1번째 값) + (n-2번째 값)**
    > 
    
    예:
    
    - 3번째 = 2번째 + 1번째
    - 4번째 = 3번째 + 2번째
    - 5번째 = 4번째 + 3번째
    
    → 둘을 더해서 `APPEND ... TO numbers`로 다음 피보나치 값을 추가한다.
    
- 여기서 중요한 전제는 `numbers`가 “1부터 차곡차곡 채워진 표준 테이블”이라는 점이다. 이 전제가 깨지면 `numbers[ idx ]` 접근에서 런타임 에러가 날 수 있다(해당 인덱스가 없을 때).
- 첫 번째 DO 루프가 끝나면 `numbers`에는 20개의 피보나치 수가 저장된다. 즉 계산 단계와 출력 단계를 분리해둔 구조다.
- 두 번째 단계는 출력 포맷팅이며 `LOOP AT numbers INTO DATA(number).`로 `numbers`를 순회한다.
    - `DATA(number)`는 인라인 선언으로 현재 행의 값을 받는다.
- 출력용 카운터는 `DATA(counter) = 0.`으로 초기화하고, 루프마다 `counter = counter + 1.`로 증가시킨다.
    - 사실 `sy-tabix`를 써도 되지만, 학습 목적상 카운터 변수를 따로 둔 형태다.
- 출력 문자열은 문자열 템플릿 + 포맷 옵션으로 만든다.
    - `|{ counter WIDTH = 4 }: { number WIDTH = 10 ALIGN = RIGHT }|`
        
        의미는 다음과 같다.
        
    1. `counter`를 폭 4로 맞춰 출력(자리 정렬)
    2. `number`를 폭 10으로 맞추고 오른쪽 정렬
    3. `"번호: 값"` 형태의 행 문자열을 만든다.
- 만들어진 문자열은 `APPEND ... TO output`로 출력 테이블에 누적된다. 즉 `output`은 “출력 라인 목록”이다.
- 현재 코드의 출력 호출 위치가 `LOOP` 안에 있다.
    - `out->write( data = output name = ... )`가 반복마다 실행된다.
    - 그래서 출력이 1줄, 2줄, 3줄… 누적된 상태로 매번 다시 출력되는 형태가 된다.
    - 일반적으로는 `LOOP`가 끝난 뒤 한 번만 `out->write`를 호출하는 것이 자연스럽고 효율적이다.
- `out->write`의 `name = |The first { max_count } Fibonacci Numbers|`는 출력 블록에 제목을 붙이는 용도다. `max_count`가 바뀌면 제목도 자동으로 바뀐다.
- 정리하면 이 예제의 학습 포인트는 다음과 같다.
    1. `DO ... TIMES` 반복과 `sy-index` 사용
    2. `CASE`로 초기값 처리(피보나치의 첫 2항)
    3. 내부 테이블에 값 누적(`APPEND`)
    4. 테이블 표현식 `itab[ idx ]`로 인덱스 접근
    5. 문자열 템플릿의 폭/정렬 포맷 사용
    6. `LOOP AT ...`로 테이블 순회 후 출력 데이터 준비
- 개선 포인트(코드 품질 관점)는 다음 2가지가 핵심이다.
    1. `out->write`는 루프 밖에서 1회 호출하는 편이 좋다(현재는 루프마다 반복 출력)
    2. `numbers`는 “반드시 해당 인덱스가 존재한다”는 전제에 의존하므로, 구조를 바꾸면 인덱스 오류 가능성이 있다

A. 글로벌 클래스의 역할

```abap
CLASS ZCL_G01_LOCAL_CLASS DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC.
```

1. 이 클래스는 **전역(Global) 클래스**다.
2. ADT에서 실행 가능한 프로그램 역할을 한다.
3. `IF_OO_ADT_CLASSRUN`을 구현했기 때문에
    
    → **Run(F9)** 하면 `MAIN` 메소드가 실행된다.
    
4. 이 클래스는 **컨트롤러 / 실행 담당** 역할이다.

B. 글로벌 클래스의 특징 요약

- `PUBLIC` : 다른 객체에서 접근 가능
- `FINAL` : 상속 불가
- `CREATE PUBLIC` : 외부에서 객체 생성 가능
- `IF_OO_ADT_CLASSRUN~MAIN` : 실행용 진입 메소드

 즉, **프로그램 본체** 역할

C. 로컬 클래스란 무엇인가

```abap
CLASS lcl_connection DEFINITION.
```

1. **로컬 클래스(Local Class)** 는
    - 글로벌 클래스 안에서만 사용 가능
    - 해당 프로그램 전용 보조 클래스
2. 다른 프로그램이나 클래스에서는 접근 불가
3. 보통 **헬퍼 클래스 / 내부 모델 클래스**로 사용

D. 로컬 클래스 `lcl_connection`의 의미

```abap
DATA carrier_id    TYPE /dmo/carrier_id.
DATA connection_id TYPE /dmo/connection_id.
```

- 항공사 ID (`carrier_id`)
- 항공편 연결 ID (`connection_id`)

→ **“항공편 연결(Connection) 하나를 표현하는 객체”**

E. 로컬 클래스의 속성 설명

```abap
DATA carrier_id    TYPE /dmo/carrier_id.
DATA connection_id TYPE /dmo/connection_id.
```

1. 인스턴스 속성
2. 객체마다 서로 다른 값을 가질 수 있음
3. 하나의 항공 연결 정보를 담는 데이터 구조

F. CLASS-DATA의 의미

```abap
CLASS-DATA conn_counter TYPE i.
```

- 클래스 전체가 공유하는 변수
- `lcl_connection` 객체가 몇 개 만들어졌는지 카운트 용도로 추정
- 모든 인스턴스가 **하나의 값**을 공유

→ 연결 객체 개수 카운터 용도

G. 로컬 클래스 접근 제한 구조

```abap
PUBLIC SECTION.
PROTECTED SECTION.
PRIVATE SECTION.
```

- 현재는 `PUBLIC SECTION`만 사용
- 학습용으로 **클래스 구조 골격만 만들어 둔 상태**
- 이후 메소드/로직 추가 가능

H. 로컬 클래스 구현부

```abap
CLASS lcl_connection IMPLEMENTATION.
ENDCLASS.
```

- 아직 메소드 구현 없음
- **데이터 구조만 정의된 상태**
- 다음 단계에서 생성자, 메소드 추가 가능

I. 글로벌 클래스의 MAIN 메소드

```abap

ENDMETHOD.
```

- 나중에 여기T서 `lcl_connection` 객체 생성, 데이터 세팅, 출력 테스트 등을 하게 될 구조

→ 결과 화면 

1. 전체 목적은 로컬 클래스(`lcl_connection`)에 메소드(세팅/출력)를 만들고, 글로벌 클래스(`zcl_g01_methods`)에서 객체를 여러 개 생성해 테이블에 담은 뒤 출력하는 OOP 예제다. 
    1. 이전처럼 필드를 직접 만지는 방식 대신 메소드 호출로 값 세팅을 강제하고, 유효성 검증 실패 시 예외로 제어하는 흐름을 보여준다.
2. 글로벌 클래스 `zcl_g01_methods`는 실행 컨트롤러 역할이다. 
    1. `IF_OO_ADT_CLASSRUN~MAIN` 안에서 `connection TYPE REF TO lcl_connection`(단일 객체 참조)와 `connections TYPE TABLE OF REF TO lcl_connection`(객체 참조 목록)을 만들고, `NEW #( )`로 객체 3개를 생성한 다음 각각 `set_attributes`로 값(LH/0400, AA/0017, SQ/0001)을 세팅하고 `APPEND`로 테이블에 저장한다. 
    2. 즉 “객체 생성 → 값 설정 → 목록에 보관” 패턴이다.
3. 값 세팅 부분이 전 버전과 가장 큰 차이인데, 원래 하려던 직접 대입(`connection->carrier_id = ...`)은 주석 처리되어 있고, 대신 `connection->set_attributes( ... )`를 호출한다.
    1. 이 호출은 `TRY...CATCH cx_abap_invalid_value`로 감싸져 있다. 의미는 “세팅 메소드가 입력값이 이상하면 예외를 던질 수 있으니, 호출자는 실패를 잡아 처리한다”이다. 
    2. 실패하면 `out->write( 'Method call failed' )`가 실행되고, 성공하면 `APPEND connection TO connections`가 수행된다(즉 실패한 객체는 목록에 안 들어감)
4. 출력 단계에서는 `LOOP AT connections INTO connection.`으로 참조 테이블을 순회하면서 각 객체에 대해 `out->write( connection->get_output( ) )`를 호출한다. 
    1. 여기서 출력 문자열을 MAIN에서 만들지 않고, 객체가 자신의 상태를 “어떻게 보여줄지”를 `get_output`으로 제공한다. ***이게 캡슐화 포인트*
5. 로컬 클래스 `lcl_connection` 정의부를 보면, 실제 데이터는 `carrier_id`, `connection_id` 두 필드가 들고 있다. `CLASS-DATA conn_counter`는 클래스 전체 공유 변수인데, 현재 코드에서는 증가/사용 로직이 없어서 확장용(인스턴스 카운트 같은 용도)으로만 선언된 상태다.
6. `set_attributes` 메소드는 사실상 Setter 역할이다.
    1. IMPORTING으로 두 값을 받고, 둘 중 하나라도 INITIAL이면 `RAISE EXCEPTION TYPE cx_abap_invalid_value`로 예외를 발생시켜 “빈 값 금지” 규칙을 강제한다. 검증이 통과하면 `carrier_id = i_carrier_id`, `connection_id = i_connection_id`로 객체 내부 상태를 세팅한다. 즉 이 메소드가 “객체가 유효한 상태만 갖도록 하는 관문”이다.
7. `get_output` 메소드는 Getter라기보단 표현용 조회 메소드이다. 
    1. `RETURNING VALUE(r_output) TYPE string_table`로 여러 줄 문자열 테이블을 반환한다. 구현은 구분선 한 줄 + Carrier/Connection 라인을 `APPEND |...| TO r_output`로 쌓는다. 그래서 호출자(MAIN)는 포맷을 몰라도 `get_output()`만 호출하면 동일한 형식으로 출력된다.
8. 이 코드에서 주의할 점/개선 포인트
    1.  `lcl_connection`의 필드가 `PUBLIC SECTION`에 있어 이론상 외부에서 직접 변경이 가능하다(주석 처리로 “직접 대입하지 마”를 보여주지만, 구조적으로 막진 못함). 캡슐화를 제대로 하려면 필드를 `PRIVATE SECTION`으로 옮기고 메소드로만 접근하게 하는 게 정석이다.
    2.  `get_output` 반환 타입이 `string_table`인데, `out->write`가 테이블을 출력할 수 있어서 동작은 자연스럽지만, “한 줄 string 반환”을 원하면 `TYPE string`으로 바꿔도 된다(요구사항에 따라). 
    3. `conn_counter`는 선언만 되어 있으니 실제 목적이 있으면 생성자에서 증가시키는 식으로 연결해야 의미가 생긴다.
