# Defining a Basic CDS View 이론

## 1. 여기서 배우는 것

- CDS Data Definition 생성 방법
- FROM 절과 Element List 관리
- ABAP Annotation 사용 기초
- CDS에서 번역 가능한 텍스트 처리 개념
- 다른 CDS View를 데이터 소스로 사용하는 방법
- Element Annotation 전파 제어 개념

## 2. CDS View란 무엇인가

- CDS View는 **데이터를 저장하지 않는다**
- 항상 **하나 이상의 데이터 소스**를 참조하여 데이터를 읽는다
    - 데이터베이스 테이블
    - 다른 CDS View
- SQL View보다 의미 중심(semantic) 모델링에 가깝다

## 3. Data Definition 생성 방법

### 3.1 기본 생성 절차

1. ADT(Project Explorer)에서 프로젝트 우클릭
2. `New → ABAP Repository Object`
3. `Core Data Services → Data Definitions` 선택
4. `Next`
5. 아래 정보 입력
    - Package
    - Name
    - Description
    - Original Language
    - Referenced Object (선택)

### 참고

- Referenced Object에 **DB 테이블 또는 CDS View**를 지정하면
    
    → 해당 객체의 필드를 기준으로 CDS View가 생성됨
    

### 3.2 단축 생성 방법

- 이미 패키지에 Data Definition이 있는 경우
    1. `Core Data Services → Data Definitions` 확장
    2. `Data Definitions` 우클릭
    3. `New Data Definition`

## 4. Data Definition Template

### 4.1 템플릿 사용 개념

- 새 Data Definition 생성 시
    - 빈 화면으로 시작 가능
    - 또는 **Template 기반으로 시작 가능**

### 4.2 권장 템플릿

- **Define View Entity**
    - 단순 CDS View 정의에 가장 적합
    - 하나의 데이터 소스를 기반으로 함

### 4.3 템플릿 재선택

- 에디터 하단의 **Templates 탭**에서 언제든 변경 가능
- 사용자 정의 템플릿 생성 가능
    - `Window → Preferences → Data Definition Templates`

## 5. CDS Data Definition 기본 구조

```abap
define view entity Z_XXX
  as select from <data_source>
{
  element_list
}
```

구성 요소:

- `define view entity` : CDS View 정의
- `as select from` : 데이터 소스 지정 (FROM 절)
- `{ }` : Element List

## 6. FROM 절 (데이터 소스)

- CDS View의 **데이터 기반(Data Foundation)**
- 지정 가능한 대상
    - 데이터베이스 테이블
    - 다른 CDS View

### 특징

- CDS View는 항상 FROM 절이 필요
- 코드 자동완성(Ctrl + Space) 사용 가능
- 자동완성에는 **데이터 소스로 적합한 객체만 표시**

## 7. Element List (요소 목록)

### 7.1 Element List란

- CDS View 사용자에게 **노출되는 필드 목록**
- 중괄호 `{ }` 안에 작성
- 콤마(`,`)로 구분
- **마지막 요소 뒤에는 콤마 사용 금지**

```abap
{
  field1,
  field2,
  field3
}
```

### 7.2 Projection 개념

- 가장 단순한 Element List
- 데이터 소스의 필드만 그대로 나열
- 계산식, 함수 없음

### 7.3 필드 추가 방법

- `{ }` 안에서 `Ctrl + Space`
- 데이터 소스의 컬럼 목록 표시됨

## 8. KEY 요소

### 8.1 KEY의 의미

- `KEY` 키워드는 CDS View의 **기본 키** 정의
- Element List의 **맨 앞에 하나의 블록으로 위치해야 함**

```abap
{
  key employee_id,
  first_name,
  last_name
}
```

### 8.2 주의사항

- KEY는 필수는 아니지만 **강력히 권장**
- KEY가 없으면 Syntax Check 경고 발생
- 일반적으로
    - View의 KEY = 데이터 소스의 KEY
- 예외
    - 클라이언트 필드(MANDT)는 보통 제외

## 9. Alias (AS)

### 9.1 Alias란

- Element 이름을 새로 정의
- `AS` 키워드 사용

```abap
employee_id as EmployeeId
```

### 9.2 Alias 사용 이유

- DB 필드는 대문자/소문자 구분 없음
- CDS View에서는
    - 가독성을 위해 Mixed Case 권장
- 예
    - `employee_id → EmployeeId`
    - `first_name → FirstName`

### 9.3 Alias 생략 시

- 데이터 소스의 필드명이 그대로 사용됨

## 10. Insert All Elements

### 10.1 기능 설명

- `Ctrl + Space → Insert all elements`
- 효과
    - 모든 필드 자동 추가
    - KEY 자동 반영
    - Mixed Case Alias 자동 생성
    - 클라이언트 필드는 제외

### 10.2 자동 적용 시점

- Referenced Object로
    - DB 테이블
    - CDS View
        
        를 지정해 생성한 경우
        
        → 기본적으로 Insert All Elements 적용됨
        

## 11. DDL Formatter & 주석

### 11.1 CDS 문법

- CDS Data Definition Language (DDL) 사용

### 11.2 Formatter 실행

- 에디터에서
    - `Source Code → Format`
    - 또는 `Ctrl + F1`

### 11.3 Formatter 설정

- `Window → Preferences`
- `DDL Formatter` 검색
- ABAP 시스템 기반 기본 포맷 제공
- 사용자 정의 프로파일 생성 가능

### 12.1 한 줄 주석 (//)

### 1) 행 중간에서 사용

```abap
birth_date as BirthDate, // 직원 생년월일
```

- `//` 이후의 모든 텍스트는 주석으로 처리됨
- 같은 줄에서 코드 뒤에 설명을 붙일 때 사용

### 2) 행 맨 앞에서 사용

```abap
// 이 행 전체는 주석이다
```

- 행의 시작에 `//`를 두면 **해당 행 전체가 주석**
- ABAP에서 행 맨 앞의  주석과 유사한 개념

### 12.2 블록 주석 (/* */)

```abap
entry_date /* 입사일 */ as EntryDate
```

- `/*` 와 `/` 사이의 텍스트만 주석
- `/` 뒤의 코드는 **정상 코드로 인식됨**

### 특징

- 한 줄 중간에 설명을 넣을 수 있음
- 여러 줄 주석도 가능

### 12.3 블록 주석 사용 시 주의사항

- `/* */` 주석은 **가독성이 떨어질 수 있음**
- 특히 한 줄이 길 경우
    - 주석 뒤의 실제 코드가 쉽게 눈에 띄지 않음
- 권장 방식
    - 설명이 길어질 경우
    - 코드를 **새 줄로 분리**
    - 또는 `//` 단독 행 주석 사용

### 12.4 주석 단축키

- ABAP과 동일
- 선택한 여러 행에 대해
    - `Ctrl + <` : 주석 추가
    - `Ctrl + >` : 주석 제거
- 행 맨 앞에 `//` 기준으로 처리됨

## 13. CDS View 예제에서 주석 해석

```abap
define view entity ZS4D430_R_Employee
  as select from zs4d430_employ
{
  key employee_id as EmployeeId,
  first_name      as FirstName,
  last_name       as LastName,
  birth_date      as BirthDate,   // 생년월일
  // entry_date    as EntryDate
}
```

- `// 생년월일`
    
    → birth_date 필드에 대한 설명
    
- `// entry_date as EntryDate`
    
    → 해당 필드는 **완전히 주석 처리되어 View에 포함되지 않음**

# Using ABAP Annotations in CDS Views 이론

## 1. 여기서 배우는 것

- CDS View에서 ABAP Annotation을 사용할 수 있다
- CDS에서 번역 가능한 텍스트(Translatable Texts)를 정의할 수 있다

## 2. SAP Annotation 개요

SAP Annotation은 CDS 객체에 **기술적·의미적 속성**을 부여하는 메타데이터이다.

평가 방식에 따라 두 가지로 구분된다.

### 2.1 ABAP Annotations

- ABAP Runtime에서 평가됨
- CDS 객체가 **활성화될 때** 또는 **ABAP SQL로 접근될 때** 영향
- ABAP SQL의 동작 방식에 직접적인 영향을 줄 수 있음

### 2.2 Framework-specific Annotations

- OData, UI, Analytics 등 **특정 프레임워크 전용**
- ABAP Runtime이나 Activation에는 영향 없음
- 각 프레임워크가 자체 API로 해석

## 3. Annotation의 Scope(적용 범위)

Annotation은 적용 대상에 따라 다섯 가지로 구분된다.

### 3.1 Entity Annotations

- CDS View / CDS View Entity 전체에 적용
- `DEFINE VIEW` 또는 `DEFINE VIEW ENTITY` 앞에 작성

### 3.2 Element Annotations

- SELECT List의 개별 필드에 적용
- 해당 필드 바로 앞에 작성

### 3.3 Parameter Annotations

- CDS View Parameter에 적용
- Parameter 선언 앞에 작성

### 3.4 Extension Annotations

- CDS View Extension에 적용
- `EXTEND VIEW` 또는 `DEFINE VIEW ENTITY` 앞에 작성

### 3.5 Function Annotations

- CDS Table Function에 적용
- `DEFINE TABLE FUNCTION` 앞에 작성

## 4. CDS View Entity에서 중요한 ABAP Annotations

### 4.1 `@AbapCatalog.dataMaintenance`

- CDS View Entity의 **데이터 미리보기 제한**
- 예: RESTRICTED → 직접 데이터 조회 제한

### 4.2 `@AbapCatalog.viewEnhancementCategory`

- CDS View Extension을 통해
    - 확장 가능한지
    - 어떻게 확장되는지 정의
- 일반적으로 `#NONE` 사용

### 4.3 `@AccessControl.authorizationCheck`

- ABAP SQL로 CDS 접근 시
    - 암묵적 권한 체크 수행 여부
- 값 예:
    - `#NOT_REQUIRED`
    - `#CHECK`

### 4.4 `@ObjectModel.usageType.<…>`

- CDS View Entity의 **성능 관련 특성 정의**
- 주요 하위 속성
    - `serviceQuality`
    - `sizeCategory`
    - `dataClass`

### 4.5 `@AbapCatalog.entityBuffer.definitionAllowed`

- CDS View Entity에 대해
    - 테이블 버퍼링 허용/비허용 설정

## 5. CDS DDIC-based View에서만 사용 가능한 Annotation

다음 Annotation들은 **CDS View Entity에서는 사용 불가**하고

**기존 CDS View(DDIC 기반)** 에서만 의미가 있다.

### 5.1 `@AbapCatalog.sqlViewName`

- CDS View에 대응되는 **DB View 이름**
- CDS View Entity에서는 필요 없음

### 5.2 `@ClientHandling.type`

- View의 Client Dependency 정의
- 기본값: `#INHERITED`
    - 데이터 소스 중 하나라도 Client-specific이면 View도 Client-specific

### 5.3 `@ClientHandling.algorithm`

- Client-specific View를 ABAP SQL로 접근할 때
    - Client 처리 방식 정의

### 5.4 `@AbapCatalog.compiler.compareFilter`

- Path Expression에서 Filter 조건 평가 방식 제어

### 5.5 `@AbapCatalog.buffering.<…>`

- SAP Buffering에서 CDS View의 버퍼링 정의
- 최신 CDS View Entity에서는 더 이상 사용되지 않음

## 6. View Element를 위한 ABAP Annotation (Semantics)

### 6.1 Amount Semantics

```abap
@Semantics.amount.currencyCode: 'currency'
price,
currency,
```

- 금액 필드(price)가
    - 어떤 통화 필드(currency)를 사용하는지 명시
- CDS View Entity에서는 **필수**

### 6.2 Quantity Semantics

```abap
@Semantics.quantity.unitOfMeasure: 'DistanceID'
distance,
distid as DistanceID,
```

- 수량 필드(distance)가
    - 어떤 단위 필드(DistanceID)를 사용하는지 명시
- CDS View Entity에서는 **필수**

### 6.3 중요 특징

- Semantics Annotation 대부분은
    - CDS 위에 올라가는 Framework(UI, Analytics 등)가 해석
- CDS View Entity에서는
    - Amount / Quantity 연결 Annotation이 **강제됨**
- 과거 CDS View에서는 optional이었음

## 7. Grouped Annotations

### 7.1 개별 Annotation 작성 방식

```abap
@ObjectModel.usageType.serviceQuality: #X
@ObjectModel.usageType.sizeCategory: #S
@ObjectModel.usageType.dataClass: #MIXED
```

### 7.2 그룹화된 Annotation 작성 방식

```abap
@ObjectModel: {
  usageType: {
    serviceQuality: #X,
    sizeCategory:  #S,
    dataClass:     #MIXED
  }
}
```

### 특징

- 같은 Category의 Annotation을 묶어서 가독성 향상
- 기능적 차이는 없음
- 실무에서는 **Grouped 방식 권장**

## 9. Translatable Texts를 위한 Annotation

### 9.1 View Annotation

```abap
@EndUserText.label: 'Employee'
```

- CDS View 전체에 대한 라벨
- 최대 60자

### 9.2 Element Annotation

```abap
@EndUserText.label: 'Date of Birth'
@EndUserText.quickInfo: 'Date of birth of this employee'
birth_date as BirthDate,
```

- `label`
    - 최대 60자
- `quickInfo`
    - 문자열, 길이 제한 없음
    - Element Annotation에서만 가능

### 9.3 상속(Inheritance) 관련 주의사항

- Element가 Dictionary Data Element 기반이면
    - 기본적으로 Field Label에서 텍스트 상속
- CDS Annotation은
    - 해당 텍스트를 **덮어쓰고 싶을 때만 필요**

## 10. Active Annotations View

### 10.1 기능

- CDS Entity와 Element에 적용된
    - 모든 Annotation 요약 표시

# Defining Relationships Between Database Tables 이론

## 1. 학습 목표 (Lesson Objectives)

- 데이터 모델에서 **엔티티 간 관계** 이해
- ABAP Dictionary 테이블에서 **Foreign Key Dependency 정의**
- 관계의 **Cardinality(다중성)** 개념 이해

## 2. 데이터 모델에서의 관계 (Relationships in Data Models)

### 2.1 기본 개념

- 데이터 모델의 엔티티(Entity)는 **서로 관계를 가짐**
- 예시:
    - Employee ↔ Department
    - 직원은 부서에 소속됨
    - 부서는 한 명의 직원에 의해 관리될 수 있음

## 3. Cardinality / Multiplicity (다중성)

### 3.1 Cardinality란?

- 한 엔티티의 **하나의 인스턴스**가
    
    다른 엔티티의 **몇 개 인스턴스와 연결될 수 있는지**를 나타냄
    
- UML 표기법 사용 (`0..*`, `1..1` 등)

### 3.2 Employee – Department 예제

### 관계 1: 직원 → 부서 (is assigned to)

- 직원은 반드시 **하나의 부서**에 속함
    
    → `1..1`
    
- 부서는 **여러 명의 직원**을 가질 수 있음
    
    → `0..*`
    

### 관계 2: 부서 → 직원 (is led by)

- 부서는 **정확히 한 명의 리더 직원**을 가짐
    
    → `1..1`
    
- 직원은 **리더가 아닐 수도 있음**
    
    → `0..1`
    

## 4. ABAP CDS vs ABAP Dictionary 관계 표현 방식

| 구분 | 사용 개념 | 설명 |
| --- | --- | --- |
| ABAP CDS | Association | CDS View Entity 간 관계 |
| ABAP Dictionary | Foreign Key Dependency | 테이블 간 관계 |

## 5. Foreign Key Dependency (ABAP Dictionary)

### 5.1 Foreign Key Dependency란?

- 두 Dictionary 테이블 간의 **의미적 관계 정의**
- 한 테이블의 필드 값이
    
    **다른 테이블의 키 값 중 하나여야 함**
    

### 5.2 중요한 점 (What they are / are not)

### 하는 것

- 관계의 **문서화**
- (클래식 ABAP에서) 입력 체크, 값 도움(Value Help)

### 하지 않는 것

- DB 레벨의 Foreign Key 제약 ❌
- DB 무결성 강제 ❌

## 6. Foreign Key Table / Check Table

| 용어 | 설명 |
| --- | --- |
| Foreign Key Table | 참조하는 테이블 (값을 가진 쪽) |
| Check Table | 참조 대상 테이블 (허용 값 정의) |
| Check Field | Foreign Key Table에서 체크되는 필드 |

## 7. 예제: EMPLOYEE – DEPARTMENT

### 7.1 테이블 역할

- EMPLOYEE
    - Foreign Key Table
    - `DEPARTMENT_ID` 필드가 체크 대상
- DEPARTMENT
    - Check Table
    - Primary Key: `CLIENT`, `ID`

### 7.2 Foreign Key 구성

- Foreign Key = `(CLIENT, DEPARTMENT_ID)`
- Check Table Key = `(CLIENT, ID)`

조건:

- `EMPLOYEE.CLIENT = DEPARTMENT.CLIENT`
- `EMPLOYEE.DEPARTMENT_ID = DEPARTMENT.ID`

## 8. Foreign Key Dependency의 의미

- EMPLOYEE의 `DEPARTMENT_ID` 값은
    
    **같은 CLIENT의 DEPARTMENT 테이블에 존재해야 함**
    
- 존재하지 않으면 **유효하지 않은 값**

## 9. Foreign Key 정의 문법 (DEFINE TABLE)

```abap
department_id : department_id
  with foreign key depment
    where client = employee.client
      and id     = employee.department_id;
```

### 문법 규칙

- Foreign Key는 **Foreign Key Table 필드 정의 안에서 작성**
- `WITH FOREIGN KEY <check_table>`
- WHERE 절:
    - 왼쪽: Check Table의 Key 필드
    - 오른쪽: Foreign Key Table 필드
- 도메인은 반드시 동일해야 함

## 10. Foreign Key의 Semantic Attributes

### 10.1 Foreign Key Annotations

- `@AbapCatalog.foreignKey.label`
- `@AbapCatalog.foreignKey.keyType`
- `@AbapCatalog.foreignKey.screenCheck`
- (클래식 UI용 메시지 설정 가능)

※ 최신 ABAP / Fiori 환경에서는 대부분 의미 없음

## 11. Cardinality in Foreign Key Relationships

### 11.1 표기법

```abap
with foreign key [n, m]
```

| 값 | 의미 |
| --- | --- |
| n | Check Table 1건당 Foreign Key Table 레코드 수 |
| m | Foreign Key Table 1건당 Check Table 레코드 수 |

### 11.2 오른쪽 값(m) 먼저 결정

| m 값 | 의미 |
| --- | --- |
| 1 | 필수 (반드시 존재) |
| 0..1 | 선택 (없어도 됨) |

### 11.3 왼쪽 값(n)

| n 값 | 의미 |
| --- | --- |
| 1 | 정확히 하나 |
| 0..1 | 최대 하나 |
| 1..* | 하나 이상 |
| 0..* | 제한 없음 |

## 12. 예제 해석 (직원–부서)

- 직원은 반드시 부서에 속함
    
    → `m = 1`
    
- 부서는 여러 직원을 가질 수 있음
    
    → `n = 0..*`
    

결과:

```
[0..*, 1]
```

## 13. 핵심 정리

- Foreign Key Dependency는 **DB 제약이 아닌 ABAP 레벨 관계 정의**
- Cardinality는 **관계의 수적 특성**을 설명
- 현대 ABAP에서는
    - 관계 표현: CDS Association
    - 테이블 무결성: 애플리케이션 로직에서 관리

## 14. Lesson Summary

- 데이터 모델의 관계 이해
- Foreign Key Dependency 정의 가능
- Cardinality 해석 가능

# 1. 기본 전제 (꼭 먼저 이해)

- **부모 테이블 (Check Table)**
    
    → 참조 “당하는” 테이블
    
- **자식 테이블 (Foreign Key Table)**
    
    → 외래키를 “가지는” 테이블
    
- **관계 차수(cardinality)는 항상 자식 테이블에서 정의**
- 문법:

```abap
with foreign key [n, m] parent_table
```

# 2. 부모(Parent) 기준 관계 차수 3가지

> 부모 테이블 1건당 자식 레코드가 몇 개인가?
> 

이건 `[n, m]`에서 **왼쪽(n)** 값으로 표현됨.

## 2-1. 부모 1건 ↔ 자식 0..* (가장 일반적, 1:N)

### 의미

- 부모 1건에 자식이 **없을 수도 있고, 여러 개 있을 수도 있음**
- 가장 흔한 “마스터–트랜잭션 / 부서–사원” 구조

### 코드 예시

```abap
department_id : zg01_department_id
  with foreign key [0..*,1] zg01depment
    where client = zg01employ.client
      and id     = zg01employ.department_id;
```

### 해석

- 부서 1개 ← 사원 0명 이상
- 사원 1명 → 부서 정확히 1개

## 2-2. 부모 1건 ↔ 자식 0..1 (선택적 1:1)

### 의미

- 부모 1건에 자식이 **없거나 1개만**
- “부서 – 부서장”, “사원 – 개인 사물함” 같은 구조

### 코드 예시

```abap
department_head : zg01_employee_id
  with foreign key [0..1,0..1] zg01employ
    where client      = zg01depment.client
      and employee_id = zg01depment.department_head;
```

### 해석

- 사원 1명
→ 부서장의 역할을
→ 0번 또는 1번만 가질 수 있
- 부서 1개
→ 부서장을
→ 0명 또는 1명만 가질 수 있다

## 2-3. 부모 1건 ↔ 자식 1..* (드문 케이스)

### 의미

- 부모 1건에 **자식이 최소 1개 이상 반드시 존재**
- 실무에선 거의 없음 (데이터 생성 순서 제약이 큼)

### 코드 예시

```abap
parent_id : z_parent_id
  with foreign key [1..*,1] zparent
    where id = zchild.parent_id;
```

### 해석

- 부모가 있으면 반드시 자식이 하나 이상 있어야 함
- 생성/삭제 관리가 매우 까다로움

# 3. 자식(Child) 기준 관계 차수 3가지

> 자식 레코드 1건이 부모를 반드시 가지는가?
> 

이건 `[n, m]`에서 **오른쪽(m)** 값으로 표현됨.

## 3-1. 자식 → 부모 1 (필수)

### 의미

- 자식 레코드는 **반드시 부모를 참조**
- NULL 불가 관계

### 코드 예시

```abap
with foreign key [0..*,1] zg01depment
```

### 해석

- 사원은 반드시 부서에 소속됨
- FK 필드는 논리적으로 필수

## 3-2. 자식 → 부모 0..1 (선택)

### 의미

- 자식이 부모를 **가질 수도, 안 가질 수도 있음**

### 코드 예시

```abap
with foreign key [0..1,0..1] zg01employ
```

### 해석

- 부서에 부서장이 없을 수도 있음
- FK 필드 NULL 허용 의미

## 3-3. 자식 → 부모 N (❌ 불가능)

### 의미

- 자식 1건이 부모 여러 건을 동시에 참조
- **관계형 모델 + ABAP Dictionary에서 불가능**

### 대신 쓰는 방법

- 중간 테이블 (Mapping Table)

```
EMPLOYEE <-> EMPLOYEE_PROJECT <-> PROJECT
```

### 핵심 포인트

- `[*,*]` 같은 차수는 존재하지 않음
- 다대다는 반드시 테이블로 풀어야 함

# 4. 부모/자식 관계 차수 한눈 요약표

| 관점 | 차수 | 의미 |
| --- | --- | --- |
| 부모 기준 | 0..* | 부모 1건에 자식 여러 개 |
| 부모 기준 | 0..1 | 부모 1건에 자식 최대 1개 |
| 부모 기준 | 1..* | 부모 1건에 자식 최소 1개 |
| 자식 기준 | 1 | 자식은 부모 필수 |
| 자식 기준 | 0..1 | 자식은 부모 선택 |
| 자식 기준 | * | 불가능 (중간 테이블 필요) |

# 5. 이 단원의 진짜 학습 목표 한 문장

> 관계 차수(cardinality)는
> 
> 
> **부모/자식 양쪽의 의미를 한 줄의 foreign key 정의로 표현하는 것**이며,
> 
> **정의는 항상 자식 테이블에서만 수행한다.**
>
