## ZD750AW_ITAB_G01 - LOOP & VALUE #

→ 내가 선택한 도착 도시를 출발 도시로 하여 존재하는 예약 건 조회

```csharp
*&---------------------------------------------------------------------*
*& Report D750AW_ITAB_T1
*&---------------------------------------------------------------------*
*&
*&---------------------------------------------------------------------*
REPORT ZD750AW_ITAB_G01.

DATA: GT_FLIGHTS            TYPE STANDARD TABLE OF SPFLI WITH NON-UNIQUE KEY CARRID CONNID,
      GS_FLIGHT             TYPE SPFLI,
      GS_CONNECTING_FLIGHT  TYPE SPFLI,
      GT_CONNECTING_FLIGHTS LIKE GT_FLIGHTS.

PARAMETERS: PA_CAR TYPE SPFLI-CARRID,
            PA_CON TYPE SPFLI-CONNID.

START-OF-SELECTION.

  SELECT * FROM SPFLI INTO TABLE GT_FLIGHTS.

  READ TABLE GT_FLIGHTS INTO GS_FLIGHT WITH TABLE KEY CARRID = PA_CAR CONNID = PA_CON.

*  LOOP AT gt_flights INTO gs_connecting_flight WHERE cityfrom = gs_flight-cityto. " 주석 처리 
  LOOP AT GT_FLIGHTS INTO GS_CONNECTING_FLIGHT WHERE CITYFROM = GT_FLIGHTS[ CARRID = PA_CAR CONNID = PA_CON ]-CITYTO.   " 1)방법 1
    APPEND GS_CONNECTING_FLIGHT TO GT_CONNECTING_FLIGHTS. " 찾은 연결편을 연결편 목록 테이블에 추가한다."
  ENDLOOP.

  " 2)방법 2
  GT_CONNECTING_FLIGHTS = VALUE #( FOR CONN IN GT_FLIGHTS
  WHERE ( CITYFROM = GS_FLIGHT-CITYTO )
  ( CARRID = CONN-CARRID CONNID = CONN-CONNID CITYFROM = CONN-CITYFROM CITYTO = CONN-CITYTO ) ).

  WRITE: |Connecting Flights For Flight { PA_CAR }{ PA_CON } from { GS_FLIGHT-CITYFROM } to { GS_FLIGHT-CITYTO }:|.
  ULINE.
  LOOP AT GT_CONNECTING_FLIGHTS INTO GS_CONNECTING_FLIGHT.
    WRITE: / GS_CONNECTING_FLIGHT-CARRID,
    GS_CONNECTING_FLIGHT-CONNID,
    GS_CONNECTING_FLIGHT-CITYFROM,
    GS_CONNECTING_FLIGHT-CITYTO.
  ENDLOOP.
```

1. GT_FLIGHTS[ CARRID = PA_CAR CONNID = PA_CON ]-CITYTO : `GT_FLIGHTS`에서 키로 한 줄을 바로 꺼내서 그 줄의 `CITYTO` 필드를 가져온다. 즉, GT_FLIGHTS에서 키로 한 줄을 바로 찾아 그 줄의 CITYTO 필드를 즉시 가져오는 **table expression(신문법)**
2. **VALUE #( FOR CONN IN GT_FLIGHTS WHERE ( CITYFROM = GS_FLIGHT-CITYTO ) … )** :

LOOP + WHERE + APPEND 를 **하나의 표현식으로 만든 신문법 버전**.

결과적으로 연결편 테이블을 한 번에 생성한다.

### 템플릿 3) loop+append 대신 value+for로 필터 테이블 만들기

```abap
" gt_source에서 조건 맞는 것만 새 테이블로 만들기
gt_target = VALUE #(
```

## ZD750AW_G01_OSQL2 - OPEN SQL 2

: 추가 필드 3개를 **select 자체에서 sql expression으로 계산**

### 1. select 구문의 핵심 차이

S1 select 특징

- join해서 “원본 데이터”만 가져온다
    
    `c~form`, `c~name`, `c~discount`, `b~loccuram` 같은 계산 재료 포함
    
- 계산은 select 밖에서 함

S2 select 특징

- join한 뒤 field list에서 바로 계산해버림
    
    즉, `full_name`, `discounted_price`, `discounted`를 select 결과로 바로 만든다
    

### 2. S2에서 새로 등장한 신문법/표현식 포인트

2-1. 문자열 표현식으로 full_name 생성

```abap
c~form && ' ' && c~name AS full_name
```

- select 결과 컬럼에 `full_name`이라는 별칭으로 생성
- S1에서는 loop에서 `form && ' ' && name` 했던 걸 sql로 옮긴 것

2-2. numeric expression + cast로 discounted_price 생성

S2 코드 핵심:

```abap
CAST( b~loccuram AS FLTP ) *
( CAST( 100 AS FLTP ) - CAST( c~discount AS FLTP ) ) /
CAST( 100 AS FLTP ) AS discounted_price
```

- 이렇게 cast를 많이 하는 이유

- `/` 연산은 float 계열 계산이 필요할 때가 많고
- 정수/packed 혼합 계산에서 의도치 않은 타입 규칙을 피하려고
- 안전하게 전부 `FLTP`로 올려서 계산

대체안(주석으로 있는 Alternative)은 일부만 fltp로 cast해서 cast 개수를 줄이는 방식

2-3. case expression으로 discounted 생성

S2 코드:

```abap
CASE c~discount
  WHEN '000' THEN @abap_false
  ELSE @abap_true
END AS discounted
```

S1 loop의 이 부분을 sql로 옮김:

```abap
CASE gs_booking-discount.
  WHEN '000'. discounted = abap_true/false ...
ENDCASE.
```

- S2에서는 `@abap_false`, `@abap_true` 처럼 abap 상수를 sql 안에서 쓰기 위해 `@`를 붙임

### 3. loop 존재 여부 차이

| 항목 | S1 | S2 |
| --- | --- | --- |
| loop | 있음 | 없음(주석 처리) |
| 계산 위치 | abap loop | sql field list |
| modify transporting | 사용 | 불필요 |

- S1에서 loop가 하던 일

- full_name 만들기
- discounted_price 계산
- discounted 플래그 세팅
- MODIFY로 itab 업데이트
    
    → S2에서는 이 3개를 select가 이미 만들어서 loop 자체가 필요 없어짐

## ZD750AW_G01_OSQL3 - JOIN & UNION

### 0) 이 프로그램이 하려는 핵심 목표

- 항공사(`pa_car`) 기준으로 예약(`sbook`)을 보고
- 비즈니스 파트너별 결제 합계(`SUM( loccuram )`)를 뽑는다
- 비즈니스 파트너는 2종류
    1. 고객(`scustom`) → 타입 'FC'
    2. 여행사(`stravelag`) → 타입 'TA'
- 둘 결과를 합쳐서(`UNION`) 한 테이블(`gt_partners`)로 만들고 ALV 출력

### 1) 두 코드의 가장 큰 차이 한 줄

- 첫 코드: 고객 조회 select, 여행사 조회 select를 **각각 endselect 루프 + append로 따로 만들려던 예전 스타일(주석 처리됨)**
- 두 번째 코드: 두 select를 **UNION으로 한 번에 합쳐서 INTO TABLE로 끝내는 신문법 스타일(실제 동작)**

### 2) 데이터 선언 차이

| 항목 | 첫 코드 | 두번째 코드 |
| --- | --- | --- |
| gs_partner | 있음(워크에어리어) | 없음 |
| gt_partners | 있음 | 있음 |
- 예전 스타일은 `SELECT ... INTO gs_partner. ... APPEND ... ENDSELECT.` 구조라 work area 필요
- 신문법은 `INTO TABLE @gt_partners`로 한 번에 담아서 work area 불필요

### 3) select 로직 차이(구현 방식)

3-1) 예전 스타일(주석 처리된 로직이 의미하는 것)

- 고객용 select 1번 실행 → 결과 한 줄씩 받아서
    - `buspatype = 'FC'` 넣고
    - `append`
- 여행사용 select 1번 실행 → 결과 한 줄씩 받아서
    - `buspatype = 'TA'` 넣고
    - `append`
- 문제점
    - select를 두 번 따로 수행
    - endselect(커서 루프) 사용
    - append로 합치기
    - 코드 길고 반복 많음

3-2) 신문법 스타일(실제 동작 코드)

- select 1: 고객 결과를 그룹핑해서 만들고 buspatype을 sql에서 상수로 생성
- `UNION`
- select 2: 여행사 결과도 동일 구조로 만들고 buspatype을 sql에서 상수로 생성
- 마지막에 `INTO TABLE @gt_partners`로 한 번에 받음

장점

- 결과 합치기가 sql 단계에서 끝남
- abap loop/append 필요 없음
- 코드가 “무엇을 구하는지” 중심으로 읽힘

### 4) 신문법 코드 핵심 구간 해석

4-1) 고객(FC) 파트

```abap
SELECT c~id AS buspanum,
       ...
       SUM( b~loccuram ) AS paymentsum,
       'FC' AS buspatype
  FROM sbook AS b RIGHT OUTER JOIN scustom AS c
    ON b~customid = c~id
 WHERE carrid = @pa_car
   AND ( b~cancelled <> 'X' OR b~cancelled IS NULL )
 GROUP BY ...
```

- `SUM( b~loccuram )` : 파트너별 결제 합계
- `'FC' AS buspatype` : 결과 컬럼을 sql에서 바로 만들기(추가 필드)
- `GROUP BY` : 집계함수(sum)를 쓰면 group by 필수
- `RIGHT OUTER JOIN` : 고객 테이블(scust)을 기준으로 남기려는 의도(고객이 예약이 없어도 나오게 하려는 방향)
- 그런데 where에 `carrid = @pa_car` 같은 조건이 `b`쪽 필드라
    
    예약이 없는 고객은 `b~carrid`가 null이라 where에서 걸러질 수 있음
    
    “예약 없는 고객도 포함”이 목적이면 where 조건 위치/구성이 더 중요해짐
    

4-2) 여행사(TA) 파트

```abap
SELECT c~agencynum AS buspanum,
       ...
       SUM( b~loccuram ) AS paymentsum,
       'TA' AS buspatype
  FROM sbook AS b RIGHT OUTER JOIN stravelag AS c
    ON b~agencynum = c~agencynum
 WHERE carrid = @pa_car
   AND ( b~cancelled <> 'X' OR b~cancelled IS NULL )
 GROUP BY ...
```

- 고객 파트와 동일한 구조
- join 대상만 travel agency로 바뀜
- buspatype 상수만 'TA'

4-3) UNION

```abap
... (고객 select)
UNION
... (여행사 select)
INTO TABLE @gt_partners.
```

- `UNION`은 두 select 결과를 세로로 합침
- 조건: 두 select의 컬럼 수/순서/타입이 호환되어야 함
- 이 코드에서는 둘 다 buspanum, name, city, country, currency, paymentsum, buspatype 동일 구조라 union 가능

### 5) 핵심 로직을 이해하는 데 필요한 키워드 요약

| 키워드 | 의미 |
| --- | --- |
| SUM( ) | 파트너별 결제합 계산(집계) |
| GROUP BY | sum 쓰면 그룹핑 필수 |
| OUTER JOIN | 한쪽 데이터가 없어도 상대를 남김 |
| 'FC' / 'TA' AS buspatype | 결과에 구분 필드 추가 |
| UNION | 고객 결과 + 여행사 결과를 하나로 합침 |
| @pa_car / @gt_partners | host variable(신문법) |

### 6) 정리용 결론

- 예전 버전은 고객/여행사를 각각 조회하고 abap에서 buspatype을 채운 뒤 append로 합치려는 구조였다
- 신문법 버전은 sql에서 buspatype을 상수 컬럼으로 생성하고, 두 결과를 union으로 합쳐서 한 번에 internal table로 받는다
- 핵심은 “집계(sum) + group by + outer join + union”을 이용해 파트너별 결제합 리스트를 만든다는 점이다

    | 공통 처리 | 이름 결합, 환율 변환, ALV 출력 등 동일 |  |

### *환율 변환 및 추가 필드 가공*

```abap
LOOP AT gt_bookings INTO gs_booking.
  gs_booking-full_name = gs_booking-form && ` ` && gs_booking-name.

  CALL FUNCTION 'CONVERT_TO_LOCAL_CURRENCY'
    EXPORTING
      date             = gs_booking-order_date
      foreign_amount   = gs_booking-loccuram
      foreign_currency = gs_booking-loccurkey
      local_currency   = pa_cur
      type_of_rate     = 'M'
    IMPORTING
      local_amount     = gs_booking-converted_price
    EXCEPTIONS
      no_rate_found    = 1
      overflow         = 2
      no_factors_found = 3
      no_spread_found  = 4
      derived_2_times  = 5.
  IF sy-subrc <> 0.
    CLEAR gs_booking-converted_price.
  ENDIF.

  gs_booking-target_currency = pa_cur.

  MODIFY gt_bookings FROM gs_booking TRANSPORTING full_name converted_price target_currency.
ENDLOOP.
```

- `full_name`: 고객의 호칭(form)과 이름(name) 결합
- `CONVERT_TO_LOCAL_CURRENCY` 함수를 통해 예약 금액을 파라미터 기준 통화로 환산
- 실패 시 `converted_price` 필드를 비움
- `target_currency` 필드에 변환된 통화 정보 저장
- `MODIFY` 구문을 통해 가공된 필드만 내부 테이블에 업데이트 (`TRANSPORTING` 사용)
