```sql
*&---------------------------------------------------------------------*
*& Report ZABAP_56_G01
*&---------------------------------------------------------------------*
*& 항공편(SFLIGHT) 데이터 조회
*& - 선택 조건: 항공사, 연결편
*& - 계산 필드: 탑승률, 잔여 좌석
*&---------------------------------------------------------------------*
REPORT ZABAP_56_G01.

*---------------------------------------------------------------------*
* Data Declaration
*---------------------------------------------------------------------*
DATA: GT_FLIGHT TYPE BC400_T_FLIGHTS,
      GS_FLIGHT LIKE LINE OF GT_FLIGHT.

*---------------------------------------------------------------------*
* Selection Screen
*---------------------------------------------------------------------*
SELECT-OPTIONS:
  SO_CAR FOR GS_FLIGHT-CARRID,  "항공사
  SO_CON FOR GS_FLIGHT-CONNID.  "연결편

*---------------------------------------------------------------------*
* Old Open SQL (주석 처리됨)
*---------------------------------------------------------------------*
* 1. DB에서 데이터 조회
* 2. LOOP를 통해 ABAP에서 탑승률 계산
* 3. MODIFY로 내부 테이블 갱신
* → 처리 단계가 많고 성능 및 가독성 저하
*---------------------------------------------------------------------*
*SELECT CARRID CONNID FLDATE SEATSMAX SEATSOCC
*  FROM SFLIGHT
*  INTO CORRESPONDING FIELDS OF TABLE GT_FLIGHT
*  WHERE CARRID IN SO_CAR
*    AND CONNID IN SO_CON.
*
*LOOP AT GT_FLIGHT INTO GS_FLIGHT.
*  GS_FLIGHT-PERCENTAGE =
*    GS_FLIGHT-SEATSOCC * 100 / GS_FLIGHT-SEATSMAX.
*
*  MODIFY GT_FLIGHT FROM GS_FLIGHT.
*  CLEAR GS_FLIGHT.
*ENDLOOP.

*---------------------------------------------------------------------*
* New Open SQL (계산 필드 포함 – 주석 처리됨)
*---------------------------------------------------------------------*
* - DB 레벨에서 계산 수행
* - LOOP 불필요
* - DIVISION 함수로 0 나누기 방지 및 소수점 처리
*---------------------------------------------------------------------*
*SELECT CARRID, CONNID, FLDATE, SEATSMAX, SEATSOCC,
*       DIVISION( SEATSOCC * 100, SEATSMAX, 2 ) AS PERCENTAGE "계산을 DB 레벨에서 수행"
*  FROM SFLIGHT
*  WHERE CARRID IN @SO_CAR
*    AND CONNID IN @SO_CON
*  INTO CORRESPONDING FIELDS OF TABLE @GT_FLIGHT.

*---------------------------------------------------------------------*
* New Open SQL + 잔여 좌석 수 포함 (최종 사용)
*---------------------------------------------------------------------*
* PERCENTAGE : 좌석 점유율 (%)
* SEATSFREE  : 최대 좌석 - 예약 좌석
* @DATA      : Inline Declaration
*---------------------------------------------------------------------*
SELECT
    CARRID,                                "항공사 코드
    CONNID,                                "연결편 번호
    FLDATE,                                "비행 날짜
    SEATSMAX,                              "최대 좌석 수
    SEATSOCC,                              "예약된 좌석 수
    DIVISION( SEATSOCC * 100, SEATSMAX, 2 ) AS PERCENTAGE, "탑승률(%)
    SEATSMAX - SEATSOCC AS SEATSFREE       "남은 좌석 수 추가
  FROM SFLIGHT
  WHERE CARRID IN @SO_CAR                  "항공사 조건
    AND CONNID IN @SO_CON                  "연결편 조건
  INTO TABLE @DATA(GT_DATA).               "내부 테이블 선언 + 데이터 수신 동시 수행

*---------------------------------------------------------------------*
* Output
*---------------------------------------------------------------------*
CL_DEMO_OUTPUT=>DISPLAY( GT_DATA ).
```

→ New Open SQL은 계산을 DB에서 처리하여 LOOP를 제거하고 성능과 가독성을 동시에 개선한다.

## ZHA400_G01_NEW_OSQL – New Open SQL 실습

```sql
* Old Open SQL
  SELECT A~CARRID A~CARRNAME A~CURRCODE
      SUM( B~LOCCURAM ) AS REVENUE_FIRST    "Sum only bookings with class = 'F'
      SUM( B~LOCCURAM ) AS REVENUE_BUSINESS "Sum only bookings with class = 'C'
      SUM( B~LOCCURAM ) AS REVENUE_ECONOMY  "Sum only bookings with class = 'Y'
    FROM SCARR AS A LEFT OUTER JOIN SBOOK AS B
            ON A~CARRID = B~CARRID
    INTO TABLE CT_CARRIERS
    WHERE B~CANCELLED <> 'X' OR B~CANCELLED IS NULL
  GROUP BY A~CARRID A~CARRNAME A~CURRCODE
  ORDER BY A~CARRID.
    
* New Open SQL
  SELECT FROM SCARR AS A
  LEFT OUTER JOIN SBOOK AS B ON A~CARRID = B~CARRID
  FIELDS
    A~CARRID,
    A~CARRNAME,
    A~CURRCODE,
    SUM( CASE B~CLASS WHEN 'F' THEN B~LOCCURAM ELSE 0 END ) AS revenue_first,
    SUM( CASE B~CLASS WHEN 'C' THEN B~LOCCURAM ELSE 0 END ) AS revenue_business,
    SUM( CASE B~CLASS WHEN 'Y' THEN B~LOCCURAM ELSE 0 END ) AS revenue_economy
  WHERE B~CANCELLED <> 'X'
    OR B~CANCELLED IS NULL
  GROUP BY A~CARRID, A~CARRNAME, A~CURRCODE
  ORDER BY A~CARRID
  INTO TABLE @CT_CARRIERS.
```

### 📊 Old Open SQL vs New Open SQL 변화 비교표

| 구분 | Old Open SQL | New Open SQL | 변화 요약 |
| --- | --- | --- | --- |
| SELECT 구조 | `SELECT 필드 FROM 테이블` | `SELECT FROM 테이블 FIELDS 필드` | FROM을 먼저 쓰고 FIELDS로 필드 명시 |
| 필드 나열 방식 | SELECT 절에 바로 나열 | `FIELDS` 절에서 명시 | 가독성 및 구조 명확 |
| 계산 필드 | 단순 `SUM(필드)` 반복 | `SUM( CASE WHEN … )` | 조건부 집계 가능 |
| 클래스별 매출 | ❌ 주석 의도만 있음 | ✅ 실제 클래스별 합계 | 로직 정확성 개선 |
| 계산 위치 | ABAP 또는 단순 DB 집계 | DB에서 조건 포함 계산 | LOOP 제거, 성능 향상 |
| JOIN 표현 | SELECT 문 내부에 포함 | FROM 뒤에 JOIN 분리 | 가독성 개선 |
| LEFT JOIN 안정성 | WHERE에 B 조건 사용 | (권장) ON에 B 조건 | LEFT JOIN 의미 보존 |
| WHERE 절 | ABAP 변수 바로 사용 | ABAP 변수 앞에 `@` | 호스트 변수 명시 |
| INTO TABLE | `INTO TABLE CT_CARRIERS` | `INTO TABLE @CT_CARRIERS` | New Open SQL 문법 필수 |
| GROUP BY | 공백 구분 | 콤마 구분 | SQL 표준에 가까움 |
| ORDER BY | 어디든 사용 | 문장 끝에서 사용 | 문장 구조 정리 |

## ABAP Database Connectivity : ADBC 이론

1. ADBC란 무엇인가?
    1. ABAP에서 Native SQL을 안전하게 실행하기 위한 객체 기반 API
2. 구조 설명

```
ABAP Program
   ↓
ADBC (Native SQL API)
   ↓
DB Interface (DBI)
   ↓
Database-Specific Library (DBSL)
   ↓
Database
```

1. ADBC 전형적인 호출 순서 (Typical Call Sequence)
    
    1) 데이터베이스 연결 선택
    
    - `CL_SQL_CONNECTION` 클래스의 `GET_CONNECTION( )` 메서드 호출
    
    2) SQL Statement 객체 생성
    
    - `CL_SQL_STATEMENT` 클래스 인스턴스 생성
    
    3) SQL 문자열 생성
    
    - Native SQL 문장을 **문자열(String)** 형태로 구성
    
    4) Native SQL 실행
    
    - `CL_SQL_STATEMENT->EXECUTE_QUERY( )` 호출
    
    5) 결과 수신 대상 지정
    
    - 결과를 받을 내부 테이블 또는 변수 지정
    - `CL_SQL_RESULT_SET` 클래스의 `SET_PARAM( )` 또는 `SET_PARAM_TABLE( )` 사용
    
    6) 결과 조회
    
    - `CL_SQL_RESULT_SET->NEXT_PACKAGE( )` 호출
    
    7) 결과 닫기 및 리소스 해제
    
2. **ADBC 코드 예제 해설 (Typical Call Sequence 기준)**
    1. 필요한 데이터 선언 (Required data declarations)
        
        ```abap
        DATA:
          lo_sql     TYPE REF TO cl_sql_statement,
          lv_sql     TYPE string,
          lx_sql     TYPE REF TO cx_sql_exception,
          lo_result  TYPE REF TO cl_sql_result_set,
          lr_flight  TYPE REF TO data,
          lt_flight  TYPE STANDARD TABLE OF sflight.
        ```
        
        설명
        
        - `lo_sql`
            - SQL 실행을 담당하는 Statement 객체
        - `lv_sql`
            - Native SQL 문장을 담는 문자열 변수
        - `lx_sql`
            - Native SQL 실행 중 발생하는 예외 객체
        - `lo_result`
            - SQL 실행 결과(Result Set)를 담는 객체
        - `lt_flight`
            - 실제 조회 결과를 담을 내부 테이블
        - `lr_flight`
            - 내부 테이블을 Result Set과 연결하기 위한 참조 변수
    2. Statement 객체 생성 + DB 연결 (Create statement object and DB connection)
        
        ```abap
        CREATE OBJECT lo_sql
          EXPORTING
            con_ref = cl_sql_connection=>get_connection( 'HANADB' ).
        ```
        
        설명
        
        - `CL_SQL_STATEMENT` 객체 생성
        - `GET_CONNECTION( 'HANADB' )`
            - Secondary DB(HANA DB)에 대한 연결 획득
        - Primary DB만 쓸 경우 이 단계는 생략 가능
    3. Native SQL 문자열 작성 (Execute query, passing SQL query as string)
        
        ```abap
        CONCATENATE
          'SELECT * FROM SFLIGHT WHERE'
          ' MANDT = ' sy-mandt
          ' AND CARRID = ''LH'''
          INTO lv_sql.
        ```
        
        설명
        
        - Native SQL은 반드시 **문자열 형태**로 작성
        - `MANDT`를 직접 조건으로 명시
        - Open SQL과 달리 클라이언트 자동 처리 없음
    4. Native SQL 실행
        
        ```abap
        lo_result = lo_sql->execute_query( lv_sql ).
        ```
        
        설명
        
        - SQL 문자열을 DB로 전송하여 실행
        - 실행 결과는 `CL_SQL_RESULT_SET` 객체로 반환
    5. 결과 받을 내부 테이블 연결 (Assign internal table for result)
        
        ```abap
        GET REFERENCE OF lt_flight INTO lr_flight.
        lo_result->set_param_table( lr_flight ).
        ```
        
        설명
        
        - 내부 테이블 `lt_flight`의 참조를 생성
        - Result Set과 내부 테이블을 연결
    6. 결과 조회 (Retrieve result)
        
        ```abap
        lo_result->next_package( ).
        ```
        
        설명
        
        - Result Set에서 실제 데이터를 읽어옴
        - 내부적으로 DB → ABAP 데이터 이동
    7. Result Set 닫기 및 리소스 해제 (Close result)
        
        ```abap
        lo_result->close( ).
        ```
        
        설명
        
        - DB 커서 및 리소스 해제
        - 반드시 호출해야 함
    8. 예외 처리 (Handle exceptions)
        
        ```abap
        CATCH cx_sql_exception INTO lx_sql.
        ```
        
        설명
        
        - Native SQL 문법 오류
        - DB 실행 오류
        - 권한/스키마 오류 등 처리

## SAP HANA Full Text Search 이론

  1.  SAP HANA Full Text Search 개념

→ 문자열 검색에서 “정확한 일치”가 아닌 “의미적 유사성”을 기준으로 검색

1. 특징
- Error-tolerant search (퍼지 검색)
- Linguistic search (언어 기반 검색)
- 의미적으로 유사한 단어 검색
    
    유사도를 **0 ~ 1 사이의 SCORE 값**으로 환산 , 일정 임계값(Threshold) 이상만 결과로 반환
    
    → 기본 임계값 : 0.8 (80% 이상 유사도)
    
- 대소문자, 철자 차이 자동 보정

1. Fuzzy Search vs LIKE vs Equals 비교

| 검색 방식 | 특징 | 한계 |
| --- | --- | --- |
| `=` | 정확 일치 | 문자열 검색에 거의 사용 불가 |

| Full Text Search | 유사도 기반 검색 | 설정 필요 |

2. 특징
    - 내부 테이블을 거의 사용하지 않음
    - **DB에서 필요한 데이터만 가져옴**
    - 필터, 정렬, 집계, 그룹핑을 **DB에서 수행**
3. Classical ALV vs ALV IDA 구조 비교
    
    Classical ALV 구조
    
    - DB → ABAP Internal Table → ALV → UI
    - 모든 데이터가 ABAP 서버를 거침
    
    ALV with IDA 구조
    
    - DB(CDS View) → ALV IDA → UI
    - ABAP 서버는 **조건 전달자 역할만 수행**
4. Classical ALV vs ALV IDA 기능 비교 표

## **ABAP CDS (S4D430)** 이론

1. ABAP CDS란 무엇인가?
    1. DB 위에 “똑똑한 뷰(View)”를 만들어서 데이터 처리 + 의미(설명)까지 같이 정의하는 기술
    2. DB에 있는 데이터를, 화면·분석·프로그램이 바로 쓰기 좋게 미리 정리해 둔 설계도
    
2. ABAP CDS가 하는 3가지 핵심 역할
    
    1) 데이터 모델 정의 (Data Models on the Database)
    
    - DB View 정의
    - JOIN, 계산 컬럼, 집계
    - SQL 로직을 **DB에서 실행**
    
    2) 메타데이터 제공 (Metadata for UI and Consumers)
    
    - 컬럼 이름의 의미
    - 화면에 어떻게 보여줄지
    - 검색 가능 여부
    - 분석용 여부
    
    3) ABAP RESTful Programming Model 지원
    
    - Fiori, OData, RAP과 바로 연결
    - 비즈니스 객체의 동작 정의
    
3. CDS View 종류
    
    1) CDS DDIC-based View (구형, 거의 안 씀)
    
    - `DEFINE VIEW`
    - Dictionary View를 내부적으로 생성
    - **현재는 obsolete (구식)**
    
    2) CDS View Entity (현재 표준)
    
    - `DEFINE VIEW ENTITY`
    - Dictionary 객체를 따로 만들지 않음, **Dictionary에 존재하지 않음**
    - DB 객체를 직접 정의
    
    **→ 요즘 배우는 CDS는 전부 이거**
    
    4. CDS와 ABAP Dictionary의 관계 (기초 구조 이해)
    
    4.1 Database Table / Data Element / Domain
    
    - CDS는 **테이블 위에서 동작**
    - 필드 의미는 **Data Element**에서 상속
    - Domain은 **기술적 타입(CHAR, NUMC, CURR 등)** 정의
    
    정리 포인트:
    
    - CDS는 **테이블을 대체하지 않는다**
    - CDS = 테이블을 “의미 있게 포장”한 레이어
    
    5. CDS View Entity 기본 구조
    
    5.1 기본 문법
    
    ```abap
    define view entity ZC_Example
      as select from ztable
    {
      key field1,
          field2
    }
    ```
    
    핵심 포인트:
    
    - `define view entity` → 요즘 표준
    - `key` → 논리 키 (ALV, Fiori에서 중요)
    - SQL 기반이지만 **Open SQL 아님**
    
    5.2 CDS View Entity 특징
    
    - Dictionary View 자동 생성 ❌
    - DB View 직접 생성 ⭕
    - 성능 최적화 (HANA 친화적)
    
    6. CDS Annotation (가장 중요한 파트)
    
    6.1 Annotation이란?
    
    - `@`로 시작하는 메타데이터
    - **데이터의 의미 + 사용 목적을 정의**
    
    6.2 Annotation 분류 (시험·실무 핵심)
    
    1) UI Annotation
    
    ```abap
    @EndUserText.label
    @UI.lineItem
    @UI.identification
    ```
    
    - Fiori 화면 자동 생성에 사용
    
    2) Search Annotation
    
    ```abap
    @Search.searchable: true
    @Search.defaultSearchElement: true
    ```
    
    - HANA Full Text Search
    - Fuzzy Search 가능
    
    3) Analytics Annotation
    
    ```abap
    @Analytics.dataCategory: #FACT
    @Analytics.measure
    ```
    
    - 분석용 CDS (리포트, KPI)
    
    4) Authorization Annotation
    
    ```abap
    @AccessControl.authorizationCheck: #CHECK
    ```
    
    - DCL(Role)과 연결
    
    7. CDS와 ALV with IDA
    
    7.1 관계
    
    - ALV IDA는 **CDS를 직접 데이터 소스로 사용**
    - 내부 테이블 사용 ❌
    - 페이징·정렬·필터링을 **DB에서 처리**
    
    7.2 CDS 기반 ALV IDA 장점
    
    - CDS `key` 정의 활용
    - Annotation 기반 UI 자동 구성
    - 대량 데이터 고성능 처리
    
    8. CDS와 HANA Full Text / Fuzzy Search
    
    8.1 기존 검색의 한계
    
    - `=` : 정확 일치만 가능
    - `LIKE` : 부분 일치, 정확도 낮음
    
    8.2 CDS + Search Annotation
    
    - 오타 허용 검색
    - 유사 단어 검색
    - score 기반 결과 반환
    
    9. CDS와 RAP (ABAP RESTful Programming Model)
    
    9.1 역할 분담
    
    - CDS Data Definition → 데이터 구조
    - CDS Behavior Definition → CUD 동작 정의
    - ABAP Code → 비즈니스 로직 구현

```sql
*        INTO TABLE GT_SCARR

*  LOOP AT GT_SCARR INTO GS_SCARR.
```

```sql
* New Open SQL
  SELECT FROM SCARR AS A
    INNER JOIN SFLIGHT AS F
      ON F~CARRID = A~CARRID
    FIELDS
      A~CARRID   AS CARRID,
      A~CARRNAME AS CARRNAME,
      A~CURRCODE AS CURRCODE,

      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '01' THEN F~PAYMENTSUM ELSE 0 END ) AS JANSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '02' THEN F~PAYMENTSUM ELSE 0 END ) AS FEBSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '03' THEN F~PAYMENTSUM ELSE 0 END ) AS MARSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '04' THEN F~PAYMENTSUM ELSE 0 END ) AS APRSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '05' THEN F~PAYMENTSUM ELSE 0 END ) AS MAYSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '06' THEN F~PAYMENTSUM ELSE 0 END ) AS JUNSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '07' THEN F~PAYMENTSUM ELSE 0 END ) AS JULSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '08' THEN F~PAYMENTSUM ELSE 0 END ) AS AUGSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '09' THEN F~PAYMENTSUM ELSE 0 END ) AS SEPSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '10' THEN F~PAYMENTSUM ELSE 0 END ) AS OCTSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '11' THEN F~PAYMENTSUM ELSE 0 END ) AS NOVSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '12' THEN F~PAYMENTSUM ELSE 0 END ) AS DECSUM

    WHERE A~CARRID IN @SO_CARR
      AND F~FLDATE BETWEEN @LV_FROM AND @LV_TO
    GROUP BY A~CARRID, A~CARRNAME, A~CURRCODE
    ORDER BY A~CARRID
    INTO TABLE @GT_DATA.
    
ENDCASE.
```
