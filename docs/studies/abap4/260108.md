# 1. ZABAP_56_G01 – New Open SQL 실습

## 1.1 프로그램 개요

항공편(`SFLIGHT`) 데이터를 조회하는 예제이다.

- 선택 조건: 항공사, 연결편
- 계산 필드:
  - 탑승률(`PERCENTAGE`)
  - 잔여 좌석(`SEATSFREE`)
- 핵심 포인트:
  - 기존에는 ABAP `LOOP`에서 계산
  - New Open SQL에서는 **DB 레벨에서 계산**
  - 결과적으로 **LOOP 제거**, **성능 향상**, **가독성 개선**

---

## 1.2 전체 코드

```abap
*&---------------------------------------------------------------------*
*& Report ZABAP_56_G01
*&---------------------------------------------------------------------*
*& 항공편(SFLIGHT) 데이터 조회
*& - 선택 조건: 항공사, 연결편
*& - 계산 필드: 탑승률, 잔여 좌석
*&---------------------------------------------------------------------*
REPORT ZABAP_56_G01.

*---------------------------------------------------------------------*
* Data Declaration
*---------------------------------------------------------------------*
DATA: GT_FLIGHT TYPE BC400_T_FLIGHTS,
      GS_FLIGHT LIKE LINE OF GT_FLIGHT.

*---------------------------------------------------------------------*
* Selection Screen
*---------------------------------------------------------------------*
SELECT-OPTIONS:
  SO_CAR FOR GS_FLIGHT-CARRID,  "항공사
  SO_CON FOR GS_FLIGHT-CONNID.  "연결편

*---------------------------------------------------------------------*
* Old Open SQL (주석 처리됨)
*---------------------------------------------------------------------*
* 1. DB에서 데이터 조회
* 2. LOOP를 통해 ABAP에서 탑승률 계산
* 3. MODIFY로 내부 테이블 갱신
* → 처리 단계가 많고 성능 및 가독성 저하
*---------------------------------------------------------------------*
*SELECT CARRID CONNID FLDATE SEATSMAX SEATSOCC
*  FROM SFLIGHT
*  INTO CORRESPONDING FIELDS OF TABLE GT_FLIGHT
*  WHERE CARRID IN SO_CAR
*    AND CONNID IN SO_CON.
*
*LOOP AT GT_FLIGHT INTO GS_FLIGHT.
*  GS_FLIGHT-PERCENTAGE =
*    GS_FLIGHT-SEATSOCC * 100 / GS_FLIGHT-SEATSMAX.
*
*  MODIFY GT_FLIGHT FROM GS_FLIGHT.
*  CLEAR GS_FLIGHT.
*ENDLOOP.

*---------------------------------------------------------------------*
* New Open SQL (계산 필드 포함 – 주석 처리됨)
*---------------------------------------------------------------------*
* - DB 레벨에서 계산 수행
* - LOOP 불필요
* - DIVISION 함수로 0 나누기 방지 및 소수점 처리
*---------------------------------------------------------------------*
*SELECT CARRID, CONNID, FLDATE, SEATSMAX, SEATSOCC,
*       DIVISION( SEATSOCC * 100, SEATSMAX, 2 ) AS PERCENTAGE "계산을 DB 레벨에서 수행"
*  FROM SFLIGHT
*  WHERE CARRID IN @SO_CAR
*    AND CONNID IN @SO_CON
*  INTO CORRESPONDING FIELDS OF TABLE @GT_FLIGHT.

*---------------------------------------------------------------------*
* New Open SQL + 잔여 좌석 수 포함 (최종 사용)
*---------------------------------------------------------------------*
* PERCENTAGE : 좌석 점유율 (%)
* SEATSFREE  : 최대 좌석 - 예약 좌석
* @DATA      : Inline Declaration
*---------------------------------------------------------------------*
SELECT
    CARRID,                                "항공사 코드
    CONNID,                                "연결편 번호
    FLDATE,                                "비행 날짜
    SEATSMAX,                              "최대 좌석 수
    SEATSOCC,                              "예약된 좌석 수
    DIVISION( SEATSOCC * 100, SEATSMAX, 2 ) AS PERCENTAGE, "탑승률(%)
    SEATSMAX - SEATSOCC AS SEATSFREE       "남은 좌석 수 추가
  FROM SFLIGHT
  WHERE CARRID IN @SO_CAR                  "항공사 조건
    AND CONNID IN @SO_CON                  "연결편 조건
  INTO TABLE @DATA(GT_DATA).               "내부 테이블 선언 + 데이터 수신 동시 수행

*---------------------------------------------------------------------*
* Output
*---------------------------------------------------------------------*
CL_DEMO_OUTPUT=>DISPLAY( GT_DATA ).
```

---

## 1.3 처리 흐름 비교

### 기존 방식 (Old Open SQL)

```text
DB에서 조회
   ↓
GT_FLIGHT 적재
   ↓
LOOP
   ↓
탑승률 계산
   ↓
MODIFY
   ↓
출력
```

### New Open SQL 방식

```text
DB에서 조회 + 계산 수행
   ↓
GT_DATA에 결과 적재
   ↓
출력
```

---

## 1.4 핵심 요약

- `DIVISION( SEATSOCC * 100, SEATSMAX, 2 )`
  - DB에서 탑승률 계산
  - 0 나누기 방지 및 소수점 처리
- `SEATSMAX - SEATSOCC AS SEATSFREE`
  - 남은 좌석 수를 DB에서 계산
- `@DATA(GT_DATA)`
  - 인라인 선언 + 결과 수신을 동시에 처리

### 한 줄 정리
> New Open SQL은 계산을 DB에서 처리하여 LOOP를 제거하고 성능과 가독성을 동시에 개선한다.

# 2. ZHA400_G01_NEW_OSQL – New Open SQL 실습

## 2.1 코드 비교

### Old Open SQL

```abap
SELECT A~CARRID A~CARRNAME A~CURRCODE
    SUM( B~LOCCURAM ) AS REVENUE_FIRST    "Sum only bookings with class = 'F'
    SUM( B~LOCCURAM ) AS REVENUE_BUSINESS "Sum only bookings with class = 'C'
    SUM( B~LOCCURAM ) AS REVENUE_ECONOMY  "Sum only bookings with class = 'Y'
  FROM SCARR AS A LEFT OUTER JOIN SBOOK AS B
          ON A~CARRID = B~CARRID
  INTO TABLE CT_CARRIERS
  WHERE B~CANCELLED <> 'X' OR B~CANCELLED IS NULL
GROUP BY A~CARRID A~CARRNAME A~CURRCODE
ORDER BY A~CARRID.
```

### New Open SQL

```abap
SELECT FROM SCARR AS A
LEFT OUTER JOIN SBOOK AS B ON A~CARRID = B~CARRID
FIELDS
  A~CARRID,
  A~CARRNAME,
  A~CURRCODE,
  SUM( CASE B~CLASS WHEN 'F' THEN B~LOCCURAM ELSE 0 END ) AS revenue_first,
  SUM( CASE B~CLASS WHEN 'C' THEN B~LOCCURAM ELSE 0 END ) AS revenue_business,
  SUM( CASE B~CLASS WHEN 'Y' THEN B~LOCCURAM ELSE 0 END ) AS revenue_economy
WHERE B~CANCELLED <> 'X'
  OR B~CANCELLED IS NULL
GROUP BY A~CARRID, A~CARRNAME, A~CURRCODE
ORDER BY A~CARRID
INTO TABLE @CT_CARRIERS.
```

---

## 2.2 처리 구조 시각화

```text
SCARR (항공사 마스터)
   LEFT OUTER JOIN
SBOOK (예약 정보)
   ↓
클래스별 매출 집계
   ├─ F → revenue_first
   ├─ C → revenue_business
   └─ Y → revenue_economy
   ↓
CT_CARRIERS
```

---

## 2.3 Old Open SQL vs New Open SQL 변화 비교표

| 구분 | Old Open SQL | New Open SQL | 변화 요약 |
|---|---|---|---|
| SELECT 구조 | `SELECT 필드 FROM 테이블` | `SELECT FROM 테이블 FIELDS 필드` | FROM을 먼저 쓰고 FIELDS로 필드 명시 |
| 필드 나열 방식 | SELECT 절에 바로 나열 | `FIELDS` 절에서 명시 | 가독성 및 구조 명확 |
| 계산 필드 | 단순 `SUM(필드)` 반복 | `SUM( CASE WHEN … )` | 조건부 집계 가능 |
| 클래스별 매출 | ❌ 주석 의도만 있음 | ✅ 실제 클래스별 합계 | 로직 정확성 개선 |
| 계산 위치 | ABAP 또는 단순 DB 집계 | DB에서 조건 포함 계산 | LOOP 제거, 성능 향상 |
| JOIN 표현 | SELECT 문 내부에 포함 | FROM 뒤에 JOIN 분리 | 가독성 개선 |
| LEFT JOIN 안정성 | WHERE에 B 조건 사용 | (권장) ON에 B 조건 | LEFT JOIN 의미 보존 |
| WHERE 절 | ABAP 변수 바로 사용 | ABAP 변수 앞에 `@` | 호스트 변수 명시 |
| INTO TABLE | `INTO TABLE CT_CARRIERS` | `INTO TABLE @CT_CARRIERS` | New Open SQL 문법 필수 |
| GROUP BY | 공백 구분 | 콤마 구분 | SQL 표준에 가까움 |
| ORDER BY | 어디든 사용 | 문장 끝에서 사용 | 문장 구조 정리 |

---

## 2.4 핵심 해석

이 예제의 핵심은 `CASE + SUM`이다.

```abap
SUM( CASE B~CLASS WHEN 'F' THEN B~LOCCURAM ELSE 0 END ) AS revenue_first
```

즉,

- `CLASS = 'F'` 인 경우만 `LOCCURAM` 누적
- 나머지는 `0`
- 결과적으로 First / Business / Economy 매출을 **한 번의 SELECT에서 모두 집계**

# 3. SAP HANA SQL vs Open SQL 문법 차이 이론

## 3.1 SAP HANA Native SQL / Classic Open SQL / Enhanced Open SQL 중요 문법 차이

| 구분 | SAP HANA Native SQL | Classic Open SQL | Enhanced Open SQL |
|---|---|---|---|
| **필드 목록 구분자** | 콤마( `,` ) | 공백 | 콤마( `,` ) |
| **필드 목록 위치** | FROM 절 앞 | FROM 절 앞 | FROM 절 앞 또는 뒤 (`FIELDS` 키워드 사용) |
| **테이블-필드 구분자** | 점( `.` ) | 물결(`~`) | 물결(`~`) |

---

## 3.2 SAP HANA 특화 기능 차이 (Code-to-Data 관점)

| 구분 | SAP HANA Native SQL | Classic Open SQL | Enhanced Open SQL |
|---|---|---|---|
| **스키마 처리** | 기본 스키마 암시적 사용 또는 테이블 앞에 스키마 명시 | 기본 스키마만 사용 | 기본 스키마만 사용 |
| **클라이언트 처리** | 암시적 클라이언트 없음, WHERE/JOIN에 명시 필요 | DB 인터페이스에 의한 암시적 처리, `CLIENT SPECIFIED` 가능 | DB 인터페이스에 의한 암시적 처리, `USING CLIENT` 가능 |
| **조회 건수 제한** | `TOP n` | `UP TO n ROWS` | `UP TO n ROWS` |
| **CDS View 접근** | DB에 존재하는 SQL View만 접근 가능 | SQL View만 접근 가능 | CDS View 직접 접근 가능 |
| **CORRESPONDING FIELDS** | 지원 안 함 | 지원 | 지원 |
| **FOR ALL ENTRIES** | 지원 안 함 | 지원 | 지원 |

---

## 3.3 SELECT 문 기준 추가 차이

| 항목 | Open SQL | SAP HANA Native SQL |
|---|---|---|
| **필드 나열 방식** | `SELECT f1 f2 f3` | `SELECT f1, f2, f3` |
| **ORDER BY 구분자** | 공백 | 콤마 |
| **MANDT 처리** | 자동 추가 | 직접 조건 필요 |
| **WHERE 절 역할** | 클라이언트 고려 불필요 | 클라이언트 조건 필수 |

---

## 3.4 JOIN 문 기준 추가 차이

| 항목 | Open SQL | SAP HANA Native SQL |
|---|---|---|
| **JOIN 필드 구분자** | `a~field` | `a.field` |
| **클라이언트 조인** | 자동 | `a.mandt = b.mandt` 직접 작성 |
| **JOIN 안정성** | ABAP 레벨에서 보호 | 개발자 책임 |
| **데이터 누락 위험** | 낮음 | 높음 (`MANDT` 누락 시) |

---

## 3.5 Enhanced Open SQL이 생긴 이유

| 문제점 | Classic Open SQL | Enhanced Open SQL |
|---|---|---|
| HANA 친화 문법 | ❌ | ⭕ |
| 가독성 | 낮음 | 높음 |
| DB 계산 활용 | 제한적 | 적극 활용 |
| CDS View | ❌ | ⭕ |
| Code Push-down | ❌ | ⭕ |

---

## 3.6 핵심 요약

```text
Classic Open SQL
  → 단순 조회 중심

Enhanced Open SQL
  → HANA 친화 문법 + DB 계산 활용 + 가독성 개선

Native SQL
  → DB 고유 기능 사용 가능하지만 DB 종속성 높음
```

# 4. ABAP Database Connectivity : ADBC 이론

## 4.1 ADBC란 무엇인가?

ADBC는  
**ABAP에서 Native SQL을 안전하게 실행하기 위한 객체 기반 API** 이다.

즉,

- Open SQL로 부족한 기능을 쓸 때
- DB 고유 SQL을 직접 실행해야 할 때
- Native SQL을 객체 지향 방식으로 다루기 위한 수단

---

## 4.2 구조 설명

```text
ABAP Program
   ↓
ADBC (Native SQL API)
   ↓
DB Interface (DBI)
   ↓
Database-Specific Library (DBSL)
   ↓
Database
```

---

## 4.3 ADBC 전형적인 호출 순서 (Typical Call Sequence)

### 1) 데이터베이스 연결 선택
- `CL_SQL_CONNECTION` 클래스의 `GET_CONNECTION( )` 메서드 호출

### 2) SQL Statement 객체 생성
- `CL_SQL_STATEMENT` 클래스 인스턴스 생성

### 3) SQL 문자열 생성
- Native SQL 문장을 **문자열(String)** 형태로 구성

### 4) Native SQL 실행
- `CL_SQL_STATEMENT->EXECUTE_QUERY( )` 호출

### 5) 결과 수신 대상 지정
- 결과를 받을 내부 테이블 또는 변수 지정
- `CL_SQL_RESULT_SET` 클래스의 `SET_PARAM( )` 또는 `SET_PARAM_TABLE( )` 사용

### 6) 결과 조회
- `CL_SQL_RESULT_SET->NEXT_PACKAGE( )` 호출

### 7) 결과 닫기 및 리소스 해제

---

## 4.4 호출 흐름 시각화

```text
DB 연결 선택
   ↓
Statement 객체 생성
   ↓
SQL 문자열 작성
   ↓
EXECUTE_QUERY
   ↓
결과 받을 내부 테이블 연결
   ↓
NEXT_PACKAGE
   ↓
CLOSE
```

---

## 4.5 ADBC 코드 예제 해설 (Typical Call Sequence 기준)

### 1) 필요한 데이터 선언

```abap
DATA:
  lo_sql     TYPE REF TO cl_sql_statement,
  lv_sql     TYPE string,
  lx_sql     TYPE REF TO cx_sql_exception,
  lo_result  TYPE REF TO cl_sql_result_set,
  lr_flight  TYPE REF TO data,
  lt_flight  TYPE STANDARD TABLE OF sflight.
```

#### 설명

- `lo_sql`
  - SQL 실행을 담당하는 Statement 객체
- `lv_sql`
  - Native SQL 문장을 담는 문자열 변수
- `lx_sql`
  - Native SQL 실행 중 발생하는 예외 객체
- `lo_result`
  - SQL 실행 결과(Result Set)를 담는 객체
- `lt_flight`
  - 실제 조회 결과를 담을 내부 테이블
- `lr_flight`
  - 내부 테이블을 Result Set과 연결하기 위한 참조 변수

---

### 2) Statement 객체 생성 + DB 연결

```abap
CREATE OBJECT lo_sql
  EXPORTING
    con_ref = cl_sql_connection=>get_connection( 'HANADB' ).
```

#### 설명

- `CL_SQL_STATEMENT` 객체 생성
- `GET_CONNECTION( 'HANADB' )`
  - Secondary DB(HANA DB)에 대한 연결 획득
- Primary DB만 사용할 경우 이 단계는 생략 가능

---

### 3) Native SQL 문자열 작성

```abap
CONCATENATE
  'SELECT * FROM SFLIGHT WHERE'
  ' MANDT = ' sy-mandt
  ' AND CARRID = ''LH'''
  INTO lv_sql.
```

#### 설명

- Native SQL은 반드시 **문자열 형태**로 작성
- `MANDT`를 직접 조건으로 명시해야 함
- Open SQL과 달리 클라이언트 자동 처리 없음

---

### 4) Native SQL 실행

```abap
lo_result = lo_sql->execute_query( lv_sql ).
```

#### 설명

- SQL 문자열을 DB로 전송하여 실행
- 실행 결과는 `CL_SQL_RESULT_SET` 객체로 반환

---

### 5) 결과 받을 내부 테이블 연결

```abap
GET REFERENCE OF lt_flight INTO lr_flight.
lo_result->set_param_table( lr_flight ).
```

#### 설명

- 내부 테이블 `lt_flight`의 참조를 생성
- Result Set과 내부 테이블을 연결

---

### 6) 결과 조회

```abap
lo_result->next_package( ).
```

#### 설명

- Result Set에서 실제 데이터를 읽어옴
- 내부적으로 DB → ABAP 데이터 이동

---

### 7) Result Set 닫기 및 리소스 해제

```abap
lo_result->close( ).
```

#### 설명

- DB 커서 및 리소스 해제
- 반드시 호출해야 함

---

### 8) 예외 처리

```abap
CATCH cx_sql_exception INTO lx_sql.
```

#### 설명

- Native SQL 문법 오류
- DB 실행 오류
- 권한/스키마 오류 등 처리

# 5. ZABAP_57_G01 - ADBC 실습

## 5.1 실습 목적

ADBC를 사용해 Native SQL로 `SPFLI`를 조회하는 실습이다.

---

## 5.2 필요한 변수 선언

```abap
DATA : gv_sql TYPE string,
       gv_mesg TYPE string,
       gt_flight  TYPE  TABLE OF sflight,
       go_sql     TYPE REF TO cl_sql_statement,
       go_result  TYPE REF TO cl_sql_result_set,
       gx_root TYPE REF TO cx_roof.
```

### 실습 메모
- 원문 그대로 적으면 `cx_roof`로 되어 있는데, 보통 예외 루트 클래스는 `cx_root`이다.
- 강의/실습 오타 가능성 체크 필요

---

## 5.3 실습 절차 시각화

```text
변수 선언
   ↓
TRY-CATCH 작성
   ↓
SQL Statement 객체 생성
   ↓
Native SQL 문자열 작성
   ↓
EXECUTE_QUERY 실행
   ↓
SET_PARAM_TABLE로 결과 연결
   ↓
NEXT_PACKAGE로 데이터 수신
   ↓
CLOSE로 리소스 해제
   ↓
출력
```

---

## 5.4 Try-Catch 구문 코드

```abap
TRY.

*   SQL Statement 오브젝트 생성시 연결할 데이터베이스 설정.
    CREATE OBJECT GO_SQL
      EXPORTING
        CON_REF = CL_SQL_CONNECTION=>GET_CONNECTION( 'DEFAULT' ).

*   Query 작성, 변수에 할당.
    CONCATENATE `SELECT * FROM SPFLI WHERE MANDT = ` SY-MANDT
                ` AND CARRID = 'LH'` INTO GV_SQL.

*   작성한 Query 실행.
    GO_RESULT = GO_SQL->EXECUTE_QUERY( GV_SQL ).

*   읽어올 데이터를 할당할 변수 설정.
    GET REFERENCE OF GT_SPFLI INTO GR_DATA.
    GO_RESULT->SET_PARAM_TABLE( ITAB_REF = GR_DATA ).

*   DB로 부터 데이터 가져옴.
    GO_RESULT->NEXT_PACKAGE( ).

*   리소스 릴리즈.
    GO_RESULT->CLOSE( ).

    CL_DEMO_OUTPUT=>DISPLAY( GT_SPFLI ).

  CATCH CX_SQL_EXCEPTION INTO GX_ROOT.
    GV_MESG = GX_ROOT->GET_TEXT( ).
    MESSAGE GV_MESG TYPE 'I'.
ENDTRY.
```

## 5.5 오류 포인트: 공백 누락

### 문제 코드

```abap
CONCATENATE `SELECT * FROM SPFLI WHERE MANDT = ` SY-MANDT
                ` AND CARRID = 'LH'` INTO GV_SQL.
```

### 왜 문제인가?

`CONCATENATE`는 **자동으로 공백을 넣어주지 않는다.**

그래서 실제 문자열이 아래처럼 될 수 있다.

```sql
SELECT*FROM SPFLIWHERE MANDT=800AND CARRID='LH'
```

### 문제점

- `SPFLIWHERE`
- `800AND`

이 둘은 SQL 문법상 잘못된 형태다.

### 정리

- `  AND → O`
- `` `AND → X``

즉, SQL 문자열 조합 시 공백까지 포함해서 작성해야 한다.

# 6. SAP HANA Full Text Search 이론

## 6.1 SAP HANA Full Text Search 개념

문자열 검색에서  
**정확한 일치(Exact Match)** 가 아니라  
**의미적 유사성(Semantic Similarity)** 을 기준으로 검색하는 방식이다.

---

## 6.2 특징

- Error-tolerant search (퍼지 검색)
- Linguistic search (언어 기반 검색)
- 의미적으로 유사한 단어 검색
- 대소문자, 철자 차이 자동 보정

또한 유사도를 **0 ~ 1 사이의 SCORE 값**으로 환산하여  
일정 임계값(Threshold) 이상만 결과로 반환한다.

- 기본 임계값: `0.8`
- 즉, **80% 이상 유사도**

---

## 6.3 Fuzzy Search vs LIKE vs Equals 비교

| 검색 방식 | 특징 | 한계 |
|---|---|---|
| `=` | 정확 일치 | 문자열 검색에 거의 사용 불가 |
| `LIKE` | 부분 일치 | 정확도 제어 불가, 오타 취약 |
| Full Text Search | 유사도 기반 검색 | 설정 필요 |

---

## 6.4 Full-Text Index란 무엇인가?

SAP HANA에서 **Fuzzy Search를 성능 좋게 사용하려면 반드시 Full-Text Index가 필요**하다.

### Full-Text Index 개념

- 테이블에 **논리적으로 연결된 숨겨진 컬럼**
- SAP HANA가 검색을 빠르게 수행하기 위해 사용
- 일반 `SELECT`에서는 보이지 않음
- 검색용 내부 구조라고 이해하면 됨

---

## 6.5 동작 시각화

```text
사용자 입력
   ↓
Search Help / 돋보기 클릭
   ↓
Type-Ahead (자동 제안)
   ↓
Full Text Search 수행
   ↓
SCORE 계산
   ↓
Threshold 이상 결과만 반환
```

---

## 6.6 화면/설정 포인트 정리

### 1) 사용자가 입력 필드 옆 돋보기(Search Help) 아이콘 클릭
- Type-Ahead(자동 제안 검색)
- Full Text Search 동작

### 2) Autosuggest in Input Fields
- 입력 필드에서 **자동 제안(Type-Ahead)** 기능 활성화

### 3) Multi-column full text search (database-specific)
- SAP HANA Full Text Search 사용 활성화
- 여러 컬럼을 대상으로 Full Text / Fuzzy 검색 수행

### 4) Accuracy Value for Error-Tolerant Full Text Search
- Fuzzy Search의 유사도 임계값(Threshold)
- 예: `0.8`

# 7. HANA ALV : ALV IDA 이론

## 7.1 ALV with IDA란 무엇인가?

---

## 7.2 특징

- 내부 테이블을 거의 사용하지 않음

DB (CDS View)

ALV IDA

UI

# 8. ABAP CDS (S4D430) 이론

## 8.1 ABAP CDS란 무엇인가?

ABAP CDS는

1. DB 위에 “똑똑한 뷰(View)”를 만들어서  
   **데이터 처리 + 의미(설명)** 까지 같이 정의하는 기술

2. DB에 있는 데이터를  
   화면·분석·프로그램이 바로 쓰기 좋게  
   미리 정리해 둔 설계도

---

## 8.2 ABAP CDS가 하는 3가지 핵심 역할

### 1) 데이터 모델 정의 (Data Models on the Database)

- DB View 정의
- JOIN
- 계산 컬럼
- 집계
- SQL 로직을 **DB에서 실행**

### 2) 메타데이터 제공 (Metadata for UI and Consumers)

- 컬럼 이름의 의미
- 화면에 어떻게 보여줄지
- 검색 가능 여부
- 분석용 여부

### 3) ABAP RESTful Programming Model 지원

- Fiori, OData, RAP과 바로 연결
- 비즈니스 객체의 동작 정의

---

## 8.3 CDS View 종류

### 1) CDS DDIC-based View (구형, 거의 안 씀)

- `DEFINE VIEW`
- Dictionary View를 내부적으로 생성
- 현재는 **obsolete (구식)**

### 2) CDS View Entity (현재 표준)

- `DEFINE VIEW ENTITY`
- Dictionary 객체를 따로 만들지 않음
- **Dictionary에 존재하지 않음**
- DB 객체를 직접 정의

> 요즘 배우는 CDS는 거의 전부 이 방식

---

## 8.4 CDS와 ABAP Dictionary의 관계

### 8.4.1 Database Table / Data Element / Domain

- CDS는 **테이블 위에서 동작**
- 필드 의미는 **Data Element**에서 상속
- Domain은 **기술적 타입(CHAR, NUMC, CURR 등)** 정의

### 정리 포인트

- CDS는 **테이블을 대체하지 않는다**
- CDS = 테이블을 “의미 있게 포장”한 레이어

---

## 8.5 CDS View Entity 기본 구조

### 8.5.1 기본 문법

```abap
define view entity ZC_Example
  as select from ztable
{
  key field1,
      field2
}
```

### 핵심 포인트

- `define view entity` → 현재 표준
- `key` → 논리 키 (ALV, Fiori에서 중요)
- SQL 기반이지만 **Open SQL은 아님**

---

## 8.5.2 CDS View Entity 특징

- Dictionary View 자동 생성 ❌
- DB View 직접 생성 ⭕
- 성능 최적화 (HANA 친화적)

---

## 8.6 CDS Annotation (가장 중요한 파트)

### 8.6.1 Annotation이란?

- `@`로 시작하는 메타데이터
- **데이터의 의미 + 사용 목적을 정의**

---

## 8.6.2 Annotation 분류

### 1) UI Annotation

```abap
@EndUserText.label
@UI.lineItem
@UI.identification
```

- Fiori 화면 자동 생성에 사용

### 2) Search Annotation

```abap
@Search.searchable: true
@Search.defaultSearchElement: true
```

- HANA Full Text Search
- Fuzzy Search 가능

### 3) Analytics Annotation

```abap
@Analytics.dataCategory: #FACT
@Analytics.measure
```

- 분석용 CDS (리포트, KPI)

### 4) Authorization Annotation

```abap
@AccessControl.authorizationCheck: #CHECK
```

- DCL(Role)과 연결

---

## 8.7 CDS와 ALV with IDA

### 8.7.1 관계

- ALV IDA는 **CDS를 직접 데이터 소스로 사용**
- 내부 테이블 사용 ❌
- 페이징·정렬·필터링을 **DB에서 처리**

### 8.7.2 CDS 기반 ALV IDA 장점

- CDS `key` 정의 활용
- Annotation 기반 UI 자동 구성
- 대량 데이터 고성능 처리

---

## 8.8 CDS와 HANA Full Text / Fuzzy Search

### 8.8.1 기존 검색의 한계

- `=` : 정확 일치만 가능
- `LIKE` : 부분 일치, 정확도 낮음

### 8.8.2 CDS + Search Annotation

- 오타 허용 검색
- 유사 단어 검색
- score 기반 결과 반환

---

## 8.9 CDS와 RAP (ABAP RESTful Programming Model)

### 8.9.1 역할 분담

- CDS Data Definition → 데이터 구조
- CDS Behavior Definition → CUD 동작 정의
- ABAP Code → 비즈니스 로직 구현

# 11. ZQUIZ_18_G01 - Classic Open SQL → New Open SQL 버전

## 11.1 기존 코드 개념

기존 방식은 다음 흐름이었다.

```text
SFLIGHT 조회
   ↓
GT_SFLIGHT 저장
   ↓
SCARR 조회
   ↓
GT_SCARR 저장
   ↓
항공사 LOOP
   ↓
해당 항공사 Flight LOOP
   ↓
월 추출
   ↓
CASE로 월별 합계 누적
   ↓
GT_DATA APPEND
```

즉,  
**조회 2번 + 내부테이블 LOOP 2중 반복 + CASE 누적 계산** 구조였다.

---

## 11.2 기존 코드

```abap
** 항공편 데이터 조회 (해당 년도 + 항공사)
*      SELECT *
*        INTO TABLE GT_SFLIGHT
*        FROM SFLIGHT
*        WHERE CARRID IN SO_CARR
*          AND FLDATE BETWEEN LV_FROM AND LV_TO.
*
*      IF SY-SUBRC <> 0.
*        MESSAGE '해당 조건에 맞는 Flight 데이터가 없습니다.' TYPE 'I'.
*        EXIT.
*      ENDIF.
*
*      SORT GT_SFLIGHT BY CARRID FLDATE.
*
** 항공사 정보 조회
*      SELECT *
*        INTO TABLE GT_SCARR
*        FROM SCARR
*        WHERE CARRID IN SO_CARR.
*
*      SORT GT_SCARR BY CARRID.
*
*  ENDCASE.

** 월별 합계 집계.
*  LOOP AT GT_SCARR INTO GS_SCARR.
*
*    CLEAR GS_DATA.
*    GS_DATA-CARRID   = GS_SCARR-CARRID.
*    GS_DATA-CARRNAME = GS_SCARR-CARRNAME.
*    GS_DATA-CURRCODE = GS_SCARR-CURRCODE.
*
*    LOOP AT GT_SFLIGHT INTO GS_SFLIGHT
*         WHERE CARRID = GS_SCARR-CARRID.
*
*      LV_MONTH = GS_SFLIGHT-FLDATE+4(2).
*
*      CASE LV_MONTH.
*        WHEN '01'. GS_DATA-JANSUM = GS_DATA-JANSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '02'. GS_DATA-FEBSUM = GS_DATA-FEBSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '03'. GS_DATA-MARSUM = GS_DATA-MARSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '04'. GS_DATA-APRSUM = GS_DATA-APRSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '05'. GS_DATA-MAYSUM = GS_DATA-MAYSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '06'. GS_DATA-JUNSUM = GS_DATA-JUNSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '07'. GS_DATA-JULSUM = GS_DATA-JULSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '08'. GS_DATA-AUGSUM = GS_DATA-AUGSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '09'. GS_DATA-SEPSUM = GS_DATA-SEPSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '10'. GS_DATA-OCTSUM = GS_DATA-OCTSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '11'. GS_DATA-NOVSUM = GS_DATA-NOVSUM + GS_SFLIGHT-PAYMENTSUM.
*        WHEN '12'. GS_DATA-DECSUM = GS_DATA-DECSUM + GS_SFLIGHT-PAYMENTSUM.
*      ENDCASE.
*
*    ENDLOOP.
*
*    APPEND GS_DATA TO GT_DATA.
*
*  ENDLOOP.
```

---

## 11.3 변경된 New Open SQL 코드

```abap
* New Open SQL
  SELECT FROM SCARR AS A
    INNER JOIN SFLIGHT AS F
      ON F~CARRID = A~CARRID
    FIELDS
      A~CARRID   AS CARRID,
      A~CARRNAME AS CARRNAME,
      A~CURRCODE AS CURRCODE,

      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '01' THEN F~PAYMENTSUM ELSE 0 END ) AS JANSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '02' THEN F~PAYMENTSUM ELSE 0 END ) AS FEBSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '03' THEN F~PAYMENTSUM ELSE 0 END ) AS MARSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '04' THEN F~PAYMENTSUM ELSE 0 END ) AS APRSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '05' THEN F~PAYMENTSUM ELSE 0 END ) AS MAYSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '06' THEN F~PAYMENTSUM ELSE 0 END ) AS JUNSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '07' THEN F~PAYMENTSUM ELSE 0 END ) AS JULSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '08' THEN F~PAYMENTSUM ELSE 0 END ) AS AUGSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '09' THEN F~PAYMENTSUM ELSE 0 END ) AS SEPSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '10' THEN F~PAYMENTSUM ELSE 0 END ) AS OCTSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '11' THEN F~PAYMENTSUM ELSE 0 END ) AS NOVSUM,
      SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '12' THEN F~PAYMENTSUM ELSE 0 END ) AS DECSUM

    WHERE A~CARRID IN @SO_CARR
      AND F~FLDATE BETWEEN @LV_FROM AND @LV_TO
    GROUP BY A~CARRID, A~CARRNAME, A~CURRCODE
    ORDER BY A~CARRID
    INTO TABLE @GT_DATA.

ENDCASE.
```

## 11.4 구조 비교

### 기존 방식

```text
SCARR 조회
   ↓
SFLIGHT 조회
   ↓
ABAP에서 2중 LOOP
   ↓
월 추출
   ↓
CASE 누적
   ↓
GT_DATA 생성
```

### 변경 방식

```text
SCARR + SFLIGHT JOIN
   ↓
DB에서 월 추출
   ↓
DB에서 CASE 집계
   ↓
DB에서 SUM 완료
   ↓
GT_DATA 수신
```

## 11.5 핵심 포인트

### 1) 월 추출도 DB에서 수행

```abap
SUBSTRING( F~FLDATE, 5, 2 )
```

- `FLDATE`에서 월(5~6번째 자리)을 잘라서 사용

### 2) 월별 집계를 DB에서 수행

```abap
SUM( CASE SUBSTRING( F~FLDATE, 5, 2 ) WHEN '01' THEN F~PAYMENTSUM ELSE 0 END ) AS JANSUM
```

- 1월이면 `PAYMENTSUM` 누적
- 아니면 `0`
- 이 로직을 12개월 각각 수행

### 3) LOOP 제거
- 기존의 `GT_SCARR LOOP + GT_SFLIGHT LOOP` 제거
- ABAP 후처리 없이 DB가 바로 집계

## 11.6 Code Pushdown이란?

**원래 ABAP에서 하던 데이터 처리를 DB로 “내려보내서” DB가 하게 하는 것**

즉,

- 필터링
- 계산
- 문자열 처리
- 집계
- 조건 분기

이런 작업을 ABAP 서버가 아니라 **DB에서 처리**하는 방식이다.

### 시각화

```text
기존
DB → 데이터 전송 → ABAP 계산 → 결과

Code Pushdown
DB → 계산 완료 결과만 전송 → ABAP 사용
```

---

## 11.7 이 예제가 Code Pushdown인 이유

기존에는:

- 조회 후
- LOOP 돌고
- 월 잘라서
- CASE 분기하고
- 합계 계산했다.

변경 후에는:

- `JOIN`
- `SUBSTRING`
- `CASE`
- `SUM`
- `GROUP BY`

를 전부 **DB에서 처리**한다.

즉, 이 예제는 Code Pushdown의 전형적인 예다.

# 12. 최종 핵심 요약

## 12.1 New Open SQL
- `FIELDS`, `@`, `CASE`, `SUM`, `SUBSTRING`, `DIVISION` 등을 통해
- DB에서 계산과 집계를 수행
- LOOP 감소 및 가독성 향상

## 12.2 ADBC
- Native SQL을 객체 기반으로 실행하는 방식
- 문자열 SQL 작성, 실행, 결과 연결, 자원 해제 순서가 중요
- `MANDT`, 공백, 예외 처리에 주의

## 12.3 HANA Full Text Search
- 정확 일치가 아니라 유사도 기반 검색
- Fuzzy Search를 위해 Full-Text Index 필요
- Type-Ahead, Autosuggest와 연결 가능

## 12.4 ALV IDA
- 내부 테이블 중심이 아니라 DB 처리 중심
- 정렬/필터/집계/그룹핑을 DB에서 수행
- 대량 데이터 처리에 유리

## 12.5 ABAP CDS
- 데이터 모델 + 메타데이터 + 검색/권한/분석 연결
- 현재 표준은 `DEFINE VIEW ENTITY`
- ALV IDA, Full Text Search, RAP과 자연스럽게 연결

## 12.6 Dictionary 실습
- Domain은 기술적 타입
- Data Element는 의미 부여
- Eclipse ADT에서 Domain / Data Element 생성 흐름 숙지 필요

## 12.7 Code Pushdown
- 핵심 철학은 단 하나
- **“ABAP에서 처리하지 말고, DB가 잘하는 일은 DB에서 처리하자.”**
