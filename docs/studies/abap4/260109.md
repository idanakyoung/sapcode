# Creating Database Tables / Handling Changes / Dictionary Structures 정리

---

# 1. Creating Database Tables 이론

## 1.1 데이터베이스 테이블 정의의 전체 구조

데이터베이스 테이블 정의는 크게 3가지로 구성된다.

1. Semantic Table Attributes (Annotations)
2. `define table` 문 (테이블 이름)
3. Field List (필드 목록)

추가 메모:

- 에디터에서 새 테이블을 만들면 기본 템플릿이 자동 생성된다.
- 소스 구조를 이해하는 것이 가장 중요하다.

## 전체 구조 시각화

```text
Database Table Definition
 ├─ Semantic Table Attributes (Annotations)
 ├─ define table <table_name>
 └─ Field List
```

---

## 1.2 Semantic Table Attributes (Annotations)

### 개념
- `@` 로 시작하는 코드 라인
- 테이블의 **의미적 / 기술적 속성**을 정의
- 대부분 기본값이 설정되어 있음

### 특징
- 테이블 정의 자체의 일부
- UI, CDS, SQL 동작에 영향
- 나중에 변경 가능

### 예시

```abap
@EndUserText.label : 'Employee'
@AbapCatalog.tableCategory : #TRANSPARENT
@AbapCatalog.deliveryClass : #A
```

---

## 1.3 `define table` 문 (테이블 이름)

### 개념
- `define table` 다음에 나오는 이름이 실제 DB 테이블 이름
- 개발 객체 생성 시 이미 결정됨
- 소스 코드에서 이름을 바꾸면 **문법 오류 발생**

### 주의
- 이름 변경은 사실상 새 테이블 생성과 비슷한 영향

### 예시

```abap
define table zs4d430_employ {
```

---

## 1.4 Field List (필드 목록 기본 개념)

### 개념
- 테이블 컬럼 정의 영역
- 각 필드는 Data Element를 사용하는 것이 권장됨

### Data Element 사용 이유
- 타입, 길이, 설명, 라벨 재사용 가능
- UI 및 CDS에서 자동으로 의미 해석됨
- 유지보수 및 표준화에 유리

### 필드 입력 팁
- `:` 뒤에 이름 입력
- `Ctrl + Space` → 자동 완성
- `Strg + Enter` → Data Element 상세 확인

---

## 1.5 Existing Data Element 사용

### 개념
- 이미 존재하는 Data Element가 있다면 그대로 사용
- `:` 뒤에 이름 입력 후 `Ctrl + Space` 로 자동완성 가능
- `Strg + Enter` → Data Element 상세 정보 확인 가능

### 확인 가능한 정보
- Base Type
- Length
- Description
- Field Label (Short / Medium / Long)

### 예시

```abap
first_name : /dmo/first_name;
last_name  : /dmo/last_name;
```

---

## 1.6 New Data Element 생성 (Quick Fix)

### 개념
- 원하는 Data Element가 없을 때 사용
- 존재하지 않는 타입 입력 → 에러 발생
- `Ctrl + 1` 또는 에러 아이콘 클릭 → Quick Fix

### Quick Fix 기능
- 데이터 요소 자동 생성
- 기본 타입 지정

### 예시

```abap
employee_id : zs4d430_employee_id;
```

---

## 1.7 Primary Key (키 필드)

### 개념
- 모든 DB 테이블은 **Primary Key 필수**
- 키 필드는 행(Row)을 유일하게 식별
- Primary Key 기반으로 **Primary Index 자동 생성**

### 규칙
- 최소 1개, 최대 16개
- 모든 키 필드는 테이블 맨 앞에 위치
- 키 필드 사이에 일반 필드가 오면 안 됨

### 예시

```abap
key client      : abap.clnt;
key employee_id : zs4d430_employee_id;
```

---

## 1.8 NOT NULL 속성

### 키 필드
- NOT NULL 필수
- 수동으로 안 써도 활성화 시 자동 추가됨

### 일반 필드
- 대부분 허용
- 단, 일부 타입(예: STRING)은 불가

### 예시

```abap
key employee_id : zs4d430_employee_id not null;
last_name      : /dmo/last_name      not null;
```

---

## 1.9 Amount 필드 (CURR 타입)

### 개념
- CURR 타입 필드는 **통화 코드 필드 참조 필수**
- 통화 코드는 CUKY 타입 사용
- 참조는 Annotation으로 설정

### 규칙
- 통화 코드 필드를 먼저 정의
- 같은 테이블에 있는 것이 일반적
- 다른 테이블 참조도 가능

### 예시

```abap
@Semantics.amount.currencyCode : 'zs4d430_employ.currency_code'
annual_salary : zs4d430_annual_salary;
currency_code : abap.cuky;
```

---

## 1.10 Quantity 필드 (QUAN 타입)

### 개념
- QUAN 타입 필드는 **단위(Unit) 필드 참조 필수**
- UNIT 타입 사용
- Annotation: `@Semantics.quantity.unitOfMeasure`

---

## 1.11 Amount / Quantity Quick Fix

### 개념
금액/수량 필드를 추가했는데 참조 필드가 없으면 문법 오류가 발생한다.

### Quick Fix 종류
1. 참조 필드 + Annotation 자동 생성
2. Annotation만 추가

### 권장 방식
- 자동 생성 후
- 필드명과 타입을 의미 있는 Data Element로 수정하는 것이 좋음

---

## 1.12 Additional Table Properties (추가 속성)

테이블 속성은 2가지 방식으로 관리된다.

| 구분 | Annotations | Technical Table Settings |
|---|---|---|
| 위치 | 테이블 소스 코드 | 별도 개발 객체 |
| 역할 | 의미/기술 속성 | 물리 DB 속성(버퍼/로깅 등) |
| 영향 | UI, CDS, 의미 | SQL, 성능, 접근 |

### Technical Table Settings
- 테이블과 별도로 활성화 / 이동 가능
- DB 인터페이스에 직접 영향

---

## 1.13 Table Annotations 요약 정리

| Annotation | 의미 |
|---|---|
| `@EndUserText.label` | 테이블 설명 |
| `@AbapCatalog.tableCategory` | 영구/임시 데이터 구분 |
| `@AbapCatalog.deliveryClass` | 데이터 전송/업그레이드 정책 |
| `@AbapCatalog.dataMaintenance` | 데이터 유지보수 허용 여부 |
| `@AbapCatalog.enhancement.category` | 테이블 확장 가능 여부 |

---

## 1.14 General Settings 요약

| 설정 | 의미 |
|---|---|
| Data Class | 데이터 변경 빈도 분류 |
| Size Category | 예상 데이터 크기 |
| Storage Type | Row / Column 저장 방식 |
| Load Unit | 메모리 로드 방식 |
| Sharing Type | 메모리 공유 정책 |

## 2.3 데이터베이스 테이블 전체 코드

```abap
define table zcourse_g01 {

  key client : abap.clnt not null;
  key course_id : abap.char(10) not null;
      course_name : zcourse_name_g01;
      course_length : abap.int1;
      country : zland_g01;
      currency_code : abap.cuky;
      @Semantics.amount.currencyCode : 'zcourse_g01.currency_code'
      price : abap.curr(15,2);
      blocked : abap_boolean;
      description : abap.char(64);

}
```

## 2.4 구조체 생성

### 구조체 전체 코드

```abap
define structure zschange_g01 {

  changed_by : syuname;
  changed_at : timestamp;

}
```

### 사용 방식
- 만들어진 구조체를 테이블에 할당
- 테이블로 돌아가서 아래처럼 포함

```abap
include zschange_g01;
```

### 구조 시각화

```text
zcourse_g01
 ├─ client
 ├─ course_id
 ├─ course_name
 ├─ ...
 └─ INCLUDE zschange_g01
      ├─ changed_by
      └─ changed_at
```

# 4. Handling Changes to Database Tables 이론

## 4.1 Database Table Adjustment

Dictionary에서 테이블 정의를 변경하면  
**DB 테이블도 반드시 조정이 필요**하다.  
이 과정을 **Database Table Adjustment** 라고 한다.

### 변경 유형
- Direct Change (직접 변경)
- Indirect Change (간접 변경)

| 구분 | Direct | Indirect |
|---|---|---|
| 변경 대상 | 테이블 | Domain / Data Element |
| 예시 | 필드 추가 | Domain 길이 변경 |
| 영향 | 제한적 | 광범위 |
| 인지 | 쉬움 | 어려움 |

---

## 4.2 변경 반영 방식 (3가지)

| 방식 | 개념 | 데이터 |
|---|---|---|
| ALTER TABLE | 구조 직접 변경 | 유지 |
| Delete & Create | 삭제 후 재생성 | ❌ |
| Table Conversion | 새 테이블 + 데이터 변환 | 유지 (위험) |

---

## 4.3 ALTER TABLE / Delete & Create / Table Conversion 비교

| 항목 | ALTER TABLE | Delete & Create | Table Conversion |
|---|---|---|---|
| 데이터 유지 | O | X | O (위험) |
| 변경 자유도 | 낮음 | 매우 높음 | 높음 |
| 속도 | 빠름 | 빠름 | 느림 |
| 실무 사용 | 자주 | 거의 없음 | 제한적 |

---

## 4.4 Table Conversion 내부 처리 흐름

### 흐름
1. 원본 테이블 → `QCM<테이블명>` 으로 임시 복제/변환
2. 예시
   - `ZTAB` → `QCMZTAB`
   - Primary Index → `QCMZTAB~0`

### 시각화

```text
원본 테이블
   ↓
QCM<테이블명> 생성
   ↓
데이터 변환
   ↓
새 구조 반영
   ↓
최종 테이블 교체
```

---

## 4.5 Indirect Change 처리 방식

- Domain / Data Element 활성화 시
- 시스템이 **Dependent Objects 자동 탐색**

즉, 테이블을 직접 바꾸지 않아도  
참조하는 Data Element / Domain 변경이 광범위한 영향으로 이어질 수 있다.

## 5.5 실습용 클래스 코드

```abap
CLASS ZCL_G01_COURSE_DATA_FILL_G01 DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC .

  PUBLIC SECTION.

    INTERFACES IF_OO_ADT_CLASSRUN .
  PROTECTED SECTION.
  PRIVATE SECTION.
ENDCLASS.

CLASS ZCL_G01_COURSE_DATA_FILL_G01 IMPLEMENTATION.

  METHOD IF_OO_ADT_CLASSRUN~MAIN.
    DATA lt_course TYPE TABLE OF ZCOURSE_G01.

    lt_course = VALUE #(
        (
            course_id = 'BC400'
            course_name = 'ABAP Programming'
            course_length = 5
            country = 'KR'
            currency_code = 'KRW'
            price = '500000.00'
            blocked = ''
            description = 'How to develop ABAP Program'
         )
         (
            course_id = 'S4D430'
            course_name = 'ABAP Core Data Service'
            course_length = 5
            country = 'KR'
            currency_code = 'KRW'
            price = '500000.00'
            blocked = ''
            description = 'How to create and build ABAP CDS'
         )
    ).

    DELETE FROM ZCOURSE_G01.
    INSERT ZCOURSE_G01 FROM TABLE lt_course.
    out->write( 'Success, Data saved' ).

  ENDMETHOD.
ENDCLASS.
```

## 5.6 핵심 메모

- 바로 보여지는 데이터를 위해 클래스 생성
- `IF_OO_ADT_CLASSRUN` 인터페이스 추가
- Activate 후 실행
- 데이터 테이블 우클릭 → `Open With` → `Data Preview`

# 6. Handling Changes to Database Tables 실습 2

## 6.1 비교 대상

아래 세 가지 선언의 차이를 이해하는 것이 핵심이다.

```abap
department_id : abap.char(40);
department_id : abap.char(20);
department_id : zg01_department_id;
```

---

## 6.2 차이 설명

### ① `department_id : abap.char(40);`

#### 의미
- 길이를 테이블에 직접 정의한 것
- 이 테이블 안에서만 사용됨
- Domain / Data Element와 연결 없음

#### 결과
- DB 컬럼 길이 = `CHAR(40)`
- 라벨, 의미 정보 없음
- 다른 테이블과 연관 없음

---

### ② `department_id : abap.char(20);`

#### 의미
- 방식은 ①과 동일
- 단지 길이만 20으로 줄어든 것

#### 문제점
- 기존 데이터가 `CHAR(40)`인데 `CHAR(20)`으로 줄이면
- 데이터 손실 가능성 발생
- DB가 단순 `ALTER TABLE`을 허용하지 않을 수 있음

#### 결과
- DB 컬럼 길이 = `CHAR(20)`
- 길이 감소 시 Table Conversion 필요 가능

---

### ③ `department_id : zg01_department_id;`

#### 핵심 개념
- 테이블이 길이를 직접 결정하지 않음
- `zg01_department_id` Data Element / Domain이 길이와 의미를 책임짐
- 테이블은 참조만 함

#### 결과
- DB 컬럼 길이 = Domain 정의 길이
- 라벨, 설명, 의미 정보 자동 상속
- 같은 Data Element를 쓰는 모든 테이블과 연결됨

---

## 6.3 새 Data Element / Domain 생성 흐름

```text
department_id : zg01_department_id;
   ↓
없는 Data Element 발견
   ↓
새 Data Element 생성
   ↓
새 Domain 생성
   ↓
데이터 타입 / 길이 설정
   ↓
테이블 활성화
```

## 6.4 데이터 삽입용 클래스 코드

```abap
CLASS ZCL_EMPLOY_FILE_G01 DEFINITION
```

## 6.5 Dictionary Object 한눈에 정리

| 구분 | 역할 한 줄 요약 | 타입으로 사용 | DB 필드로 사용 |
|---|---|---|---|
| Domain | 데이터의 **기본 규격(길이/타입)** | ❌ | ❌ |
| Data Element | **필드 하나의 의미/라벨/규칙** | ⭕ | ⭕ |
| Structure | **여러 필드 묶음(구조체)** | ⭕ | ❌ |
| Database Table | **실제 DB 저장소** | ⭕ (비권장) | ⭕ |
| Table Type | **내부 테이블 전용 타입** | ⭕ | ❌ |

---

## 6.6 “타입으로 쓸 수 있나?” 핵심 표

| 객체 | `DATA lv_x TYPE ?` | 의미 |
|---|---|---|
| Domain | ❌ | 재료일 뿐, 타입 아님 |
| Data Element | ⭕ | 필드 하나의 공식 타입 |
| Structure | ⭕ | 여러 필드 묶음 |
| Database Table | ⭕ (권장 ❌) | 구조체처럼 동작 |
| Table Type | ⭕ | 내부 테이블용 |

---

## 6.7 Data Element 핵심 정리

| 항목 | 내용 |
|---|---|
| 의미 | 필드 하나의 업무 규격 |
| 포함 정보 | 길이, 타입, 라벨, 설명 |
| 재사용 | 여러 테이블/변수에서 가능 |
| Domain 관계 | Domain 위에 만들어짐 |
| 권장도 | ⭐⭐⭐⭐⭐ |

### 주의: Reference Type Data Element
- ABAP 변수 타입 ⭕
- DB 테이블 필드 ❌

# 7. Dictionary Structures 이론

## 7.1 Nested Structure(중첩 구조)와 Include Structure(포함 구조)의 차이

| 구분 | Nested Structure | Include Structure |
|---|---|---|
| 구조 | 구조 안에 구조 | 구조를 펼쳐서 포함 |
| 접근 | `a-b-c` | `a-c` |
| 계층 | 있음 | 없음 |
| 느낌 | 객체 구조 | 필드 확장 |
| 실무 | DTO, JSON | 테이블/공통필드 |

### 핵심 요약
- Nested: 구조 안에 구조를 넣는다
- Include: 구조를 펼쳐서 현재 구조에 합친다

---

## 7.2 Nested Structure 선언과 사용 (중첩)

중첩은 **필드 하나의 타입이 다른 구조인 형태**다.

### 예시

```abap
@EndUserText.label : 'Person (Nested)'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
define structure zs_g01_person_nested {

  first_name : /dmo/first_name;
  last_name  : /dmo/last_name;

  address    : zs_g01_address;
}
```

### ABAP 접근 예시

```abap
DATA ls_person TYPE zs_g01_person_nested.

ls_person-first_name = 'John'.
ls_person-address-city = 'Walldorf'.
ls_person-address-postal_code = '69190'.
```

### 중요 포인트
- `address` 자체가 하나의 구조 덩어리
- `ls_person-city` 처럼 바로 접근 불가
- 반드시 `ls_person-address-city` 형태로 접근

### 시각화

```text
ls_person
 ├─ first_name
 ├─ last_name
 └─ address
     ├─ city
     └─ postal_code
```

---

## 7.3 Include Structure 선언과 사용 (포함)

Include는 **기존 구조의 필드들을 현재 구조에 그대로 추가**하는 방식이다.

### 예시

```abap
@EndUserText.label : 'Person (Include)'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
define structure zs_g01_person_include {

  first_name : /dmo/first_name;
  last_name  : /dmo/last_name;

  include zs_g01_address;
}
```

### ABAP 접근 예시

```abap
DATA ls_person TYPE zs_g01_person_include.

ls_person-first_name  = 'John'.
ls_person-city        = 'Walldorf'.
ls_person-postal_code = '69190'.
```

### 중요 포인트
- include로 들어온 필드들은 펼쳐져서 같은 레벨로 보인다
- `address-city`가 아니라 `city`로 바로 접근
- 필드 충돌 주의

### 시각화

```text
ls_person
 ├─ first_name
 ├─ last_name
 ├─ city
 └─ postal_code
```

---

## 7.4 ABAP TYPES에서도 동일한 개념

Dictionary Structure의 `include`는  
ABAP 타입 선언에서는 보통 `INCLUDE TYPE`으로 대응된다.

### 예시

```abap
TYPES: BEGIN OF ty_person_include,
         first_name TYPE /dmo/first_name,
         last_name  TYPE /dmo/last_name,
         INCLUDE TYPE ty_address,
       END OF ty_person_include.
```

여기서도 INCLUDE로 들어온 필드는 같은 레벨에 펼쳐진다.

---

## 7.5 Named Include(이름 있는 Include) 선언과 사용

Named Include는  
**include한 구조에 별칭 이름을 붙이는 방식**이다.

### 예시

```abap
@EndUserText.label : 'Person (Named Include)'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
define structure zs_g01_person_named_incl {

  first_name : /dmo/first_name;
  last_name  : /dmo/last_name;

  address : include zs_g01_address;

}
```

### ABAP 접근 방식

```abap
DATA ls_person TYPE zs_g01_person_named_incl.

" 1) 포함 필드에 직접 접근
ls_person-city = 'Walldorf'.

" 2) Include 이름으로 그룹처럼 접근
ls_person-address-city = 'Walldorf'.
ls_person-address-postal_code = '69190'.
```

### Named Include 핵심

| 항목 | 내용 |
|---|---|
| 목적 | Include에 이름 부여 |
| 장점 | 그룹 접근 + 직접 접근 |
| 접근 방식 | `struct-field` / `struct-include-field` |
| 사용처 | RAP, REST 모델 |

```text
복제할 Structure용 클래스 템플릿 검색

zcl_g01_structure로 저장
   ↓
새 Structure 생성
   ↓
Structure 필드 속성 설정
   ↓
이름 / 설명 입력
   ↓
복제한 클래스에서 address 변수 타입으로 사용
   ↓

   ↓

   ↓

   ↓

   ↓

   ↓

   ↓
```

---

## 8.2 실습 핵심 포인트

### 1) 클래스 템플릿 복제
- Structure 테스트용 클래스 템플릿 검색
- `zcl_g01_structure`로 저장

### 2) 새 Structure 생성
- 이름과 설명 입력
- 필드 속성 정의 코드 작성

### 3) 클래스에서 타입으로 사용
- 새 Structure를 `address` 변수 타입으로 설정
- 또 다른 Structure를 `person` 변수 타입으로 설정

### 4) Structure 복제
- Person Structure 복제
- 복제된 이름으로 새 구조 생성
- 클래스에서 `person2` 변수 타입으로 설정

# 9. 최종 핵심 요약

## 9.1 Creating Database Tables
- 테이블 정의는
  - Annotation
  - `define table`
  - Field List
  구조로 이해하면 된다.
- Data Element 기반 설계가 가장 중요하다.
- Amount / Quantity는 참조 필드와 Annotation이 필수다.

## 9.2 실습 포인트
- 없는 Data Element는 Quick Fix로 바로 생성 가능
- Domain / Data Element / Structure를 함께 설계하는 흐름이 중요
- `include structure`로 공통 필드를 재사용할 수 있다.

## 9.3 Handling Changes to Database Tables
- Dictionary 변경은 DB Table Adjustment를 동반한다.
- 변경 방식은
  - ALTER TABLE
  - Delete & Create
  - Table Conversion
  으로 나뉜다.
- Direct Change보다 Indirect Change가 더 위험할 수 있다.

## 9.4 타입 설계 철학
- `abap.char(40)` 처럼 직접 타입을 박는 방식보다
- `zg01_department_id` 같은 Data Element 참조 방식이
  - 의미
  - 라벨
  - 재사용성
  - 유지보수
  측면에서 훨씬 좋다.

## 9.5 Dictionary Structures
- Nested = 구조 안에 구조
- Include = 구조를 펼쳐서 포함
- Named Include = include에 이름 부여
- 실무에서는 공통 필드 재사용에 Include가 자주 쓰인다.

## 9.6 가장 중요한 한 줄
> **DB 테이블은 단순 저장소가 아니라, Domain / Data Element / Structure / Annotation까지 포함한 Dictionary 설계의 결과물이다.**
